# 版本管理规范

> **版本**：v1.0
> **来源**：整合电影制作流程文档的"素材管理"概念
> **日期**：2026-05-19

---

## 核心理念

**每一次创作都值得被记录。**

版本管理不是为了限制创作，而是为了：
1. 保留历史，随时回溯
2. 追踪变更，理解演化
3. 明确责任，避免丢失

---

## 一、版本命名规则

### 1.1 版本号格式

```
v[大版本].[小版本].[修订次数]
```

| 位置 | 含义 | 示例 |
|------|------|------|
| 大版本 | 阶段变更（大纲→初稿→修改→定稿） | v1, v2, v3 |
| 小版本 | 阶段内重要节点 | v1.1, v1.2 |
| 修订次数 | 日常修订次数 | v1.1.1, v1.1.2 |

### 1.2 版本阶段定义

| 大版本 | 阶段 | 说明 |
|--------|------|------|
| v1.x.x | 大纲阶段 | 完成大纲，未写初稿 |
| v2.x.x | 初稿阶段 | 完成初稿，未修改 |
| v3.x.x | 修改阶段 | 进行修改，未润色 |
| v4.x.x | 润色阶段 | 润色中，未定稿 |
| v5.x.x | 定稿阶段 | 最终版本 |

### 1.3 版本示例

```
v1.0.0 - 大纲完成，初始锁定
v1.1.0 - 大纲第一次修订
v1.1.3 - 大纲修订第3次
v2.0.0 - 初稿完成，进入修改
v2.1.0 - 初稿修改版
v3.0.0 - 修改完成，进入润色
v5.0.0 - 最终定稿
```

## 二、版本记录

### 2.1 版本记录文件

在小说项目根目录创建 `VERSION.md`：

```markdown
# 版本记录

## 当前版本
v2.1.0

## 版本历史

| 版本 | 日期 | 阶段 | 说明 | 变更内容 |
|------|------|------|------|----------|
| v1.0.0 | YYYY-MM-DD | 大纲 | 大纲锁定 | 初始大纲 |
| v1.1.0 | YYYY-MM-DD | 大纲 | 大纲修订 | 调整主角设定 |
| v2.0.0 | YYYY-MM-DD | 初稿 | 初稿完成 | 完成全书写 |
| v2.1.0 | YYYY-MM-DD | 初稿 | 初稿修订 | 修复逻辑漏洞 |
```

### 2.2 每次创作的版本更新

每次创作后更新版本记录：

```markdown
## 变更记录

### v2.1.0 - YYYY-MM-DD
- 修复第5章逻辑漏洞
- 优化第8章主角动机
- 新增第12章伏笔

下次创作重点：
- 完成第13-15章初稿
```

## 三、备份策略

### 3.1 每日备份

每次创作后：
1. 保存当前版本到新文件
2. 记录当日修改摘要
3. 标记需要后续处理的问题

**文件命名**：
```
小说名_v2.1.0_YYYYMMDD.md
```

### 3.2 关键节点备份

以下节点必须备份：
- 大纲锁定时
- 每完成一卷初稿时
- 初稿锁定时
- 修改锁定时
- 定稿时

### 3.3 备份存储

```
备份结构：
/backups/
  /大纲阶段/
    v1.0.0_大纲锁定.md
    v1.1.0_修订版.md
  /初稿阶段/
    v2.0.0_初稿完成.md
  /修改阶段/
  /定稿阶段/
```

## 四、版本与评审的对应

```
版本 v1.x.x（大纲阶段）→ 评审前置 → 锁定 → v2.x.x（初稿阶段）
版本 v2.x.x（初稿阶段）→ 评审前置 → 锁定 → v3.x.x（修改阶段）
版本 v3.x.x（修改阶段）→ 评审前置 → 锁定 → v4.x.x（润色阶段）
版本 v4.x.x（润色阶段）→ 评审前置 → 锁定 → v5.x.x（定稿阶段）
```

**每次进入评审前，版本应该处于稳定状态。**

## 五、版本状态检查

### 5.1 每次创作前检查

- [ ] 当前版本是什么
- [ ] 上次创作遗留问题是什么
- [ ] 本次创作目标是什么

### 5.2 版本健康度检查

```markdown
## 版本健康度

版本：v2.1.0

检查项：
□ 是否有清晰的版本记录 - [是/否]
□ 是否有完整的备份 - [是/否]
□ 是否所有变更都有记录 - [是/否]
□ 是否有遗留问题未处理 - [是/否，列出]

版本状态：□ 健康 □ 需要整理
```

## 六、常见问题处理

### 6.1 忘记记录版本

如果忘记记录版本：
1. 立即补录当前版本
2. 回忆最近的变更，写入记录
3. 今后使用创作后记录的习惯

### 6.2 版本混乱

如果版本混乱：
1. 暂停当前创作
2. 整理当前版本状态
3. 补录历史版本记录
4. 建立新的版本命名习惯

### 6.3 需要回退到旧版本

如果需要回退：
1. 明确回退原因
2. 评估影响范围
3. 保留当前版本作为备份
4. 回退到目标版本
5. 记录回退原因
