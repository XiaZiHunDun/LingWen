## 一、CCGS 核心借鉴

### 1.1 协作设计原则（COLLABORATIVE-DESIGN-PRINCIPLE.md）

**核心理念**: 用户驱动的协作，而非自主执行

| CCGS 模式 | 小说工厂对应改进 |
|-----------|------------------|
| 提问 → 选项 → 决策 → 草稿 → 审批 | 主控调度在关键节点询问主公确认 |
| 增量章节写入 | 大纲分段审批写入 |
| 多文件写入前批准 | 多部门协同变更需确认 |

**应用到小说工厂**:
- 主控调度在关键决策点（大纲审核通过/不通过、发布决策）应主动向主公展示选项并请求确认
- 大纲文档分段审批写入，而非一次性生成完整版
- 作家修改多章时需展示修改摘要并请求确认

### 1.2 上下文管理（context-management.md）

**核心理念**: 文件即记忆，而非对话

| CCGS 模式 | 小说工厂对应改进 |
|-----------|------------------|
| session-state/active.md 活跃检查点 | 建立项目状态检查点文件 |
| 增量文件写入 | 增量章节写入（已实现） |
| 子代理委托 | 多 Agent 并行研究 |

**应用到小说工厂**:
- 在 `production/session-state/` 建立项目状态文件，记录当前阶段、进度、关键决策
- 在每个重要里程碑后更新状态文件
- 会话中断后优先读取状态文件恢复上下文

### 1.3 协调规则（coordination-rules.md）

**核心理念**: 垂直委派 + 横向协商 + 冲突上报

| CCGS 规则 | 小说工厂对应 |
|-----------|--------------|
| 垂直委派 | 主控 → 部门主编 → Agent |
| 横向协商 | 部门间直接协调（作家↔审核） |
| 冲突上报 | 主控仲裁 |
| 禁止单方面跨域变更 | 部门只能修改自己目录 |

**应用到小说工厂**:
- 明确各部门的职责边界
- 冲突场景统一上报主控调度仲裁

### 1.4 设计文档标准（design-docs.md）

**CCGS 必含 8 章节**: Overview、Player Fantasy、详细规则、公式、边界情况、依赖关系、调优旋钮、验收标准

**应用到小说工厂**:
- 大纲设计文档参照此结构
- 审核标准文档化

### 1.5 测试标准（test-standards.md）

**CCGS 模式**: 验证驱动开发，先写测试后实现

**应用到小说工厂**:
- 各检测器参照此模式：先定义验收标准（测试），再实现检测逻辑
- 回归测试：每次修复后自动跑一遍检测器

## 二、具体改进方案

### 方案 1：主控调度增强 - 关键决策点显式确认

**问题**: 当前主控调度在关键决策点可能自主决策，未向主公展示选项

**改进**:
1. 在 `CLAUDE.md` 主控调度部分增加"关键决策点必须展示选项"规则
2. 关键决策点包括：
   - 大纲审核结果（通过/不通过/修改）
   - 卷大纲审核结果
   - 正文审核迭代次数超限判定
   - 发布决策（S/A/B/C级）
   - 重大变更审批

**实现文件**: `novel-factory/CLAUDE.md`

```markdown
## 关键决策点 - 必须显式确认

主控在以下节点必须向主公展示选项并请求确认：

| 决策点 | 选项示例 |
|--------|----------|
| 大纲审核 | [通过] [不通过-原因] [修改后复审] |
| 发布决策 | [S级-发布] [A级-发布] [B级-打回修改] |
| 迭代判定 | [继续迭代] [跳过迭代-已满足标准] |
```

### 方案 2：项目状态检查点

**问题**: 会话中断后难以快速恢复上下文

**改进**:
1. 在 `production/session-state/` 目录建立项目状态文件
2. 每个重要里程碑后自动更新
3. 会话恢复时优先读取

**实现文件**: `novel-factory/production/session-state/active.md`

```markdown
# 星陨纪元 - 项目状态

## 当前阶段
- Phase: PHASE_5_BODY
- Step: STEP_16
- 批次: batch_003

## 进度
- [x] ch001-ch090 创作完成
- [x] ch001-ch090 审核通过
- [ ] ch091-ch180 创作中

## 关键决策
- 2026-05-15: 铁蛋生死状态矛盾已修复
- 2026-05-16: 小九形态统一为星灵少女

## 待确认事项
- [ ] ch091-ch100 审核意见待主公确认

## 下一步
1. 等待主公确认 ch091-ch100 审核意见
2. 启动作家修改流程
```

### 方案 3：各部门变更审批机制

**问题**: 部门自主修改可能超出范围或不符合主公意图

**改进**:
1. 部门修改前需展示修改摘要
2. 多文件变更需明确批准
3. 记录所有变更日志

**实现文件**: `novel-factory/.workflow_rules/部门变更审批协议.md`

```markdown
# 部门变更审批协议

## 变更分类

### A类 - 自动执行（无需审批）
- 章节内笔误修正
- 格式调整（标点、段落）
- 重复内容删除（单章内）

### B类 - 摘要确认（主控审批）
- 多章节修改
- 角色状态变更
- 情节调整
- 时间线修正

### C类 - 详细审批（主公审批）
- 核心角色生死变更
- 世界观设定变更
- 主线结局调整
- 新增/删除重要角色

## 审批流程

B类变更:
1. 部门展示变更摘要（含变更原因、影响范围）
2. 主控确认后执行
3. 执行结果记录到变更日志

C类变更:
1. 部门展示变更详情
2. 主控上报主公
3. 主公审批后执行
```

### 方案 6：验证驱动检测器开发

**CCGS 借鉴**: 验证驱动开发 + 回归测试

**改进**:
1. 先定义检测器的验收标准（什么算通过）
2. 实现检测逻辑
3. 用已知问题章节验证检测器有效性
4. 每次修复后自动跑检测器

**实现文件**: `novel-factory/tools/consistency/检测器开发规范.md`

```markdown
# 检测器开发规范

## 开发流程

1. **定义验收标准**
   - 明确检测器要发现什么问题
   - 准备正例（有问题）和负例（无问题）的测试章节

2. **实现检测逻辑**
   - 优先实现规则型检测
   - 不确定的用 LLM 辅助判断

3. **验证有效性**
   - 用已知问题章节测试，必须能发现
   - 用正常章节测试，不能误报

4. **建立回归测试集**
   - 每次修复后自动跑一遍
   - 发现新问题时添加到测试集
```
