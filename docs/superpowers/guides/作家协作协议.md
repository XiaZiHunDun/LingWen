# 作家协作协议 v1.0

> 从 implementation-agent-protocol.md 和 design-agent-protocol.md 迁移

## 设计阶段协议

### 接收任务
1. 阅读对应的设计文档（灵感文件/大纲）
2. 识别明确/模糊之处
3. 提出架构问题（使用批量提问，最多4个）

### 问题模板

```
我已阅读[文档名]。有几个架构问题：
1. [角色/情节/世界观]的[具体歧义]
2. [叙事功能]应该如何处理？
3. 设计文档没指定[具体细节]——请确认方向
4. [关键决策]是否需要我提出方案？
```

### 方案呈现

在实现前展示结构方案：

```markdown
## 提议方案

### 章节结构
- 开头：...
- 发展：...
- 高潮：...
- 结尾：...

### 角色弧线处理
- 主角：...
- 配角：...

### 伏笔埋设
- 伏笔A（回收点）：
- 伏笔B（回收点）：

这是否符合预期？在写正文之前有修改吗？
```

### 协作心态

- 先澄清再假设——规格永远不可能 100% 完整
- 先提出架构方案，不要直接写作——展示你的思考过程
- 透明地解释权衡——总是存在多种有效方案
- 明确标记与设计文档的偏离——设计师应该知道实现上的差异
- 有不确定时，提问而非假设
- 你是提供方案和推理的专家顾问，用户是做出最终决策的创意总监

### 结构化决策界面

使用批量提问一次性提出最多 4 个独立的架构问题。

**适用场景：**
- 每个需要展示方案的决策点
- 有约束答案的初始澄清问题
- 在单次提问中批量提出最多 4 个独立问题

**不适用场景：**
- 开放式的规格澄清——使用对话
- 单一的是/否确认（"可以写入文件吗？"）

## 正文阶段协议

### 写作规范

- 每章 3000±200 字
- 单行对话不超过 120 字符
- 使用命名占位符：{角色名}, {物品名}
- 场景描写带感官细节（视觉/听觉/嗅觉/触觉）

### 提交前自检

- [ ] 角色语音档案一致
- [ ] 情感范围表覆盖本章情绪
- [ ] 伏笔埋设已记录到伏笔清单
- [ ] 无与设计文档矛盾之处

### 交互模式示例

```

You（分析设计文档）：
[已阅读灵感文件和章节大纲]

 2. 场景设定——\"迷雾森林\"的氛围是诡异压抑还是神秘空灵？
 3. 配角苏漓的出场方式——偶遇还是被神秘力量引导？
 4. 伏笔\"古剑\"是否在本章揭示其名？"

User: [回答问题]

You（展示方案）：

这是否符合预期？在写正文之前有修改吗？"
```

## 迭代反馈协议

### 接收审核意见
1. 读取意见文件
2. 识别优先级：CRITICAL/HIGH/MEDIUM/LOW
3. CRITICAL 问题优先修复
4. 修复后标记版本号

### 提交修改

```markdown
## 修改记录 v2

### CRITICAL 修复
- [问题1]：已修复，修改点...
- [问题2]：已修复，修改点...

### HIGH 优化
- [优化1]：已采纳，修改方式...
- [优化2]：未采纳，原因...

### 版本对比
- 原文件：ch001_v1.md
- 新文件：ch001_v2.md
```

### 协作心态

- 基于反馈迭代，不带防御性
- 当用户的修改改善了你的建议时，给予肯定
- 测试证明它有效——主动提出编写测试（适用于有自动化测试场景）
- 解释你推荐某方案的原因（理论、案例、对齐度）

### 版本控制

- 每次修改后递增版本号
- 保留修改记录便于追踪
- 重大方向调整应重新走设计阶段协议
