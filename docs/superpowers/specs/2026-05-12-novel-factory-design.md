### 并行策略

| 层级 | 并行方式 |
|------|---------|
| **部门内** | Subagent + Team 机制并行（如同一卷的多个章节同时创作） |
| **部门间** | 多终端会话并行（不同部门在不同窗口运行） |
| **串行流程** | 单会话内角色切换（审核→读者→作家顺序反馈） |

## 二、完整工作流（25步闭环）

### 阶段一：立项（STEP 01-02）
1. 灵感部门生成结构化灵感文件
2. 作家部门生成《全文总体大纲》

### 阶段二：全文大纲迭代（STEP 03-05）
3. 审核部门审核全文总体大纲
4. 作家部门根据意见修改
5. 重复3-4，直到全量审核员一致通过

### 阶段三：卷大纲迭代（STEP 06-09）
6. 作家部门生成各卷大纲
7. 审核部门审核
8. 作家部门修改
9. 直到所有卷大纲通过

### 阶段四：阶段大纲迭代（STEP 10-13）
10. 作家部门拆分阶段大纲
11. 审核部门审核
12. 作家部门修改
13. 直到通过

### 阶段五：正文创作与双轨反馈（STEP 14-18）
14. 作家部门逐章撰写正文
15. 读者部门提交评论
16. 审核部门技术性审核
17. 作家部门修改
18. 直到章节意见≤5条且无逻辑硬伤 → 章节定稿

### 阶段六：分层汇总与终审（STEP 19-24）
19. 汇总部门生成《阶段汇总》
20. 审核部门审核阶段汇总
21. 作家部门微调（必要时回修正文）
22. 汇总部门生成《卷汇总》
23. 汇总部门生成《全文汇总》
24. 审核部门终审，通过后标记版本号

### 阶段七：归档闭环（STEP 25）
25. 意见仓库归档，作家部门更新记忆文档，可开启下一项目

│   └── 06_汇总_审核/
│       └── 卷{n}_汇总_审核记录.md

**目录设计原则**：
- 正文和审核记录均采用「卷_阶段_章节」三段式命名，层级统一扁平
- 意见仓库按审核类型分目录（01-06），避免深层嵌套
- 版本号内嵌文件名，支持全程可回溯

## 四、汇总部门职责与边界

### 汇总部门架构

| 角色 | 人数 | 职责 |
|------|------|------|
| 汇总主笔 | 1 | 负责全文/卷汇总的整合写作 |
| 汇总编辑 | 1 | 负责过渡润色、风格统一 |
| 汇总校验 | 1 | 负责人物/时间线一致性核查 |

### 汇总流程与脚本辅助

```
汇总步骤：
1. 汇总部门读取阶段/卷/全文的所有子内容
2. 脚本辅助（自动）：
   - 提取各章节关键事件清单
   - 生成时间线快照
   - 比对前后矛盾点
3. 汇总部门（人工）：
   - 拼接各部分
   - 撰写过渡段落
   - 整合成完整文档
4. 汇总部门写入汇总仓库
```

**脚本辅助部分固定，正文整合由汇总部门人工完成。**

## 五、状态机设计

### workflow_state.json 结构

```json
{
  "version": "1.1",
  "current_phase": "PHASE_1_LAUNCH",
  "current_step": "STEP_03",
  "phases": {
    "PHASE_1_LAUNCH": {
      "status": "in_progress",
      "steps": {
        "STEP_01": {"status": "completed", "output": "灵感_20260512_001.yaml"},
        "STEP_02": {"status": "completed", "output": "全文总体大纲_v1.md"},
        "STEP_03": {
          "status": "in_progress",
          "reviewers": ["审核员A", "审核员B"],
          "pending_count": 2
        }
      }
    },
    "PHASE_3_VOLUME_OUTLINE": {
      "status": "pending",
      "steps": {}
    }
  },
  "deadline_check": {
    "outline_iteration": 0,
    "max_outline_iterations": 3,
    "body_iteration": 0,
    "max_body_iterations": 2
  },
  "next_actions": ["等待审核员A、B完成审核"]
}
```

### 决策点分类

| 节点 | 推进方式 |
|------|---------|
| 大纲审核通过/不通过 | **人工确认**（我读取审核意见后决策） |
| 章节定稿判定 | **自动**（条件：意见≤5条+无逻辑硬伤） |
| 卷/全文汇总定稿 | **人工确认** |
| 死锁（超迭代上限） | **人工仲裁** |
| 审核/读者反馈收集 | **自动**（检测文件变化后推进） |

## 六、部门 Agent 画像规范

### 基础字段（所有 Agent 通用）

| 字段 | 内容 |
|------|------|
| `role` | 身份 + 职责描述 |
| `expertise` | 擅长的领域 |
| `workflow_rules` | 本部门工作规范（迭代终止条件、输出格式） |
| `output_format` | 固定输出格式说明 |
| `memory` | 动态追加的协作记录 |

### 规模配置

| 部门 | 人数 |
|------|------|
| 灵感部门 | 3 |
| 作家部门 | 10 |
| 审核部门 | 10 |
| 读者部门 | 20 |
| 汇总部门 | 3 |

## 七、迭代终止条件

| 类型 | 终止条件 |
|------|---------|
| 大纲类 | 全量审核员一致通过（无重大不通过项） |
| 正文类 | 未解决意见 ≤ 5 条，且审核员确认无逻辑硬伤，读者无"弃书级"吐槽 |
| 死锁防护 | 大纲每层最多3轮迭代，正文最多2轮读者反馈循环；超限由人工仲裁 |

---

## 八、版本管理规则

- 每次修改生成版本记录：原内容 → 修改意见 → 修改后内容
- 文件名含版本号（如 `_v1`, `_v2`）或时间戳
- 全程可回溯，不删除任何历史版本

## 九、CLAUDE.md 系统人设概要

主会话加载的 CLAUDE.md 包含：

1. **身份**：我是主控调度，负责任务编排、状态机维护、大决策确认
2. **工作原则**：用户定方向/标准/审美，我执行+建议+调度
3. **沟通风格**：简洁直接，主动汇报进度，及时提醒风险
4. **调度规则**：见本设计方案
