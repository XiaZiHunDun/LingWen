## 范围

### T1 (HIGH-PERF) — Cascade 邻接表
- **现状**：`reference_graph.py:326-334` 在 BFS 主体内做 `for eid in edge_ids: e = self._edges[eid]; if match: break`。每次 hop 对每个 neighbor 都得扫 `_index_by_node_edges[current_id]` 找匹配 edge。
- **痛点**：max_nodes_cap=100 × average degree 10 → 每次 cascade 1000 次 dict lookup + 引用解析。这是 reference_graph 主热路径，被 `apply_cascade` / dashboard cascade graph 频繁调。
- **方案**：
  - 新 `_adjacency_by_node: dict[str, dict[str, str]]` 在 `__init__` 与 `ingest_edge` / `add_edge` 同步预构建
  - key `(from_node_id, to_node_id)` 反向都有，O(1) 查
  - trigger_cascade 第 326-334 行直接 `_adjacency_by_node[current_id].get(neighbor.id)`
- **不动**：现有 `_index_by_node_edges: dict[str, set[str]]` 留给 `get_neighbors()` 与 `_compute_query_impact()` 用（两个都用 set 形态，已 O(degree)）。
- **Risk**：双索引同步一致性 — `add_edge`/`ingest_edge` 任何一处遗漏新索引 = 数据腐烂。Mitigation：抽 `_register_edge_in_indexes(edge)` 内部 helper，两索引一次写入，旧调用点路由到 helper。

### T2 (HIGH-PERF) — Qdrant async 包装
- **现状**：`qdrant_client.py:186` 用 `QdrantClient` (sync)。Dashboard FastAPI async endpoint `wrapper.search()` 同步阻塞 asyncio event loop，avg 50-500ms per call 当 Qdrant 冷启或网络 jitter。
- **痛点**：dashboard memory Gateway endpoints (`memory_context_search`) 在用户路径上，事件循环 block = 其他 endpoint 全卡。
- **方案**：
  - 加 `async def search_async(...)` 包装：用 `asyncio.to_thread(self.search, ...)` 把 sync `search` 送到 thread pool
  - 同步 `search()` 主路径 0 改：保证 6+ 现有调用点 (`memory_service.py` / `query_engine.py` / `embed_chapters.py` 等) 不破
  - `_raw_vector_search` 同样加 `async def _raw_vector_search_async(...)` 走 `to_thread`
  - `from qdrant_client import QdrantClient` 不动 (sync client 是默认底座)；不引入 `QdrantAsyncClient` (那是另一份大改造)
- **不动**：`upsert` / `delete` / `batch_search` / `search_with_filter` — 写路径 + 批量不在用户 critical path
- **Risk**：`to_thread` 默认 pool size 受 `asyncio` 控制，突发并发如果全部 search_async 可能挤占。Mitigation：thread pool 默认 40 threads per process，足够 4 worker × uvicorn 之外的并发余量；per-call 加 trace log `async_search_started` / `async_search_done` 便于监控。
- **后续 follow-up**：Phase 15.0 可考虑切到 `QdrantAsyncClient` 真正原生 async，那是一次大改造，需要全面 async 化 callers。

### T3 (HIGH-PERF) — `_LRUCache` TTL
- **现状**：`qdrant_client.py:76-100` `_LRUCache` 用 OrderedDict + RLock，仅 LRU 淘汰，**无时间维度**。
- **痛点**：(1) 写新章后旧 embedding 的 vector search cache key 还在，新 vector 不会被 cache 命中但旧 stale entry 占用 LRU slot → 命中率持续下降。(2) user A 的 chapter-A vector cache 被 user B 复用 stale 数据时不影响 (key 已含 collection_name)，但 user A 自己的 chapter-A-旧文本 cache 命中可能让 RAG 返回过时片段。
- **方案**：
  - 加 `_ttl_seconds: float` 构造参数 (默认 300s = 5 min, 跟 Phase 8 cost window 同源)
  - `_cache: dict[key, tuple[value, ts]]`（ts = `time.monotonic()`）
  - `get(key)` 先查：(a) 不存在返 None；(b) 存在但 `now - ts > ttl` 则移除并返 None；(c) 存在且未过期 → move_to_end + 返 value
  - `put(key, value)` 时若 key 存在只更新 value 而保留 ts（避免人为刷 TTL）；同 LRU eviction 逻辑
  - 现有 `_LRUCache(max_size=N)` 调用 → 改 `_LRUCache(max_size=N, ttl_seconds=300)`，配置通过 `QdrantClientWrapper` 的 yaml 字段 `retrieval.cache_ttl_seconds` 传入
- **不动**：现有 `clear()` / `clear_collection()` 语义不变
- **Risk**：TTL 设太短 = 缓存命中率下降 / 反向压力；设太长 = stale 问题未根治。Mitigation：默认 300s 是合理中庸；测试覆盖 "stale entry 在 TTL 过后被淘汰" + "TTL=0 等价禁用 cache"。
