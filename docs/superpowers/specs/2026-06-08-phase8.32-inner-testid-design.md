## Design

### 1. testid 命名约定 (3 规则)

| 规则 | 例子 | 说明 |
|---|---|---|
| Rule 1: kebab-case 镜像 stable class (Phase 8.13/8.31 模式) | `option-btn` / `defer-btn` / `status-badge` / `meta-info` / `readonly-hint` / `reset-zoom-btn` / `zoom-display` | 9/10 testid 走此规则,testid = class 名 kebab-case,**不**加 parent prefix (e.g. 不用 `decision-card-option-btn`) |
| Rule 2: semantic 命名 (class 共享需 spec 区分时) | `zoom-in-btn` / `zoom-out-btn` | 2 个 `<button class="zoom-btn">` 共享 class (zoomIn + zoomOut),spec 需区分点击哪个,testid 加 semantic 后缀,跟 Phase 8.31 sidebar-cost-tier-text (text span) 命名精神一致 |
| Rule 3: additive only (不改 class) | 全部 10 处 | 0 改 `class=` / 0 改 `:class=` / 0 改 `v-for` / 0 改 `@click` / 0 改 `:disabled`,纯 attribute 加 |

**Rule 2 例外解释**: `zoom-btn` class 在 2 个 sibling button 共享且无 parent wrapper 可走 scoped find (`opusRow.find(...)` 模式不可用,因为 zoom controls 容器内 2 button 是 sibling),**只能**用不同 testid 区分. 这是"spec 需"驱动的合理例外,不是命名不一致。

### 3. 新 spec (2 文件, 7 tests)

#### 3.1 `tests/unit/decision-card.spec.ts` (NEW, ~80L, 3 tests)

**Setup**: 无全局 mock,DecisionCard 不依赖 mermaid / echarts / composables,直接 mount + props fixture 即可。

**Test 1: `pending_decision_renders_options_defer_cancel`**
- Fixture: `{ decision_id: 'd1', status: 'pending', kind: 'outline_judgment', priority: 1, options: ['选项 A', '选项 B', '选项 C'], prompt: '...', context: {} }`
- 断言:
  - `findAll('[data-testid="option-btn"]').length === 3` (3 option 按钮)
  - `[data-testid="defer-btn"]` exists
  - `[data-testid="cancel-btn"]` exists
  - `[data-testid="status-badge"]` 0 个 (statusBadge map 不含 pending,跟 Phase 6.6.B 注释一致)
  - `[data-testid="readonly-hint"]` 0 个
  - `[data-testid="meta-info"]` 0 个 (metaInfo computed: `if (d.status === 'pending') return null;`)

**Test 2: `resolved_decision_renders_status_badge_and_meta_info`**
- Fixture: `{ decision_id: 'd2', status: 'resolved', resolved_by: '主公', resolution: '选项 A', resolved_at: '2026-06-08T10:00:00', reason: 'OK', options: ['选项 A', '选项 B'] }`
- 断言:
  - `[data-testid="option-btn"]` 0 个 (pending 才渲染)
  - `[data-testid="defer-btn"]` 0 个
  - `[data-testid="cancel-btn"]` 0 个
  - `[data-testid="status-badge"]` 1 个,`element.classes()` 含 `status-resolved` (跟 `statusBadge` map 一致)
  - `[data-testid="readonly-hint"]` 1 个,`text()` 含 "解决" (resolved 走 `v-else` 分支 L59 + 三元 fallback `'resolved' ? '解决' : '...'`)
  - `[data-testid="meta-info"]` 1 个,`text()` 含 "解决人" / "选项 A" / "时间" (跟 `metaInfo` computed join 逻辑一致)

**Test 3: `cancelled_decision_renders_readonly_hint`**
- Fixture: `{ decision_id: 'd3', status: 'cancelled', resolved_by: '主公', resolved_at: '2026-06-08T11:00:00', reason: '实验失败' }`
- 断言:
  - `[data-testid="option-btn"]` 0 个
  - `[data-testid="defer-btn"]` 0 个
  - `[data-testid="cancel-btn"]` 0 个
  - `[data-testid="status-badge"]` 1 个,`element.classes()` 含 `status-cancelled` (cancelled **在** statusBadge map L110,跟 Phase 6.6.B 注释一致)
  - `[data-testid="readonly-hint"]` 1 个,`text()` 含 "取消" (cancelled 走 `v-else` 分支 L59 + 三元 fallback `'cancelled' ? '取消' : '推迟'`)
  - `[data-testid="meta-info"]` 1 个,`text()` 含 "取消原因: 实验失败" (跟 `metaInfo` computed 走 `cancelled ? '取消' : '推迟'` 分支)

**Test 2 vs Test 3 关键区别** (spec 价值):
- resolved → statusBadge.text 含 "✅ 已解决" + class `status-resolved`;readonly-hint.text 含 "解决"
- cancelled → statusBadge.text 含 "❌ 已取消" + class `status-cancelled`;readonly-hint.text 含 "取消"
- 区别: statusBadge 的 class + text 不同(`status-resolved` vs `status-cancelled`); readonly-hint 文字不同("解决" vs "取消"); meta-info 走 `取消原因` 分支(verb = `'取消'`)
- 这是 spec 抓住的语义边界,不是 ceremonial 重复
