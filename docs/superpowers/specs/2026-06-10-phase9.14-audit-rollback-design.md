## 4. Design (7 sections)

### 4.1 数据模型 + 状态机 (Section 1)

#### 4.1.1 `ripple_audit` table (append-only, 9 字段)

```sql
CREATE TABLE IF NOT EXISTS ripple_audit (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    ripple_id TEXT NOT NULL REFERENCES reference_ripples(id),
    action TEXT NOT NULL CHECK(action IN ('created', 'applied', 'rejected', 'failed', 'rolled_back')),
    prev_status TEXT,        -- NULL for 'created' entry
    new_status TEXT NOT NULL,
    actor TEXT NOT NULL,
    origin TEXT NOT NULL CHECK(origin IN ('ui', 'cli', 'system')),
    reason TEXT,            -- 非空 for 'rejected'/'rolled_back', else NULL
    created_at TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_audit_ripple ON ripple_audit(ripple_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_audit_actor ON ripple_audit(actor, created_at DESC);
```

**关键点**:
- `id INTEGER PRIMARY KEY AUTOINCREMENT` 保证 monotonic,append-only 强保证
- `ripple_id` FK 引 `reference_ripples(id)`,**不** 设 `ON DELETE CASCADE` (Phase 9.15 统一处理)
- `prev_status` 允许 NULL (created entry 没有 prev)
- 2 indexes: 按 ripple 时间线 + 按 actor 全局查
- `created_at` ISO 8601 UTC string (跟 reference_ripples 1:1)

#### 4.1.2 状态机扩展 (软回滚到 pending)

```
                 ┌────────────────────────────────────┐
                 ▼                                    │
[created] ─→ [pending] ─apply──→ [applied] ─rollback──┤
                 │              │                      │
                 │              └─reject─→ [rejected] ─┤
                 │                                     │
                 └──failed──→ [failed] (terminal)      │
                                                      │
                          (回滚后回到 pending, 可重新 apply/reject)
```

- `rolled_back` **不是**新 terminal 状态,是 audit action
- `reference_ripples.status` rollback 后实际回到 `pending`,`applied_at` 清 NULL
- 跟 Phase 9.13 既有 `_TERMINAL_STATUSES = ("applied", "rejected", "failed")` 0 冲突 (rollback 写完后 status 已经是 pending)
- 既有 `ConflictError` 行为保留:仍是 terminal 时拒绝 transition
- 新校验:`rollback_ripple` 触发时 `current.status` 必须 ∈ {applied, rejected} (failed 不可 rollback,语义失败非业务决策)

### 4.2 Storage 层 (Section 2) — 4 additive methods

```python
# infra/cross_volume/storage.py (additive, +120L)

class RippleStorage:
    # 既有 8 method public signature 0 改

    def record_audit(  # T2 internal helper
        self,
        ripple_id: str,
        action: str,
        prev_status: str | None,
        new_status: str,
        actor: str,
        origin: str,
        reason: str | None,
    ) -> int:  # returns audit id
        """Append 1 row to ripple_audit, 独立 commit。"""
        with self._connect() as conn:
            cur = conn.execute(
                "INSERT INTO ripple_audit (ripple_id, action, prev_status, new_status, actor, origin, reason, created_at)"
                " VALUES (?, ?, ?, ?, ?, ?, ?, ?)",
                (ripple_id, action, prev_status, new_status, actor, origin, reason,
                 datetime.now(timezone.utc).isoformat()),
            )
            conn.commit()
            return cur.lastrowid

    def get_audit_history(
        self,
        ripple_id: str,
        limit: int = 50,
        offset: int = 0,
    ) -> list[AuditEntry]:
        """Time-ordered audit entries (newest first, 默认 50 条)。"""
        with self._connect() as conn:
            rows = conn.execute(
                "SELECT * FROM ripple_audit WHERE ripple_id = ? ORDER BY created_at DESC, id DESC LIMIT ? OFFSET ?",
                (ripple_id, limit, offset),
            ).fetchall()
            return [self._row_to_audit(r) for r in rows]

    def rollback_ripple(
        self,
        ripple_id: str,
        actor: str,
        origin: str,
        reason: str,
    ) -> CrossVolumeRipple:
        """原子化: status applied/rejected → pending + applied_at NULL + 写 audit。

        Validation:
            - ripple 存在 (else KeyError → 404)
            - current.status ∈ {applied, rejected} (else ValueError → 422)
            - reason 非空 (else ValueError → 422)

        用 atomic_batch 包裹 UPDATE + INSERT,确保 2 步同生共死。
        """
        if not reason or not reason.strip():
            raise ValueError("reason is required for rollback")
        with self.atomic_batch() as conn:
            row = conn.execute(
                "SELECT * FROM reference_ripples WHERE id = ?", (ripple_id,)
            ).fetchone()
            if row is None:
                raise KeyError(f"ripple {ripple_id} not found")
            current = self._row_to_ripple(row)
            if current.status not in ("applied", "rejected"):
                raise ValueError(
                    f"can only rollback applied/rejected ripples, current status: {current.status!r}"
                )
            now_iso = datetime.now(timezone.utc).isoformat()
            conn.execute(
                "UPDATE reference_ripples SET status = 'pending', applied_at = NULL WHERE id = ?",
                (ripple_id,),
            )
            conn.execute(
                "INSERT INTO ripple_audit (ripple_id, action, prev_status, new_status, actor, origin, reason, created_at)"
                " VALUES (?, 'rolled_back', ?, 'pending', ?, ?, ?, ?)",
                (ripple_id, current.status, actor, origin, reason, now_iso),
            )
        # 跟 update_ripple_status 1:1: re-fetch 避免 race + broadcast
        updated = self.get_ripple_by_id(ripple_id)
        assert updated is not None
        _broadcast_ripple_event(
            "ripple_status_changed",
            {"ripple_id": ripple_id, "new_status": "pending", "actor": actor,
             "origin": origin, "reason": reason, "applied_at": None, "action": "rolled_back"},
        )
        return updated
```

**`update_ripple_status` 改 1 处 (additive)**: 在末尾 `conn.commit()` 后,补 `self.record_audit(...)` 调用,把 actor/origin/reason 持久化。**0 改 public signature** (现有调用方 0 改),仅内部追加 audit 写入(走独立 `record_audit` 事务)。

**`append_ripple` 改 1 处 (additive)**: 同理,写完 ripple 后写 `created` audit entry (actor=ripple.created_by, origin='system')。

**`ConflictError` 复用**: 仍只 raise 在 terminal 状态,`rollback` 自己抛 ValueError (不 raise ConflictError,因为是反向操作不是 transition conflict)。

```python
    limit: int = Query(50, ge=1, le=200),

@app.post("/api/cvg/ripples/{ripple_id}/rollback",
          response_model=RippleDetailResponse)
async def rollback_ripple(
    ripple_id: str,
    request: RippleRollbackRequest,
) -> RippleDetailResponse:
    """Reverse an apply/reject: status → pending + write audit."""
    storage = _default_storage()
    try:
        ripple = storage.rollback_ripple(
            ripple_id,
            actor=request.actor,
            origin=request.origin,
            reason=request.reason,
        )
    except KeyError:
        raise HTTPException(404, f"ripple {ripple_id} not found")
    except ValueError as e:
        raise HTTPException(422, str(e))
    return _ripple_to_detail(ripple)
```

```python
@click.argument("ripple_id")
@click.option("--reason", required=True, help="Why rolling back (required, non-empty)")
@click.option("--actor", default="cli:lingwen-ripple", help="Actor identifier")
def rollback(ripple_id: str, reason: str, actor: str) -> None:
    """Rollback an applied/rejected ripple to pending.
    
    Writes 'rolled_back' audit entry + clears applied_at. Reason is required.
    
    Example:
        lingwen.py ripple rollback abc-123-def --reason "误判,需要重新评估"
    """
    if not reason.strip():
        click.echo("Error: --reason cannot be empty", err=True)
        sys.exit(1)
    storage = RippleStorage(...)
    try:
        updated = storage.rollback_ripple(
            ripple_id, actor=actor, origin="cli", reason=reason,
        )
        click.echo(
            f"✓ Rolled back {ripple_id} → status={updated.status} "
            f"(audit entry written, broadcast sent)"
        )
    except KeyError:
        click.echo(f"Error: ripple {ripple_id} not found", err=True)
        sys.exit(1)
    except ValueError as e:
        click.echo(f"Error: {e}", err=True)
        sys.exit(1)
```

**关键点**:
- `--reason` 是 `required=True`,click 自动校验非空
- 调用 storage method 而非 HTTP API (跟 Phase 9.11 backfill 1:1 pattern, 0 启动 dashboard server)

### 4.6 Error Handling + 边界 case (Section 6)

| 场景 | Storage exception | HTTP code | 端点 |
|------|------------------|-----------|------|
| ripple 不存在 | `KeyError` | 404 | `GET /audit` / `POST /rollback` |
| 当前 status 不是 applied/rejected (rollback) | `ValueError("can only rollback ...")` | 422 | `POST /rollback` |
| reason 空字符串 / 仅空白 | `ValueError("reason is required ...")` | 422 | `POST /rollback` |
| reason 缺失 (CLI `--reason` 必传) | click `MissingOption` | exit 2 | `lingwen.py ripple rollback` |
| origin 非法 (非 ui/cli/system) | Pydantic `Literal` 校验 | 422 | `POST /rollback` |
| reason 超长 (>500 chars) | Pydantic `max_length=500` | 422 | `POST /rollback` |
| audit 空 (ripple 存在但 0 audit) | `get_audit_history` return `[]` | 200 + `[]` | `GET /audit` |
| 并发 rollback 竞态 | 第二个请求 status 已 pending → ValueError | 422 | `POST /rollback` |
| 既有:terminal 状态 try apply/reject | `ConflictError` (Phase 9.13 既有) | 409 (既有) | `POST /apply` / `/reject` |

**atomic 保护**:
- `rollback_ripple` 走 `atomic_batch` (BEGIN IMMEDIATE),UPDATE reference_ripples + INSERT ripple_audit 同事务
- 并发场景: SQLite WAL + BEGIN IMMEDIATE 串行化写,第二个 rollback 看到 status=pending → ValueError raise,正确拒绝
- `record_audit` (单 row INSERT) 是 single-row commit,跟主 write path 弱一致;失败仅丢 1 条,前台可能有 1 条 timeline gap,可接受

**UI 边界**:
- drawer 打开时 ripple 已被其他 tab rollback → `GET /api/cvg/ripples/{id}` 重新拉 → status 变 pending → Apply/Reject 按钮 enable, Rollback 按钮 disable
- Rollback 按钮 disable if `status ∉ {applied, rejected}` (pre-check)
- API 返 422 → drawer 顶部红色 banner + 不关 drawer,用户可重试
- 拉 audit 失败 → drawer timeline section 显示 "Failed to load history" + retry button (不阻塞 Apply/Reject/Rollback)

**CLI 边界**:
- 0 启动 dashboard server,直连 SQLite (跟 backfill 同模式)
- DB 不存在 / 路径错 → click `FileNotFoundError` 自动 traceback, exit 1
- ripple_id 已 terminal (applied 后被 rollback 过又 re-applied) → 第二次 rollback 仍可成功 (status=applied → pending),允许多次回滚历史

**Backfill migration 风险**:
- Phase 9.14 上线后,既有的 14,xxx ripples 无 audit history (Phase 9.10/9.11/9.12 写时未写 audit)
- `GET /audit` 对老 ripple 返 `[]` (前端 drawer timeline 显示 "No history yet")
- 0 数据迁移 (YAGNI),audit 表只追踪 Phase 9.14 之后的变更
