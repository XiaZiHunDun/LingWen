## 关键设计决策

### 1. T1 schema shape: 5 fields 1:1 跟 9.16 dict literal

- **选 5 fields** (A) — `ripple_id`, `cascade_node_count`, `cascade_edge_count`, `depth_reached`, `bfs_algorithm_version` 跟 9.16 dict literal 1:1, 0 字段加 / 0 字段减
- **不选更多** (B) — 加 `created_at` / `bfs_algorithm_version` 改成 enum / etc, scope 爆炸
- **不选更少** (C) — 删 `bfs_algorithm_version` (Phase 9.15 cascade BFS 算法已标 v1/v2_weighted, 删 = 信息丢失, frontend `latestCascadeUpdates` ref 数据缺字段)

### 2. T1 typed param vs dict 兼容: 双路径

- **选双路径** (A) — `isinstance` check + `model_validate(dict)` fallback, 旧 dict caller 0 改
- **不选 typed-only** (B) — 强制所有 caller 改 typed, 改 Phase 9.16 storage.py caller + 任何旧 test stub
- **不选 dict-only** (C) — 0 IDE autocompletion, Pydantic ValidationError 缺失, 跟 spec Goals 冲突

### 3. T1 storage.py caller 改: typed build (vs dict literal)

- **选 typed build** (A) — `CascadeUpdatePayload(ripple_id=..., cascade_node_count=..., ...)` 5 fields IDE 显式, 1:1 镜像 cascaded 5 attribute
- **不选 dict literal 保留** (B) — 0 改 storage.py 但 typed param receiver 仍 work (defensive model_validate), IDE 0 autocomplete
- **不选 helper function** (C) — `build_cascade_update_payload(cascaded)` 工厂, 1 indirection, YAGNI

### 4. T2 MAX_HANDLERS number: 50

- **选 50** (A) — 远超实际 (1-3 components), 跟前端 dev affordance `window.__cascadeHandlers` 0 冲突
- **不选 10** (B) — 太低, 实际 1-3 components 0 risk, 50 留余量
- **不选 100** (C) — 内存浪费上限 100 handler 仍 leak, 50 够

### 5. T2 guard 位置: add 之前 (vs after + check)

- **选 add 之前** (A) — early return Set.size 0 增, leak 真正止于 50
- **不选 add 之后** (B) — 51st handler 进了 Set, 然后 delete → 还是有 setSize 51 一次, race condition
- **不选 setInterval cleanup** (C) — 异步回收, leak 期间内存峰值 51+ 仍有风险

### 6. T2 warn message: 中文 + 数值具体

- **选具体数字** (A) — `console.warn('[useWorkflowSocket] MAX_HANDLERS=50 reached, refusing to register cascade handler')` 一眼看到 50
