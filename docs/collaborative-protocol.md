## 核心原则

**用户驱动协作，而非自主执行。**

```
Question → Options → Decision → Draft → Approval → Write
```

---

## 一、战略决策工作流

当用户提出重大决策或发生跨部门冲突时：

### Step 1：理解全貌

- 阅读相关 pillars/约束/历史决策
- 识别真正的问题所在（通常比表面问题更深）
- 使用 `AskUserQuestion` 批量提问（最多 4 个约束性问题）

### Step 2：框架决策

- 清晰陈述核心问题
- 解释下游影响（这个决策影响什么）
- 确定评估标准（pillars、质量、范围、愿景）

### Step 3：呈现选项（2-3 个）

每个选项必须包含：
- **具体含义**：这意味着什么
- **Pros/Cons**：哪些 pillars 被服务，哪些被牺牲
- **下游后果**：对创作、节奏、设定的影响
- **风险和缓解策略**
- **参考案例**：其他小说/项目如何处理类似决策

### Step 4：给出明确推荐

- "我推荐选项 X，因为..."
- 解释推理依据（理论、先例、项目特定背景）
- 承认接受的权衡
- 明确声明：**"这是你的决定——你比任何人都更了解你的愿景"**

### Step 5：支持决策

- 文档化决策（写入 ADR 或状态文件）
- cascade 到相关部门
- 设置验证标准："如果这个决策正确，我们会看到..."

## 二、协作协议（各场景）

### 场景 A：作家提交大纲给审核

```
作家：
  "我已经完成卷1大纲草稿，请审核。"

审核：
  "收到。我需要先确认几个问题以理解你的意图：

  1. 卷1 的核心冲突是人物内因还是外因？（影响节奏设计）
  2. 支线剧情的比例目标是？（影响章节分配）
  3. 读者预期在卷1 获得什么情感体验？（影响情绪弧线设计）

  请回复这 3 个问题，我再开始审核。"

作家： [回复]

审核：
  [完成审核，输出结构化意见]
  - 选项 A（维持原方案）：... pros/cons
  - 选项 B（调整支线比例）：... pros/cons
  - 推荐：选项 B，因为...

  "推荐选项 B。但这是你的决定，你最了解读者期待。"
```

### 场景 B：两个审核员意见冲突

```
审核员A："卷2 主线太弱，建议加强主角与反派的对抗"
审核员B："卷2 主线OK，但支线薄弱，建议增加盟友戏份"

主编（narrative-director）：
  "我需要理解双方立场。让我先阅读相关文档..."

  [分析后]

  "核心问题：卷2 的核心动力是什么？

  **选项 A**：加强主角-反派对抗
  - 服务：叙事张力 pillar
  - 牺牲：角色深度 pillar（减少盟友描写）

  **选项 B**：增加盟友戏份
  - 服务：角色弧光 pillar
  - 牺牲：叙事节奏 pillar（可能拖慢节奏）

  **选项 C**：双线并行（第1-3章侧重对抗，第4-6章侧重盟友）
  - 服务：两个 pillar 都部分服务
  - 风险：技术复杂度增加

  **我的推荐**：选项 C
  但这是你的决定。卷2 的创作意图由你决定。"

用户：[做出选择]

主编：[支持决策，文档化，更新相关文件]
```

### 场景 C：修改前的 Options 确认

```
作家：
  "审核意见说主角性格不够立体，我打算修改第3章和第5章。"

主编：
  "好的。修改主角性格是高影响行动，我需要你确认修改方向：

  **选项 A**：增加内心独白（通过意识流描写展现内心冲突）
  - Pros：直接建立情感连接
  - Cons：可能拖慢节奏

  **选项 B**：通过盟友视角折射（通过其他角色的反应展现）
  - Pros：侧面描写更含蓄，符合'冷峻写实'基调
  - Cons：需要额外角色描写

  **选项 C**：增加决策困境场景（通过两难选择展现）
  - Pros：既展示性格又推动情节
  - Cons：需要精心设计情境

  推荐选项 B，因为更符合项目的'冷峻写实' pillar。
  你想用哪个方向？"
```

## 三、冲突升级路径

| 情况 | 升级到 |
|------|--------|
| 两个审核员意见冲突 | 主编（narrative-director） |
| 人设 vs 剧情冲突 | 主作家（creative-director） |
| 设定 vs 世界观冲突 | 世界观管理员（world-builder） |
| 写作风格 vs 节奏冲突 | 主编 |
| 作家之间进度冲突 | 调度 Agent（producer） |
| 读者反馈 vs 审核意见冲突 | 用户（最终仲裁） |

## 四、草案展示规则

**所有正式写入文件前必须展示草案：**

1. **大纲草案**：展示整体结构和关键情节点，获得确认后再写细节
2. **正文草案**：展示章节开头和结尾，中间部分可用概要说明，获得确认后再写全文
3. **修改草案**：展示修改思路（换一种方式说），获得确认后再执行

### 草案格式示例

```markdown
## 草案：卷1 第3章 修改方案

### 整体思路
通过增加主角与导师的对话场景，以侧面方式展现主角的内心挣扎。

### 草案结构
- 开头：主角独自在书房，面对密信犹豫（情绪钩子）
- 中段：导师出场，以问句引导主角说出内心纠结
- 结尾：主角做出决定，但读者看到的是决定过程而非结果

### 字数估算
约 2800 字（符合项目标准 3000±200）

### 风险
- 可能显得'导师话太多'——需要给导师独特的声音

是否按此方向执行？
- 确认：请按此方向写
- 调整：请说明需要调整的部分
- 否定：请说明原因，我重新思考
```

## 五、部门协作契约

### 作家部门

- 必须遵守项目的 style_guide
- 正文 3000±200 字/章
- 章节结尾留悬念或情绪钩子
- **重大修改前必须提交 Options**，不得自主决定
- 收到审核意见后，先理解再回复，不得直接抵触

### 审核部门

- 审核意见必须包含：
  - 问题描述
  - 影响的 pillar/维度
  - 修改方向建议（至少 2 个选项）
- **不得直接修改内容文件**，只输出意见
- 意见措辞要具体，避免模糊评价

### 读者部门

- 阅读时记录即时反应（喜欢/讨厌/困惑）
- 评论要带情绪，不只是分析
- 明确标注"弃书级"问题

### 汇总部门

- 汇总时只读取当前层级的子节点文件
- 不直接展开全部正文
- 汇总稿须通过一致性检查

## 六、决策文档化

### ADR（架构决策记录）

当决策影响多个卷/阶段或涉及设定变更时，必须创建 ADR：

```markdown
# ADR-[编号]：[简短标题]

## 状态
- 提出日期：[日期]
- 状态：[已接受 / 已拒绝]

## 背景
[需要决策的问题]

## 决策
[所选方案]

## 理由
[为什么做这个决策]

## 后果
- 正面：...
- 负面：...
- 风险：...

## 备选方案
- 方案A：[描述] — 为什么拒绝
- 方案B：[描述] — 为什么拒绝
```

## 七、质量门禁

### 各层级通过条件

| 层级 | 通过条件 | 评分阈值 |
|------|---------|---------|
| 全文大纲 | 全量审核员通过 | ≥75 |
| 卷大纲 | 全量审核员通过 | ≥75 |
| 阶段大纲 | 全量审核员通过 | ≥75 |
| 正文 | 未解决意见≤5，且无逻辑硬伤 | ≥70 |

### 不合格处理

1. 第一轮不合格：返回作家修改
2. 第二轮不合格：升级到主编仲裁
3. 第三轮不合格：用户介入决策
