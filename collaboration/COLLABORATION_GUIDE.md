# 协作规范总览

> **最后更新**: 2026-07-13  
> **适用范围**: 所有开发助手（本地/VM/其他）

---

## 1. 协作原则

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **黑板共享** | 所有协作文件对所有助手开放，任何人都可以更新 |
| **状态透明** | 所有任务状态、进度、问题都记录在黑板上 |
| **认领制** | 任务通过认领方式分配，先到先得 |
| **小步提交** | 每次修改后及时更新黑板，避免重复工作 |
| **互相尊重** | 不覆盖他人正在进行的工作，有疑问先沟通 |

### 1.2 助手角色（灵活，非固定）

| 角色 | 说明 |
|------|------|
| **开发助手** | 负责代码修改、功能实现、Bug修复 |
| **测试助手** | 负责运行测试、验证功能、记录结果 |
| **部署助手** | 负责服务部署、环境配置、运维操作 |
| **协调助手** | 负责任务分配、进度跟踪、风险识别 |

> **注意**: 角色是灵活的，同一助手可以在不同时间扮演不同角色。

## 2. 协作文件说明

| 文件 | 用途 | 更新频率 | 谁可以更新 |
|------|------|----------|------------|
| `CURRENT_STATUS.md` | 全局状态看板，一屏了解项目全貌 | 每次状态变更 | 所有助手 |
| `ACTIVE_TASK.md` | 当前活跃任务详情，含子任务分解 | 任务推进时 | 所有助手 |
| `BACKLOG.md` | 待办事项池，按优先级排序 | 新增/完成任务时 | 所有助手 |
| `TEST_LOG.md` | 测试执行记录，含结果和日志链接 | 每次测试后 | 所有助手 |
| `DEPLOYMENT_LOG.md` | 部署记录，含版本和步骤 | 每次部署后 | 所有助手 |
| `ENV_CONFIG.md` | 环境配置手册，含环境变量和搭建步骤 | 环境变更时 | 所有助手 |

## 3. 任务状态流转

### 3.1 状态定义

| 状态 | 图标 | 说明 |
|------|------|------|
| 待开始 | 📋 | 任务已创建，等待认领 |
| 进行中 | 🔄 | 已被认领，正在进行 |
| 待验证 | ⏳ | 开发完成，等待测试验证 |
| 已完成 | ✅ | 验证通过，任务完成 |
| 阻塞 | 🔴 | 遇到问题，需要协助 |
| 已取消 | ❌ | 任务取消，不再进行 |

### 3.2 状态流转图

```
📋 待开始
   ↓（认领）
🔄 进行中
   ↓（开发完成）
⏳ 待验证
   ↓（测试通过）
✅ 已完成
   ↑（测试失败）
🔄 进行中

🔴 阻塞 ← 任意状态遇到问题
```

## 4. 协作流程

### 4.1 标准开发流程

```
1. 查看 CURRENT_STATUS.md 和 BACKLOG.md
   ↓
2. 从 BACKLOG 中认领任务
   - 在 ACTIVE_TASK.md 中添加任务
   - 标记状态为 🔄 进行中
   - 标注认领人
   ↓
3. 开发工作
   - 编写代码
   - 本地自测
   ↓
4. 更新 ACTIVE_TASK.md
   - 标记子任务完成
   - 整体状态改为 ⏳ 待验证
   - 说明做了什么修改
   ↓
5. 提交代码
   - git add / commit / push
   - 在 ACTIVE_TASK.md 记录 commit hash
   ↓
6. 测试验证（任意助手可做）
   - 拉取代码
   - 运行测试
   - 在 TEST_LOG.md 记录结果
   ↓
7. 更新状态
   - 测试通过 → ✅ 已完成
   - 测试失败 → 🔄 进行中（说明失败原因）
```

### 4.2 紧急修复流程

```
1. 在 BACKLOG.md 或 CURRENT_STATUS.md 添加紧急事项
   - 标记为 P0 或 🔴
   - 说明紧急原因
   ↓
2. 立即认领并开始工作
   - 通知其他助手（在备注中说明）
   ↓
3. 快速修复 + 快速验证
   ↓
4. 更新所有相关文件
```

## 5. 更新规范

### 5.1 通用规则

1. **每次更新都要标注**：更新时间 + 更新者
2. **追加而不是覆盖**：日志类文件追加新记录，不删除旧记录
3. **状态变更要说明原因**：从 A 状态变到 B 状态，要写清楚为什么
4. **使用统一的格式**：遵循各文件的模板格式
5. **及时更新**：完成工作后立即更新黑板，不要等

### 5.2 命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 任务ID | `P<优先级>-<简称>` | `P1-API-TIMEOUT` |
| 测试ID | `TEST-<任务ID>-<序号>` | `TEST-P1-API-001` |
| 部署ID | `DEPLOY-<任务ID>-<序号>` | `DEPLOY-P1-API-001` |
| 助手标识 | `Local-A` / `VM-A` / `VM-B` | `Local-A` |

### 5.3 助手标识建议

- **本地助手**: `Local-A`, `Local-B`...
- **VM助手**: `VM-A`, `VM-B`...
- **其他助手**: `Cloud-A`, `CI-Bot`...

> 每个助手选择一个固定标识，保持一致性。

## 6. 冲突处理

### 6.1 任务冲突

如果两个助手同时认领同一任务：

1. **先到先得**：先在黑板上标记认领的获得任务
2. **主动避让**：后发现的助手主动切换到其他任务
3. **协作解决**：如果任务很大，可以分工协作，分别认领子任务

### 6.2 文件冲突

如果多人同时修改同一文件导致 git 冲突：

1. **优先合并**：尽量通过 git merge 解决
2. **保留双方信息**：不要删除对方的记录
3. **时间顺序**：按时间顺序排列记录
4. **标注差异**：如有不一致，标注出来供讨论

## 7. 每日协作检查清单

- [ ] 开始工作前：阅读 `CURRENT_STATUS.md` 了解全局
- [ ] 认领任务：更新 `ACTIVE_TASK.md` 标记认领
- [ ] 完成工作：更新对应文件记录结果
- [ ] 结束工作前：更新 `CURRENT_STATUS.md` 同步最新状态
- [ ] 遇到问题：标记 🔴 阻塞，说明问题和需要的帮助
