## 1 · 候选标签（3 个版本 + 推荐）

### 候选 A（**推荐 · 主推**）

> **"把 LLM 装进流水线 —— 让它像工人一样按工序、按质检、按预算写小说。"**

**优点**：
- **直观**：流水线 / 工人 / 工序 / 质检 / 预算 —— 5 个具象词，面试官秒懂
- **覆盖核心**：5 Agent（工序）+ 混合质量门（质检）+ 成本治理（预算）
- **有记忆点**：不是技术堆砌，是**类比** —— 类比比抽象好记

**适合场景**：
- 面试开场 30 秒版
- 简历项目描述
- 任何"1 句话介绍"场景

---

### 候选 B（**技术深度版 · 给技术面官**）

> **"Multi-Agent 协同 + 11 维质量门 + 真实成本治理 —— 一个工业化 LLM 应用的工程实践。"**

**优点**：
- **技术含量高**：3 个关键词都是技术点（Multi-Agent / 质量门 / 成本治理）
- **覆盖完整**：5 Agent + S1-S11 + cost_tracker.db 全在

**缺点**：
- **抽象**：面试官要"翻译"一遍才懂
- **不够记忆点**：技术堆砌，没有"画面感"

**适合场景**：
- 技术深讲开场
- 和技术面官单独聊
- 当面试官已经知道 Agent 是什么时

---

### 候选 C（**业务价值版 · 给产品 / 业务面官**）

> **"8 本量产小说的工业化生产线 —— 让 AI 写小说从'调 API'变成'批量生产'。"**

**优点**：
- **业务导向**：8 本量产、批量生产 —— 业务面官秒懂
- **对比鲜明**：调 API（1-shot） vs 批量生产（pipeline）

**缺点**：
- **不显技术深度**：技术面官会觉得"这是产品介绍"
- **掩盖工程价值**：5 Agent / 状态机 / 质量门都被淡化了

**适合场景**：
- 产品 / 业务面官
- 简历"项目价值"描述
- 演讲开场

## 3 · 完整话术（推荐组合）

### 3.1 面试开场 30 秒版（**主推**）

> **"我从 0 到 1 搭了一套工业化小说生产系统，叫**灵文**。如果让我用一句话记住这个项目——**这是把 LLM 装进流水线的工程实践，让它像工人一样按工序、按质检、按预算写小说**。** 整套系统 5 个专职 Agent + 11 维质量门 + 真实成本治理，已经量产 8 本 10 章样章，7 本零硬错。"**

### 3.2 技术深讲开场 60 秒版

> **"灵文是一个 Multi-Agent + 11 维质量门 + 真实成本治理的工业化 LLM 应用。** 5 个 Agent 协作（大纲 / 角色 / 写作 / 审核 / 润色），混合质量门（S1-S8 规则硬验证 + S9-S11 LLM 软评分），5 层真实 token 跟踪 + per-tier budget + cost cap 熔断。** 类比成工厂：Agent 是工人，Prompt 是工位说明，质量门是质检工序，状态机是流水线节拍。** 已经量产 8 本样章，3825 测试全绿。"**

### 3.3 业务 / 产品开场 30 秒版

> **"灵文是一个 8 本量产小说的工业化生产线。** 传统写小说是手工作坊——1 章 4-8 小时、跨章一致性靠人脑、伏笔回收靠作者自觉。** 我们用 AI 把它变成批量生产——1 章 5-8 分钟、跨章一致性自动化、伏笔自动跟踪。** 单章成本 ~$0.063，比人工便宜 1000 倍。"**

### 3.4 面试收尾 30 秒版

> **"做灵文教会我一件事：LLM 工程的核心不是让模型更聪明，而是让系统更可靠。** 评估、门控、可观测 —— 这些'约束'比'生成'重要 10 倍。** 如果让我重来一次，我会早期就做评估基建，早期就做 CVG，早期就定义状态机铁律。这 3 件事放到第一位，能省 6 个月。** 谢谢。"**

## 4 · 标签的"为什么"（**讲技术时用**）

### 4.1 为什么是"流水线"不是"工作流"？

- **流水线（Assembly Line）**：强调**分工 + 节拍 + 质检**
- **工作流（Workflow）**：强调**步骤 + 顺序**

流水线比工作流更适合灵文：
- **分工**：5 Agent 各司其职
- **节拍**：状态机铁律守门
- **质检**：11 维质量门

### 4.2 为什么是"工人"不是"专家"？

- **工人**：给工位 / 给标准 / 给质检
- **专家**：自主决策 / 自我管理

工人比专家更适合 LLM：
- LLM 现在没有真正的自主决策能力（详 `06-reflections.md` 颠覆 1）
- LLM 需要外部约束（角色卡 / 质量门 / 状态机）
- LLM 像工人一样，**不是老板**

### 4.3 为什么是"装进"不是"接入"？

- **装进**：成为流水线的一部分，**深度集成**
- **接入**：作为外部工具，**松散耦合**

"装进"比"接入"更适合灵文：
- LLM 不是外挂工具，是流水线核心
- 5 Agent 每个都重度用 LLM
- 状态机 / 质量门 / 成本治理都围绕 LLM 设计

## 6 · 标签使用纪律

### 6.1 不要这样说（**反面例子**）

```
"灵文是一个基于 LangChain 的多 Agent 网文生成系统"
"灵文用 GPT 写小说"
"灵文是一个 LLM 应用"
```

**WHY 不好**：
- "基于 LangChain" → 暗示你只是调用层（**错**：我们手写）
- "用 GPT" → 太泛，没有技术深度
- "LLM 应用" → 没有记忆点

### 6.2 推荐这样说（**正面例子**）

```
"把 LLM 装进流水线——让它像工人一样写小说"
"Multi-Agent + 11 维质量门 + 真实成本治理的工业化实践"
"8 本量产小说的工业化生产线"
```

**WHY 好**：
- **有画面感**：流水线、工人、量产
- **有数字**：8 本、11 维、~50×
- **有对比**：调 API vs 批量生产
