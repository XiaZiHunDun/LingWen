# 08 · 灵文 演示脚本（白板画图版 + UI 截屏版）

> **用法**：面试时演示用。**强烈推荐白板版**——这是面试官最看重的"硬功夫"。
> **核心**：白板画图 > PPT 翻页 > 代码演示。**5 分钟白板 > 10 分钟 PPT**。

---

## TL;DR

**演示策略选择**：

| 场景 | 推荐 | 时长 |
|---|---|---|
| **面试 30 min** | 白板版（5 min） | 5 min |
| **面试 15 min** | 白板版精简（3 min） | 3 min |
| **远程 / 视频** | UI 截屏版 + 录屏 | 3 min |
| **没有白板** | 纸笔画图（核心 3 步） | 2 min |

---

## 1 · 白板画图版（**强烈推荐 · 5 min**）

### 准备工作
- 1 块白板（**不要用 PPT**）
- 3 种颜色笔（蓝 / 红 / 黑）
- 5 分钟计时

### Step 1 · 画 5 个 Agent 方框（30s）

```
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│  outline_    │ │  character_  │ │  content_    │
│  master      │ │  designer    │ │  writer      │
│              │ │              │ │  作家 A-J    │
└──────────────┘ └──────────────┘ └──────────────┘

┌──────────────┐ ┌──────────────┐
│  auditor     │ │  polisher    │
│  审核员 A-J  │ │  读者 A-T    │
└──────────────┘ └──────────────┘
```

**话术**：
> "灵文有 5 个专职 Agent——大纲、角色、写作、审核、润色。**每个 Agent 是独立可替换的零件**——这就是 Multi-Agent 职责分离的核心。"

---

### Step 2 · 画 MasterController + GoT Router（60s）

```
        ┌─────────────────┐
        │ MasterController│
        │   3 铁律守门    │
        └────────┬────────┘
                 ↓
        ┌─────────────────┐
        │   GoT Router    │
        │  12 SCENARIOS   │
        │   解耦          │
        └────────┬────────┘
                 ↓
        ┌────────┴────────┐
        ↓         ↓        ↓
    [5 Agent]   [角色池切换]   [状态机]
```

**话术**：
> "5 Agent 不是乱跑的——中间有个 GoT Router 做调度，**预定义 12 个场景**，每场景独立 handler。**这为什么不用 LangChain ReAct？** 因为 ReAct 是'思考-行动'循环，对网文这种**多步 pipeline 不稳定**——3 次跑出 3 个大纲。我们要的是**预定义图，不是推理**。"

---

### Step 3 · 画 3 个 SQLite DB（30s）

```
┌─────────────────────────────────────────┐
│ workflow.db  │ cost_tracker.db │ ripple.db │
│ 状态机       │ 5 层 token      │ CVG 涟漪  │
│ + checkpoint │ per-tier budget │ cascade   │
└─────────────────────────────────────────┘
```

**话术**：
> "基础设施层有 3 个 SQLite——状态、成本、涟漪各一个。**为什么不 1 个？** 关注点不同——状态机表、成本表、涟漪表 schema 差异大，独立备份 / 迁移更方便。**gitignored**——单机零运维，CI 自动 init。"

---

### Step 4 · 画 11 维质量门（90s）—— **最有意思的一步**

```
        ┌─────────────┐
        │ 章节生成完成 │
        └──────┬──────┘
               ↓
        ┌─────────────┐     ┌──────────┐
        │ S1-S8 规则  │ →   │ 0 硬错？  │
        │ 硬验证      │     └────┬─────┘
        └──────┬──────┘          ↓ Yes
               ↓ No              │
        ┌─────────────┐          │
        │ 失败 → 返工 │          │
        └─────────────┘          ↓
                        ┌─────────────┐
                        │ S9-S11 LLM  │
                        │ 软评分      │
                        └──────┬──────┘
                               ↓
                        ┌─────────────┐
                        │ ≥ 阈值？    │
                        └──────┬──────┘
                               ↓ Yes
                        ┌─────────────┐
                        │ 入库 + 下游 │
                        └─────────────┘
```

**话术**：
> "这是灵文**最得意**的设计——11 维质量门。**为什么是 8 规则 + 3 LLM？** 因为**规则挡硬错**（0 误报、确定性），**LLM 评软伤**（语义判断、规则挡不住）。**这是'硬底线 + 软评分'的工程折中**。如果 LLM 挂了——自动切规则引擎，**生产不中断**。"

---

### Step 5 · 画数据流 + 收尾（60s）

```
用户输入：题材 / 角色名 / 章数
   ↓
[入口层] CLI / Dashboard
   ↓
[编排层] MasterController → GoT Router
   ↓
[执行层] 5 Agent 协作（角色池切换）
   ↓
[能力层] 11 维质量门 + RAG + WorldSnapshot
   ↓
[基础设施] SQLite + AI Service + Qdrant
   ↓
输出：合格章节正文 + 评分报告 + 成本账单
```

**话术**：
> "完整链路：用户输入 → MasterController 调度 → 5 Agent 协作 → 11 维质量门把关 → 合格章节入库。**单章 5-8 分钟、~$0.063、7 本零硬错。** 这就是'把 LLM 装进流水线'——给工位、给标准、给预算、给质检。"

---

### 白板版 Tips（**关键**）

| Tip | 说明 |
|---|---|
| **画图先于说话** | 5 步画完再讲，**不要边画边讲**——面试官会被你的图吸引 |
| **3 颜色够用** | 蓝（架构）/ 红（质量门）/ 黑（标注）——不要超过 3 色 |
| **边画边标数字** | "8 本样章"、"3825 测试"、"~$0.063"——数字标记在图上 |
| **画错不擦** | 画错了在旁边画新的——面试官看你**思考过程** |
| **5 分钟硬限** | 超时就停——"时间有限，我们进问答"——**留追问时间** |

## 2 · UI 截屏版（**远程 / 视频面试 · 3 min**）

### 准备工作
- Dashboard 已启动：`python dashboard/app.py` + `pnpm dev`
- 浏览器 5 个 tab 准备好（按顺序切）
- 录屏备份（防止现场卡壳）

### Step 1 · Studio 多书管理（30s）

**操作**：
- 打开 `http://localhost:5173/?nav=studio`
- 展示 8 本样章（静海 / 暗夜 / 铁道 / ...）

**话术**：
> "这是灵文 Studio——8 本样章。每本 10 章，7 本 P0=0。**整套系统支持多书并行管理**——这在传统'调 API 写 prompt'框架下做不到。"

---

### Step 2 · 七样章 full-check 面板（60s）

**操作**：
- 切到"全检" tab
- 展示 7/7 ≥ 4.0

**话术**：
> "这是 full-check 报告——LLM judge 5 维评分。**7/7 ≥ 4.0（满分 5）**。注意右下角——单章成本 ~$0.063、耗时 5-8 分钟。**比人工 4-8 小时提了 50 倍。**"

---

### Step 3 · 成本趋势 mini chart（45s）

**操作**：
- 切到"成本" tab
- 展示 7 天 / 30 天趋势

**话术**：
> "这是成本趋势——5 层 token 真实跟踪。**注意不是估算**——`cost_tracker.db` 持久化，每层都记录。**per-tier budget + cost cap 熔断**——超预算自动降级到 cheap tier。"

---

### Step 4 · 涟漪面板 CVG 影响图（45s）

**操作**：
- 切到"涟漪" tab
- 展示 CVG cascade + 影响图

**话术**：
> "这是跨卷涟漪面板——CVG cascade + 影响图。**Phase 9.10-9.19 的工程成果**——weighted BFS 找下游涟漪，cascade_runs 持久化每次跑，**涟漪出问题可回滚**。"

---

### UI 版 Tips

| Tip | 说明 |
|---|---|
| **5 tab 提前开** | 不要现场加载——演示流畅度 > 内容完整 |
| **录屏备用** | 现场卡了直接放录屏——别硬撑 |
| **不要 demo 全部** | 3 个核心页面（Studio / 成本 / 涟漪）够了 |
| **时间紧就 demo 1 个** | full-check 面板最有说服力——只看这个 |

## 3 · 演示应对 FAQ（**被追问时用**）

### "这套架构你怎么想到的？"

参考 `09-faq.md` Q15。**核心**：
> "早期 v7.0 想做'通用 Agent 平台'，试了 3 个月，发现**业务专精才走得远**。网文场景的核心是**长流程 + 角色定制 + 状态可观测**——所以才有 5 Agent 职责分离、12 SCENARIOS 解耦、3 SQLite 隔离。"

### "为什么不直接 demo 写一章？"

参考 `09-faq.md` Q11。**核心**：
> "写一章要 5-8 分钟——演示不可能等。**Demo 状态可观测比 Demo 生成更说明问题**。你看 dashboard 看到的是 8 本书 80 章的运行结果，**这就是 production track record**。"

### "这套系统能扩展到别的领域吗？"

参考 `09-faq.md` Q11。**核心**：
> "八本书、八种题材（沙漠 / 沿海 / 高山 / 喀斯特 / 铁路 / 雪线 / 都市悬疑 / 心理）已经验证 Studio 模式可复制。**抽象层在 `infra/agent_system/`**——换 `outline_master` 的 prompt 模板即可，5 Agent 骨架不动。"
