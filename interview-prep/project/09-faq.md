# 09 · 面试 FAQ（16 个高频追问）

> **用法**：面试前过一遍，每个回答能 30 秒口述清就过；说不清就重写。
> **回答原则**：决策 + WHY + 反例/数字/trade-off，不堆术语。

---

## Q1 · 为什么不用 LangChain / AutoGen 这种现成 Agent 框架？

**30s 答**：
- **决策**：纯手写 5 Agent + MasterController + 自研 `infra/ai_service/` 抽象层
- **WHY**：网文场景需要深度定制（角色池 / 12 SCENARIOS / 中文工具描述），LangChain 抽象会泄漏
- **具体坑**：
  1. LLM 抽象泄漏：换 provider 时 prompt 模板变量对不上
  2. Agent 抽象过重：内置 ReAct 不支持角色池切换
  3. 工具描述：早期版本 tool description 硬编码英文，中文不适配
- **代价**：开发慢 2-3 个月，但**可控性 / 可观测性 / 中文场景**全胜

**追问 · 那什么时候用 LangChain 合适？**
答：1-shot 简单 RAG、原型验证。**一旦进入"长流程 + 角色定制 + 状态可观测"，手写 > 框架**。

## Q2 · 怎么评估 Agent 的回答质量？

**45s 答**：
- **3 道关**：
  1. **规则硬验证**（S1-S8）：剧情完整 / 逻辑自洽 / 文笔 / 节奏等 8 维，**0 误报优先**
  2. **LLM 软评分**（S9-S11）：角色一致性 / 伏笔回收率，规则挡不住的"软伤"
  3. **LLM judge**（人工盲测）：7 书 ≥ 4.0/5（满分 5）
- **认知升级**：**评估本身的稳定性比模型能力更影响生产**
- **坑**：LLM 评自己写的内容偏高 → 必须有规则门做"硬底线"

**追问 · LLM-as-Judge 有偏见怎么办？**
答：用"反例库"——构造一批已知 bad case，看 judge 能不能识别。我们跑了 49 个 case 的离线测试（`tests/consistency/test_llm_enhanced/`）。

## Q3 · Token 成本怎么控制？

**45s 答**：
- **5 层真实跟踪**：`provider → router → AgentBase → MasterController → got_bridge`，每一层都记录真实 token，不估算
- **3 道治理**：
  1. **per-tier budget**：每个 tier（cheap/mid/premium）单独预算
  2. **cost cap**：单章 $X 熔断，超了降级到 cheap tier
  3. **budget calibrate**：基于历史 batch 自动校准（DoD D 3 章 ~$0.19 → 反推 ~$0.063/章）
- **数字**：默认 F79 budget **~$0.028/章**，Studio MiniMax 实测 **~$0.063/章**

**追问 · 为什么不直接用 GPT-4？**
答：成本差 10×。MiniMax 在中文网文场景质量够用 + 成本可控。如果用 GPT-4，**单章 $0.3+**，8 本样章 80 章就是 **$24**，没有商业可持续性。

## Q4 · 怎么处理 LLM 幻觉（编造工具或参数）？

**30s 答**：
- **3 道防线**：
  1. **JSON Schema 强校验**：`parse_response` 剥离 markdown fenced JSON，schema 验证
  2. **工具白名单**：LLM 只能调用预定义工具，名字不在白名单直接拒
  3. **重试 + Fallback**：失败重试 3 次 → 降级规则引擎保底
- **效果**：8 本样章跑下来 0 次"调用不存在的工具"

**追问 · 如果工具调用格式错乱怎么办？**
答：JSON 解析重试 + 报错注入 prompt（"上次的 JSON 格式错了，请重新输出"）。我们在 `AgentBase.parse_response` 里实现了完整链路。

---

## Q5 · Agent 死循环无限调用工具怎么办？

**20s 答**：
- **`max_iterations=5` 硬限**（每 Agent 单独配置）
- **每步计费熔断**：cost cap 触发即停
- **状态机铁律**：`workflow_validator.py` 禁止非法状态转换，物理上断死循环

**追问 · 那 5 步不够怎么办？**
答：分场景调——`chapter_writing` 走 8 步（要迭代改稿），`chapter_review` 走 3 步（要快速否决）。配置项在每个 agent 的 `max_iter`。

## Q6 · 为什么不微调模型？

**30s 答**：
- **成本 / 数据量 / 效果**都不划算：
  1. 数据：8 本样章 ~80 章，训练量不够
  2. 成本：fine-tune 一版 ~$1000+，8 本样章总成本才 ~$5
  3. 效果：prompt + 角色池 + 质量门组合拳已经拿到 7/7 ≥ 4.0
- **判断**：微调的边际收益 < 成本，**不值得**

**追问 · 什么情况下会微调？**
答：模型基础能力不够（比如中文古文生成明显差），且 prompt 优化到顶还不行。

## Q7 · Multi-Agent 之间怎么通信？

**30s 答**：
- **不走消息队列**（Kafka / Redis Pub/Sub）
- **走共享 state machine**：
  - `workflow.db` 存 step + checkpoint
  - Agent 之间通过 SQLite 读写交接（不是内存 RPC）
- **WHY**：单进程多 Agent，没有分布式需求；SQLite 事务 + gitignored 隔离 = **简单可控**

**追问 · 多进程 / 分布式怎么办？**
答：当前规模不需要。如果扩到 SaaS 多租户，会拆 `workflow.db` 走 Postgres + Redis。

---

## Q8 · 怎么保证跨章角色一致性？

**60s 答**（**这是最有意思的一个**）：
- **3 层防护**：
  1. **角色卡知识图谱**（character_designer 生成）：6 维（性格 / 语气 / 能力 / 知识 / 禁忌 / 反差），存为 JSON
  2. **Rule-based 检测器**（PersonalityChecker / CharacterChecker）：性格词 / opposite 词冲突检测
  3. **LLM 增强检测**（Phase 9.11 合并）：规则挡不住时调 LLM 做语义判断
- **跨章一致性关键不是单章质量，是"约束传播"**：
  - WorldSnapshot（N² 矛盾检测，0.03ms/scan）：检查 "LOCATED_IN_DESTROYED" 这类状态矛盾
  - 角色卡是知识图谱，新章写之前先 RAG 检索历史章
- **数字**：8 本样章跨章一致性问题 **0 例**

**追问 · 遇到过最难的 bug？**
答：ch241-300 PersonalityChecker 集中报错——根因是它**只接 dict 格式**，但 character_profiles 实际有 3 种格式（list / dict / object）。修复就是**兼容性归一化**（`if isinstance(raw, list): ... elif isinstance(raw, dict): ...`）。这种细节框架不会帮你做。

## Q9 · 数据隐私 / 客户数据怎么处理？

**30s 答**：
- **离线优先**：Qdrant 走 NoOp 降级（v8.1+），无第三方上传
- **本地存储**：3 个 SQLite 全部 gitignored，落在 `infra/.state/`
- **API key**：从 env var 读，不入代码库（**`0 改 .env`** 是项目铁律）
- **生产风险**：如果上 SaaS 多租户，必须拆 Postgres + 加密 + 审计日志

## Q10 · CI 怎么保证 3825 测试不变成"挂件"？

**30s 答**：
- **6 个 workflow**：test 主门 + dashboard-frontend-ci + llm-check（路径过滤）+ golden-set matrix + e2e-live + manual
- **关键设计**：
  1. **Golden Set matrix**：7 本样章 ch001/005/010 都跑回归
  2. **llm-check 路径过滤**：改 `projects/**` / `infra/**` 或 label `llm-check` 才跑（避免每次 push 都烧 API）
  3. **pre-commit hook**：本地 ruff + vitest single run，5 秒级
- **结果**：push 后 30 分钟内知道全绿 or 哪里炸

**追问 · 3825 测试会不会太重？**
答：pytest 90s + dashboard pytest 77s + vitest 5s = **可接受范围**。拆 Phase 是关键——别一次性堆 3000 测试，**每个 Phase 加 100-200 测试**。

## Q11 · 怎么扩展到别的领域（比如剧本 / 短视频脚本）？

**30s 答**：
- **Studio 模式已验证**：八本书、八种题材（沙漠 / 沿海 / 高山 / 喀斯特 / 铁路 / 雪线 / 都市悬疑 / 心理）都是同一套 pipeline 跑通
- **抽象层在 `infra/agent_system/`**：换 `outline_master` 的 prompt 模板即可，**5 Agent 骨架不动**
- **创作者产品线**（v1-v7 已落地）：人主笔的"陪伴" + "推进"模式，证明骨架既能全自动也能半自动

**追问 · 什么领域不适合？**
答：需要**强实时反馈**的（对话 / 游戏 NPC）——我们这套是批处理 pipeline，不是流式。

## Q12 · 这个项目遇到的最大挑战是什么？

**45s 答**（**这是反思题，重点答**）：
- **早期最大挑战：跨章一致性**——LLM 单章写得好，跨章就崩（角色性格漂移 / 伏笔忘记 / 时间线混乱）
- **解决方案演进**：
  - v7.0 之前：靠 prompt 强约束 → 不稳定
  - v8.0：加角色卡 + RAG → 改善但仍有
  - v9.11：**LLM 增强检测器**（规则 + LLM 混合）→ 突破
  - **Phase 1.1 WorldSnapshot**：N² 矛盾检测，**0.03ms/scan 300× 优于目标**
- **认知**：**长程依赖是 LLM 最大的工程难题**，比"生成质量"难 10×

## Q13 · 未来方向？

**30s 答**：
- **短期（v12+）**：创作者产品线深化（陪伴 / 推进 / Studio 三模式），覆盖从"全自动"到"人主笔"的连续谱
- **中期**：多模态（封面 / 插图 / 配音）—— LLM 不只写文本，要做完整内容
- **长期**：把"流水线"开放给第三方作者——**Studio SaaS 化**

**追问 · 不做 Agent 平台吗？**
答：**不**。早期试过想做"通用 Agent 平台"，发现**业务专精才走得远**。"通用"是别人的战场（Coze / Dify），我们做**网文场景的深度**。

## Q15 · 如果重来一次，你会怎么做？

**45s 答**（**这是 L 栏核心问题**）：
1. **早期就做评估基建**——不要后期补。我 v7.0 才开始有规则门，浪费了 v6.0 之前的所有 LLM 生成内容（无法量化）
2. **CVG 不要等到 Phase 9.10**——跨卷涟漪应该在 v1 就设计，否则写到第 3 本书才发现前 2 本的伏笔断了

4. **Mermaid 文档化前置**——架构图画清楚能省 30% 的会议沟通成本

**追问 · 还有什么不做的？**
答：**不微调模型**——再选一次还是不微调。Prompt + 角色池 + 质量门组合拳够用。

## Q16 · 你在这个项目里最得意的 1 个设计决策是什么？

**60s 答**（**这是"区分度"问题**）：
- **混合质量门（S1-S11）**——这个设计把"评估"从"事后人工"变成"事中自动化"
- **为什么得意**：
  1. **业务驱动**：网文质量门是用户痛点，不是"为了做技术而做"
  2. **可演化**：从规则 8 维扩到规则 + LLM 11 维，没破架构
  3. **可降级**：LLM 挂了规则顶上，生产不中断
- **类比**：就像工厂的 QC 工序——质量不是工人（LLM）自己保证的，是流水线（系统）保证的

**追问 · 这个决策最难的是什么？**
答：把"软质量"（文笔 / 情感）拆成可验证维度。早期想"全靠 LLM judge"，发现**偏见 + 慢 + 贵**，**规则 + LLM 混合才是工程正解**。
