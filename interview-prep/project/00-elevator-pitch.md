## 1 分钟版（自我介绍版 · ~250 字）

### 项目（一句话定位）

灵文——**工业化小说生产系统**，从 prompt 到出版的"流水线"。

### 我的角色（**重中之重**）

**0-1 架构 + 全栈主程**。
- 5 个核心 Agent 全部亲手设计实现
- 不止调 API：从状态机、质量门、成本治理全套都做了
- 不是"调 API 搭工作流"那种调用层工程师

### 技术架构（四块肌肉）

| 模块 | 设计 |
|---|---|
| **Multi-Agent + GoT 路由** | 5 Agent + 12 SCENARIOS 解耦，每场景独立 handler |
| **混合质量门** | S1-S8 规则硬验证 + S9-S11 LLM 软评分 |
| **真实成本治理** | 5 层 token 跟踪 + per-tier budget + cost cap 熔断 |
| **状态机** | 3 个 SQLite（workflow / cost / ripple）+ 三条铁律 |

### 量化结果（一组硬数字）

- 单章 **4-8 小时 → 5-8 分钟**（提速 ~50×）
- 单章成本 **~$0.028-$0.063**（真实跟踪，不估算）
- **8 本样章 · 7 本 P0=0** · LLM judge 7/7 ≥ 4.0
- **3825 测试全绿**（3274 pytest + 359 dashboard + 192 vitest）

## 3 分钟版（深度开场版 · ~500 字）

### 第一段 · 项目 + 我的角色（30s）

我从 0 到 1 搭了一套**工业化小说生产系统**——灵文。
- **不是**"调 GPT API + 写 prompt 模板"
- **而是**5 个专职 Agent（大纲 / 角色 / 写作 / 审核 / 润色）+ 状态机 + 质量门 + 成本治理的全套工程
- **结果**已量产 8 本 10 章样章（7 本零硬错、LLM judge 7/7 ≥ 4.0），跑了 15 个 Phase、3825 个测试

### 第二段 · 最有挑战的 1 个技术亮点（90s）—— Multi-Agent + 混合质量门

**为什么不用 LangChain 单 Agent？** 网文是多步 pipeline（大纲→角色→正文→审核→润色），单 Agent 上下文会爆炸，而且我要角色池切换。

**我的方案**：
1. **5 Agent 职责分离**：每个 Agent 是独立可替换的"零件"，而不是粘在一起的怪物
2. **GoT (Graph of Thought) 路由**：12 SCENARIOS 解耦，每场景独立 handler 函数
3. **角色池机制**：作家 A-J / 审核员 A-J / 读者 A-T 共 40 个角色，`switch_role("writer_b")` 切档

**质量门也是混合的**：
- **规则硬验证**（S1-S8）：剧情完整 / 逻辑自洽 / 文笔 / 节奏 / 可读性等
- **LLM 软评分**（S9-S11）：角色一致性 / 逻辑自洽度 / 伏笔回收
- **Fallback**：LLM 服务挂了自动切规则引擎，生产不中断

### 第三段 · 1 个反思（30s）—— 评估比生成更难

做这个项目最大的认知升级：**真正的工程价值在"评估"和"门控"，不在"生成"**。
- LLM 是流水线上的工人，必须给工位、给标准、给质检，否则就乱来
- LLM-as-Judge 也有偏见（评自己写的内容偏高），需要规则门做"硬底线"
- 跨章一致性靠 WorldSnapshot（矛盾检测 0.03ms/scan，300× 优于目标）+ 角色卡知识图谱

### 第四段 · 收尾（30s）

如果让我用一句话总结这个项目：**这是把 LLM 装进流水线的工程实践**——让它像工人一样按工序、按质检、按预算写小说，最终量产 8 本样章。
