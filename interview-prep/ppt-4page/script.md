## Slide 1 · 我是谁 + Tech Tag（纯文字）

```
灵文 (LingWen) · 工业化小说生产系统

把 LLM 装进生产线，从"作家助手"升级成"流水线"

Solo Developer · 12 周 · 359 章 · 3825 tests
```

### 讲稿（90 秒）

> "我叫 XXX，灵文是我过去 12 周 solo 做的工业化小说生产系统。
>
> 关键词是**'装进生产线'**——市面 99% 的 AI 写作工具都是单 Agent 对话框（chatbot 思路），我想验证的是**多 Agent 流水线**这条路能不能走通，能不能用 AI 一天产出 10 章万字小说、并且**质量门控得住**。
>
> 数字：12 周 solo，359 章长篇连载（已发布），3274 单元测试 + 192 前端测试全绿，0 失败。
>
> Tech Tag 我打的是 **Multi-Agent Orchestration**——不是 RAG、不是 Fine-tuning、不是 Prompt Engineering，是**怎么编排 5 个 Agent + 怎么用 GoT 解耦 + 怎么用三层兜底保证生产不挂**。"

### 关键金句
- "把 LLM 装进生产线"
- "从 chatbot 升级成流水线"
- "5 Agent + GoT + 三层兜底"

### 翻车预警

## Slide 2 · 系统全景（架构图）

### 屏显内容（PPT 上）

```
标题：5 层架构 · 解耦 + 横切共用

[图：slide-02-architecture.png · 5 层全景图]

下方 4 个 takeaway：
1. 入口 2 路（CLI + Dashboard）· 编排层 MasterController 3 铁律守门
2. 执行层 5 核心 Agent（职责分离 > 数量堆叠）
3. 能力层横切共用（质检 / 记忆 / 世界观 / 跨卷）
4. 基础设施 3 SQLite（workflow / cost / ripple · 全 gitignored）
```

### 讲稿（90 秒）

> "这是系统的 5 层架构全景图。
>
> **第 1 层 入口**：CLI 单机版 + Dashboard Web 版（Vue 3 + FastAPI），两路入口共享同一套编排逻辑。
>
> **第 2 层 编排**：MasterController 守 3 铁律——max_iterations 防死循环、cost cap 单章预算、checkpoint 崩溃可恢复。GoT Router 把 12 个场景（write/audit/polish）解耦，每场景独立 handler。
>
> **第 3 层 执行**：5 个核心 Agent——outline_master 管大纲、character_designer 管角色、content_writer 写正文、auditor 审核、polisher 润色。**为什么是 5 个不是 50 个？**因为上下文会爆炸，5 个 Agent 每个上下文干净、职责清晰、独立可测、独立可替换。
>
> **第 4 层 能力**：4 个横切共用的能力——11 维质量门、Qdrant 记忆、WorldSnapshot 矛盾检测、CVG 跨卷涟漪。**关键：能力层横切，不复制实现**——每个 Agent 都调用同一套质检，不会出现"A 的逻辑和 B 的逻辑对不上"。
>
> **第 5 层 基础设施**：3 个 SQLite 数据库，workflow 存状态、cost 存账单、ripple 存跨卷关系。全部 gitignored，单机零运维。"

### 关键金句
- "5 Agent 不是 50 Agent — 职责分离 > 数量堆叠"
- "能力层横切共用，不复制实现"
- "3 SQLite 全 gitignored，单机零运维"

### 翻车预警
- ❌ 不要从入口讲到基础设施一字不漏 → 90 秒根本讲不完，面试官会走神
- ✅ 抓住"5 / 3 / 4 / 3" 四个数字，把层次讲清楚就够

## Slide 3 · 我最骄傲的设计（质量门图）

### 屏显内容（PPT 上）

```
标题：11 维质量门 · 规则硬 + LLM 软 + Fallback 熔断

[图：slide-03-quality-gate.png · 11 维流水线]

下方 3 个 takeaway：
1. 8 规则硬底线（剧情/逻辑/文笔/情感/节奏/可读/角色/弧光）· <500ms · 确定性 · 0 误报
2. 3 LLM 软评分（角色一致性/逻辑自洽度/伏笔回收率）· ~3s · 不返工只警告
3. Fallback 熔断：LLM 服务挂 → 自动切规则引擎 · 生产不中断
```

### 讲稿（90 秒）

> "这是我最骄傲的设计——**11 维质量门**。
>
> 11 个维度不是堆出来的，是按'该不该用 LLM 判断'分成两层：
>
> **第 1 层 8 规则硬底线**：剧情完整、逻辑自洽、文笔风格、情感共鸣、节奏控制、可读性、主角魅力、人物弧光。这 8 个维度**全部用规则判断**——不是因为我不想用 LLM，是因为**质量门是兜底防线，必须确定性**。8 个规则并跑 <500ms，0 误报，**任意 1 个不过就返工**。
>
> **第 2 层 3 LLM 软评分**：角色一致性、逻辑自洽度、伏笔回收率——这 3 个用 LLM 打分，约 3 秒。**关键决策：不返工，只警告**。为什么不返工？因为 LLM 评分本身有噪声，如果硬卡阈值会造成'AI 评分不过就重写'的死循环。
>
> **第 3 层 Fallback 熔断**：LLM 服务挂了怎么办？——自动切回纯规则引擎。8 本样章 P0=0 的工程保障**正是这个 Fallback 在扛**。
>
> 这套设计回答了一个根本问题：**AI 生产线上，'不要让坏稿出门'和'不要让生产停下来'怎么同时做到？**——分层：硬底线守出门，软评分提上限，Fallback 保生存。"

### 关键金句
- "硬底线守出门，软评分提上限，Fallback 保生存"
- "8 规则确定性 <500ms · 3 LLM 软评分不返工 · Fallback LLM 挂不中断"
- "质量门是兜底防线，必须确定性"

### 翻车预警
- ❌ 不要把 11 个维度逐一念出来 → 听众根本记不住
- ✅ 只讲分层逻辑（硬/软/Fallback）+ 1 个数字（8 本样章 P0=0）

## Slide 4 · 我怎么想的（3 个认知突破）

### 屏显内容（PPT 上）

```
标题：3 个认知突破 · solo 12 周最值钱的 3 个反思

1. 职责分离 > 数量堆叠
   5 Agent 不是因为"多就是好"，是因为每个 Agent 上下文干净才能稳定。
   50 个 Agent 上下文会爆炸到失控。

2. 规则硬 + LLM 软 + Fallback 熔断
   AI 生产线的可靠性问题不能用"换更好的模型"解决，
   必须用"分层兜底"解决——硬底线确定性、软评分不阻塞、Fallback 保生存。

3. 真实成本跟踪 > 估算
   5 层 token 真实回传 + per-tier budget + cost cap 自动降级，
   不是"算一下大概多少钱"，是把每次 LLM 调用落库、按 tier 熔断、按 cap 降级。
```

### 讲稿（90 秒）

> "最后讲 3 个认知突破——这 12 周里让我改过 3 次方向、每次都被打脸的反思。
>
> **第 1 个：职责分离 > 数量堆叠**。我一开始想堆 15-20 个 Agent，觉得'分得越细越专业'。实际跑下来——**50 个 Agent 上下文会爆炸到失控**，每个 Agent 都要传完整的世界观 + 角色卡 + 历史摘要，token 直接爆掉。最终收敛到 5 个核心 Agent，**不是因为'5 这个数字好'，是因为 5 是上下文大小和职责颗粒度的平衡点**。
>
> **第 2 个：规则硬 + LLM 软 + Fallback 熔断**。AI 生产线的可靠性问题**不能用'换更好的模型'解决**——GPT-5 也会挂、Claude 也会限流、价格也会变。**必须用'分层兜底'解决**：硬底线用规则保证确定性、软评分用 LLM 提上限但允许噪声、Fallback 在 LLM 挂时切回纯规则保生产。这是 slide 3 那张图的深层逻辑。
>
> **第 3 个：真实成本跟踪 > 估算**。前 4 周我用 token 计数估算成本，跑了 2 周发现**实际账单比估算高 40%**——因为 prompt cache、function call、tool use 这些都不在估算里。第 5 周起改成 5 层 token 真实回传落库 + per-tier budget 熔断 + cost cap 触发自动降级 cheap tier。**不是算一下大概多少钱，是把每次调用当金融交易记账**。
>
> 这 3 个反思加起来就是一句话——**'做 AI 系统的可靠性工程，不要赌模型，要兜底线'**。"

### 关键金句
- "做 AI 系统的可靠性工程，不要赌模型，要兜底线"
- "5 是上下文大小和职责颗粒度的平衡点，不是'5 这个数字好'"
- "把每次 LLM 调用当金融交易记账"

### 翻车预警
- ❌ 不要写成读后感/总结报告 → 立刻显得浅
- ✅ 每个突破都要带一个"我一开始想错了什么 → 怎么改的"的反转故事
