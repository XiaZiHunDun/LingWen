讲法提示（按画图顺序念出来）
================================================================================

Step 1 · 画 5 个方框（30s）
  "灵文有 5 个专职 Agent——大纲、角色、写作、审核、润色。每个 Agent 是
  独立可替换的零件——这就是 Multi-Agent 职责分离的核心。"

Step 2 · 画 MasterController + GoT Router（60s）
  "5 Agent 不是乱跑的——中间有个 GoT Router 做调度，预定义 12 个场景，
  每场景独立 handler。这为什么不用 LangChain ReAct？因为 ReAct 是
  '思考-行动'循环，对网文这种多步 pipeline 不稳定——3 次跑出 3 个大纲。
  我们要的是预定义图，不是推理。"

Step 3 · 画 3 个 SQLite（30s）
  "基础设施层有 3 个 SQLite——状态、成本、涟漪各一个。为什么不 1 个？
  关注点不同——状态机表、成本表、涟漪表 schema 差异大，独立备份 / 迁移
  更方便。gitignored——单机零运维，CI 自动 init。"

Step 4 · 画 11 维质量门（90s · 最有意思的一步）
  "这是灵文最得意的设计——11 维质量门。为什么是 8 规则 + 3 LLM？
  因为规则挡硬错（0 误报、确定性），LLM 评软伤（语义判断、规则挡不住）。
  这是'硬底线 + 软评分'的工程折中。如果 LLM 挂了——自动切规则引擎，
  生产不中断。"

Step 5 · 收尾（30s）
  "完整链路：用户输入 → MasterController 调度 → 5 Agent 协作 →
  11 维质量门把关 → 合格章节入库。单章 5-8 分钟、~$0.063、7 本零硬错。
  这就是'把 LLM 装进流水线'——给工位、给标准、给预算、给质检。"

================================================================================
被追问时
================================================================================

Q · 为什么不直接 demo 写一章？
A · 写一章要 5-8 分钟——演示不可能等。Demo 状态可观测比 Demo 生成更
  说明问题——8 本样章就是 production track record。

Q · 这套架构你怎么想到的？
A · 早期 v7.0 想做'通用 Agent 平台'，试了 3 个月，发现业务专精才走得远。
  网文场景的核心是长流程 + 角色定制 + 状态可观测。

Q · 能扩展到别的领域吗？
A · 八本书、八种题材已经验证 Studio 模式可复制。抽象层在
  infra/agent_system/——换 outline_master 的 prompt 模板即可，
  5 Agent 骨架不动。
