## 核心优化一：CARE框架提示词体系

### 1.1 现状分析

当前工厂的提示词分散在各方法论文档中，缺乏统一的组织框架。调研中的91写作采用了多层级提示词体系（大纲提示词/章节提示词/内容提示词），而Sudowrite则擅长描写增强。

### 1.2 CARE框架定义

CARE框架将提示词组织为四个核心要素：

| 要素 | 中文名 | 小说场景含义 | 当前工厂对应 |
|------|--------|------------|-------------|
| **C**ontext | 背景 | 当前场景设定、人物状态、世界观约束 | 03_内容仓库/阶段大纲 |
| **A**ction | 行动 | 需要生成什么类型的内容 | 06_意见仓库/审核意见 |
| **R**esult | 结果 | 质量要求、风格、结构、长度 | 评审维度体系(S1-S8) |
| **E**xample | 示例 | 期望的生成效果示例 | 核心样章验证 |

### 1.3 标准化提示词模板

## 核心优化二：伏笔追踪系统

### 2.1 现状分析

当前工厂在审核维度中未设立专门的伏笔追踪机制。调研报告指出伏笔管理包括：人物伏笔、物品伏笔、事件伏笔三种类型，需要"记录-提醒-回收验证"三阶段管理。

### 2.2 系统架构

```
06_意见仓库/
├── 05_伏笔管理/
│   ├── 伏笔登记表.md          # 伏笔记录主表
│   ├── 伏笔回收追踪.md        # 伏笔状态跟踪
│   └── 伏笔分析报告.md        # 伏笔覆盖率分析
```

### 2.3 伏笔登记表模板

```markdown
# 伏笔登记表

## 记录格式

| 伏笔ID | 类型 | 埋入章节 | 伏笔内容 | 预期回收章节 | 状态 |
|--------|------|---------|---------|------------|------|
| FP-001 | 人物 | ch005 | 角色A的神秘身世 | ch080 | 未回收 |
| FP-002 | 物品 | ch012 | 关键道具「暗皇令牌」 | ch050 | 已回收 |
| FP-003 | 事件 | ch020 | 世界观规则「灵气复苏」 | ch100+ | 进行中 |

## 伏笔类型定义

| 类型 | 说明 | 示例 |
|------|------|------|
| 人物伏笔 | 角色背景/能力/命运暗示 | 角色A小时候的特殊梦境 |
| 物品伏笔 | 关键道具的出现与消失 | 「上古遗物」首次提及 |
| 事件伏笔 | 世界观规则/历史事件 | 三十年前的「灭世之战」 |
| 情感伏笔 | 情感关系的暗示 | 角色A对角色B的微妙态度 |
| 主题伏笔 | 核心主题的铺垫 | 「什么是真正的力量」讨论 |

## 状态定义

| 状态 | 说明 |
|------|------|
| 未回收 | 伏笔已埋入，尚未在后续章节出现 |
| 进行中 | 已开始回收，但尚未完全揭示 |
| 已回收 | 伏笔已完整回收，读者可知晓全貌 |
| 已失效 | 因大纲变更，该伏笔不再适用 |
```

## 核心优化三：角色一致性检查

### 3.1 现状分析

当前工厂的 `04_人物塑造体系.md` 已建立完整的欲望-恐惧-缺陷三角和人物弧光设计，但审核S7（主角魅力）主要依赖人工判断。调研报告指出需要建立"角色特征关键词库"和"角色一致性检查工具"。

### 3.2 角色特征关键词库

```markdown
# 角色特征关键词库

### 基础属性
- 姓名：林夜
- 身份：主角/修炼者
- 核心欲望：{desire}
- 核心恐惧：{fear}
- 核心缺陷：{flaw}

### 语言风格
- 常用词汇：{word_list}
- 禁用词汇：{forbidden_words}
- 语气特征：{tone}
- 口头禅：{catchphrase}

### 行为模式
- 决策倾向：{decision_pattern}
- 冲突应对：{conflict_response}
- 情感表达：{emotional_expression}

### 成长弧光
- 初始信念：{initial_belief}
- 考验事件：{test_events}
- 最终领悟：{final_insight}

### 一致性检查点
- [ ] ch001-ch040：林夜决策是否符合「谨慎但有底线」的性格
- [ ] ch041-ch080：林夜在重大冲突中是否展现出缺陷的挣扎
- [ ] ch081+：林夜的成长是否与初始信念形成对照
```

### 3.3 自动一致性检查工具

## 核心优化四：AI服务抽象层

### 4.1 现状分析

当前工厂通过hardcoded方式调用单一AI模型。调研报告建议建立"统一的AI服务抽象层，封装不同模型提供商的API调用细节"。

### 4.2 架构设计

```

┌─────────────────────────────────────────────┐
│              调用方（Agent/审核员）           │
├─────────────────────────────────────────────┤
│              AI服务网关 (AIGateway)          │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐        │
│  │ DeepSeek │ │ Claude  │ │  千问   │  ...   │
│  └────┬────┘ └────┬────┘ └────┬────┘        │
│       │           │           │              │
│  ┌────┴───────────┴───────────┴────┐        │
│  │       模型适配器 (ModelAdapter)    │        │
│  └────────────────┬──────────────────┘        │
│                   │                           │

│  │       统一响应格式 (AIResponse)    │        │

└─────────────────────────────────────────────┘
```

### 4.3 核心接口定义

#### 4.3.1 AI服务网关接口

```yaml
AI服务网关接口:
  名称: AIGateway
  方法:
    - name: generate
      参数:
        - prompt: string (CARE格式提示词)
        - model: enum (deepseek/claude/qwen/minimax)
        - parameters:
            temperature: float [0.0-1.0]
            max_tokens: int
            top_p: float
        - context: object (可选上下文)
      返回: AIResponse
      说明: 统一生成接口

    - name: batch_generate
      参数:
        - prompts: string[]
        - model: enum
        - parallel: boolean
      返回: AIResponse[]

    - name: switch_model
      参数:
        - from_model: enum
        - to_model: enum
      说明: 运行时切换模型

    - name: get_model_status
      返回: ModelStatus[]
      说明: 获取各模型健康状态
```

#### 4.3.2 统一响应格式

```yaml
AIResponse:
  success: boolean
  content: string (生成内容)
  model: string (实际调用的模型)
  tokens_used: int
  latency_ms: int
  error: string (失败时错误信息)
  fallback_used: boolean (是否使用了降级模型)
```

### 4.5 熔断降级机制

```markdown
## 熔断降级规则

### 触发条件
1. **延迟熔断**：单次请求超过10秒
2. **错误熔断**：连续3次请求失败
3. **成功率熔断**：最近100次请求成功率<80%

### 降级流程
```
请求 → AIGateway
  → 检查熔断状态
  → 选择可用模型（按场景优先级）
  → 调用模型
  → 成功 → 返回响应
  → 失败 → 尝试降级模型
  → 所有模型失败 → 返回缓存响应/报错
```

### 缓存策略
- 相同提示词的响应缓存5分钟
- 缓存键：SHA256(prompt + model + temperature)
```

## 扩展优化六：温度参数指导

### 6.1 调研发现

调研报告指出：
- Temperature 0.3-0.5：稳定输出，适合设定严格场景
- Temperature 0.8-0.9：创意输出，适合灵感激发

### 6.2 指导原则

| 创作阶段 | 建议温度 | 理由 |
|---------|---------|------|
| 大纲生成 | 0.5-0.7 | 需要逻辑性但保留创意空间 |
| 细纲扩展 | 0.4-0.6 | 遵循结构，保持一致性 |
| 正文初稿 | 0.6-0.8 | 创意发挥，突破思维定式 |
| 润色修改 | 0.3-0.5 | 保持一致性，微调为主 |
| 头脑风暴 | 0.8-0.9 | 最大化创意多样性 |

### 6.3 实施建议

在AI服务抽象层（核心优化四）基础上，增加温度参数指导层：
- 为每个CARE模板预设推荐温度
- 支持创作者自定义温度偏好
- 记录实际使用温度用于分析优化

### 6.4 工作量估算

**1周**：在提示词模板中标注推荐温度
**2周**：开发温度偏好配置界面
**总计：约3周（可与核心优化四并行）**

## 扩展优化七：ReAct架构

### 7.1 调研发现

就要创作平台采用ReAct架构（主专家+多个子专家），用户通过自然语言指令完成从构思到创作的完整流程。对比当前工厂的命令式调度，Agentic方式"更好地平衡了用户控制和自动化效率"。

### 7.2 架构对比

| 维度 | 当前工厂 | ReAct架构 |
|------|---------|----------|
| 交互方式 | 命令式调度 | 对话式引导 |
| 决策模式 | 主控预设 | 动态规划 |
| 适应性 | 固定流程 | 场景自适应 |
| 学习成本 | 较高 | 较低 |

### 7.3 潜在价值

1. **降低使用门槛**：创作者可以用自然语言描述需求
2. **增强适应性**：可根据反馈动态调整创作策略
3. **提升用户体验**：类似就要创作的对话式交互

### 7.4 实施建议

该优化依赖AI服务抽象层（核心优化四）成熟后推进。

**阶段规划**：
- 阶段1：设计主专家+子专家角色定义
- 阶段2：实现ReAct循环核心逻辑
- 阶段3：与现有22步工作流对齐
- 阶段4：用户验收测试

### 7.5 工作量估算

**8-12周**（需核心优化四完成后）
