# Phase 7.5 — polish_merge 升级到 LLM S1-S8 多维加权评分

> Doc 4 §11 Phase 7.5 (Phase 7.x polish 段,继 7.1/7.2/7.3/7.4)
> 父阶段: Phase 7.4 (max(len) 选最长) → **7.5 (LLM 评分)**

## Context

Phase 7.4 把 polish_merge 简化为 `max(contents, key=len)` — 选最长 variant 兜底。**问题**:

1. **长度 ≠ 质量** — 对话优化 variant 通常比节奏调整短(对话重写会精简冗余),max(len) 会一直偏袒 emotional_pacing variant, 跟实际质量无关
2. **没有评分透明度** — max(len) 决策不可解释,无法写回 dashboard 让用户看到"为什么选这个"
3. **浪费并行机会** — 2 个 LLM 调并行跑了 2 个 variant,结果用规则挑 1 个,LLM 投入的 token 跟决策脱钩

**本阶段目标**: polish_merge 调 LLM (HAIKU tier) 评估 2 variant 的 S1-S8 8 维分数,加权求总分,选高者。评分详情写回 NodeExecution.output, 供 dashboard 显示 (Phase 7.6 雷达图)。

**当前 baseline**: 2063 passed, 0 failed, 17 skipped (Phase 7.4 fixup 末)
**目标 baseline**: 2069 passed (6 新测试) + 0 failed, ruff 0

## 1. Feature A — MasterController.polish_merge_synthesis (S1-S8 加权评分)

### 现状

`master_controller.py` 无 S1-S8 评分方法。`polish_chapter` / `polish_emotional_pacing` / `polish_ai_trace_removal` 都返 `str`,无评分信息。

### 目标

**新方法**:
```python
def polish_merge_synthesis(
    self,
    content_a: str,
    content_b: str,
    *,
    labels: tuple[str, str] = ("A", "B"),
) -> Dict[str, Any]:
    """Phase 7.5 — LLM S1-S8 多维加权评分, 选高者
    
    Returns:
        {
            "content": str,             # winner 内容
            "winner": str,              # labels[0] or labels[1]
            "scores_a": dict[str, int], # {S1: 8, S2: 9, ..., S8: 7}
            "scores_b": dict[str, int],
            "scores_total_a": float,    # 0-10 平均
            "scores_total_b": float,
            "scores_delta": float,      # total_a - total_b
        }
    
    韧性契约: LLM 失败 → 兜底返 max(len) 选较长, scores 全 None, winner="len_fallback"
    """
```

**S1-S8 8 维** (沿用 Doc 1 审核维度):
- S1 剧情完整性 (Plot Completeness)
- S2 逻辑自洽 (Logic Consistency)
- S3 文笔风格 (Writing Style)
- S4 情感共鸣 (Emotional Resonance)
- S5 节奏控制 (Pacing Control)
- S6 可读性 (Readability)
- S7 主角魅力 (Protagonist Appeal)
- S8 人物弧光 (Character Arc)

**加权**: 等权 `weight=0.125 each`, `total = mean(s1..s8)`

### 韧性契约 (跟 Phase 7.2/7.4 模式一致)

```python
def polish_merge_synthesis(self, content_a, content_b, *, labels=("A", "B")):
    if not content_a or not content_b:
        return self._merge_synthesis_len_fallback(content_a, content_b, labels, reason="empty_content")
    if content_a == content_b:
        return {
            "content": content_a,
            "winner": labels[0],
            "scores_a": {}, "scores_b": {},
            "scores_total_a": 0.0, "scores_total_b": 0.0,
            "scores_delta": 0.0,
            "fallback": "identical",
        }
    try:
        return self._merge_synthesis_llm(content_a, content_b, labels)
    except Exception as e:
        logger.warning("polish_merge_synthesis: LLM failed, len fallback: %s", e)
        return self._merge_synthesis_len_fallback(content_a, content_b, labels, reason="llm_fail")
```

### 私有方法 `_merge_synthesis_llm` + `_merge_synthesis_len_fallback`

```python
def _merge_synthesis_llm(self, content_a, content_b, labels):
    prompt = build_merge_synthesis_prompt(content_a, content_b, labels)
    system = get_merge_synthesis_system_prompt()
    response = self.polisher.chat(  # 借用 polisher 的 chat() (AgentBase 继承)
        prompt=prompt, system=system, temperature=0.2, max_tokens=2000
    )
    parsed = self.polisher.parse_response(response, format_type="json")
    scores_a = {k: int(parsed[f"scores_{labels[0]}"][k]) for k in _S1_S8_KEYS}
    scores_b = {k: int(parsed[f"scores_{labels[1]}"][k]) for k in _S1_S8_KEYS}
    total_a = sum(scores_a.values()) / 8.0
    total_b = sum(scores_b.values()) / 8.0
    winner = labels[0] if total_a >= total_b else labels[1]
    content = content_a if winner == labels[0] else content_b
    return {
        "content": content,
        "winner": winner,
        "scores_a": scores_a,
        "scores_b": scores_b,
        "scores_total_a": total_a,
        "scores_total_b": total_b,
        "scores_delta": total_a - total_b,
    }

def _merge_synthesis_len_fallback(self, content_a, content_b, labels, *, reason):
    if not content_a:
        return {"content": content_b or "", "winner": labels[1] if content_b else "", "fallback": reason, ...}
    if not content_b:
        return {"content": content_a, "winner": labels[0], "fallback": reason, ...}
    winner = labels[0] if len(content_a) >= len(content_b) else labels[1]
    content = content_a if winner == labels[0] else content_b
    return {"content": content, "winner": winner, "fallback": reason,
            "scores_a": {}, "scores_b": {}, "scores_total_a": 0.0, "scores_total_b": 0.0, "scores_delta": 0.0}
```

**`_S1_S8_KEYS = ("S1", "S2", "S3", "S4", "S5", "S6", "S7", "S8")`** 模块级常量。

## 2. Feature B — Prompts 新增 (polisher/prompts.py)

### 新增 2 个 builder

```python
# infra/agent_system/agents/polisher/prompts.py

_S1_S8_NAMES = {
    "S1": "剧情完整性 (Plot Completeness)",
    "S2": "逻辑自洽 (Logic Consistency)",
    "S3": "文笔风格 (Writing Style)",
    "S4": "情感共鸣 (Emotional Resonance)",
    "S5": "节奏控制 (Pacing Control)",
    "S6": "可读性 (Readability)",
    "S7": "主角魅力 (Protagonist Appeal)",
    "S8": "人物弧光 (Character Arc)",
}

def build_merge_synthesis_prompt(content_a: str, content_b: str, labels: tuple[str, str] = ("A", "B")) -> str:
    """构建 polish_merge S1-S8 多维评分 prompt"""
    label_a, label_b = labels
    s1_s8_list = "\n".join(f"- {k}: {v}" for k, v in _S1_S8_NAMES.items())
    return f"""你是资深网文质量审核官, 需要对比 2 个章节润色 variant, 按 S1-S8 8 维评分 (0-10 分, 越高越好), 并选出优胜者。

## 评分维度
{s1_s8_list}

## Variant {label_a}
{content_a[:3000]}

## Variant {label_b}
{content_b[:3000]}

## 输出格式 (严格 JSON)
{{
  "scores_{label_a}": {{"S1": <int>, "S2": <int>, "S3": <int>, "S4": <int>, "S5": <int>, "S6": <int>, "S7": <int>, "S8": <int>}},
  "scores_{label_b": {{"S1": <int>, ..., "S8": <int>}},
  "reason": "<50 字以内, 解释为什么选 {label_a} 或 {label_b}>"
}}

只返 JSON, 不要其他文字。"""


def get_merge_synthesis_system_prompt() -> str:
    """polish_merge S1-S8 评分系统提示"""
    return """你是一位资深网文质量审核官, 擅长多维度对比 2 个润色版本。
你的工作: 客观评分 S1-S8 8 维, 选总平均分高者。
不要因为 variant 长度或风格偏好而偏袒, 严格按 8 维定义评估。"""
```

### 防御性: 不依赖 variant_loader

`merge_synthesis` 不需要读者偏好 (合并是中性评估), 跳过 `_load_variant_enhancement`,简化逻辑。

## 3. Feature C — got_bridge `_handler_polish_merge` 改造

### 现状 (Phase 7.4)

```python
def _handler_polish_merge(master, inputs):
    contents = []
    for upstream in inputs.values():
        if isinstance(upstream, dict) and "content" in upstream:
            contents.append(upstream["content"])
    if not contents:
        return {"_error": "polish_merge requires >= 1 upstream with content"}
    return {"content": max(contents, key=len)}
```

### 目标 (Phase 7.5)

```python
def _handler_polish_merge(master, inputs):
    """Phase 7.5: 调 master.polish_merge_synthesis, LLM S1-S8 加权选高者"""
    # 收集 2 个 variant (按 input key 顺序, 确定性)
    variants: dict[str, str] = {}
    for upstream_id, upstream in sorted(inputs.items()):
        if isinstance(upstream, dict) and "content" in upstream:
            variants[upstream_id] = upstream["content"]
    if not variants:
        return {"_error": "polish_merge requires >= 1 upstream with content"}
    if len(variants) == 1:
        # 兜底: 单 variant 直接选
        only_id, only_content = next(iter(variants.items()))
        return {
            "content": only_content,
            "winner": only_id,
            "scores_a": {}, "scores_b": {},
            "scores_total_a": 0.0, "scores_total_b": 0.0,
            "scores_delta": 0.0,
            "fallback": "single_upstream",
        }
    # 2+ variants 调 LLM
    ids = list(variants.keys())
    label_a, label_b = ids[0], ids[1]
    return master.polish_merge_synthesis(
        content_a=variants[label_a],
        content_b=variants[label_b],
        labels=(label_a, label_b),
    )
```

**返回结构**: 跟 `polish_merge_synthesis` 一致 — 透传 winner / scores_* / fallback 字段, 让 dashboard 可读。

**不破坏**: 现有 6 个 Phase 7.4 测试 (5+1) 仍 PASS — fallback="single_upstream" / 旧 max(len) 路径仍存在,新测试验证新行为。

## 9. 风险 + 缓解

| 风险 | 缓解 |
|------|------|
| LLM 评分不稳定 (同输入可能返不同 winner) | temperature=0.2 + system prompt 明确"客观评分" + 加 1 韧性兜底 (LLM 失败 → max(len)) |
| LLM JSON parse 失败 | parse_response 失败 → 兜底 max(len), reason="llm_fail" |
| S1-S8 维度对 LLM 来说可能太抽象 | 沿用 Doc 1 已有定义, dim name 跟 test_decision_integration 用的 audit 报告 schema 一致 |
| Token cost 翻倍 (新增 1 LLM) | 接受: 这是设计目标 (质量 > 成本), 用 HAIKU 控制在 ~$0.001/call |
| Stub 测试 跨 4 个文件同步 | subagent 一次性扫描 4 个 stub 文件, 全部加 polish_merge_synthesis 方法 |
| polish_merge LLM 调失败让节点 FAILED → backtrack | 韧性契约: 失败 → 兜底 max(len), 节点仍 COMPLETED, 不触发 backtrack |
| YAML version 2→3 破坏 `load_workflow` 兼容性 | load_workflow 不验证 version 字段, 任意 int 都能加载 |
| dashboard /mermaid 端点没改, version 字段不显示 | YAGNI, dashboard 看 polish_merge 节点 status 即可 |
