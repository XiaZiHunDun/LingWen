# Phase 7.4 — Multi-Variant Polish 节点 (并行多变体 + AGGREGATION 合并)

> Doc 4 §11 Phase 7.4 (Phase 7.x polish 段,继 7.1/7.2/7.3)
> 父阶段: Phase 7.1 (生产路径 bug 修复) → 7.2 (PolisherTools LLM 化) → 7.3 (novel_writing 加 polish 节点) → **7.4 (多变体并行 + AGGREGATION)**

## Context

Phase 7.3 在 `novel_writing.yaml` 中加入了**单节点** `polish_chapter` (scenario=ai_trace_removal),把 PolisherTools 3 路径(dialogue LLM + pacing LLM + remove_ai_gloss 规则)串行执行。**但生产用起来有 2 个明显不足**:

1. **优化维度太单薄** — 单一 scenario "ai_trace_removal" 调一次 master.polish_chapter 串行跑 3 路径,产物是一份被 "全方位润色过" 的章节。问题是**润色维度之间没有显式分离**:节奏没改好的话,内容已被 AI 痕迹规则洗过 → 二次返工成本高。
2. **并行机会浪费** — `emotional_pacing`(对话+节奏 LLM)和 `ai_trace_removal`(规则 + AI 痕迹检测)是**正交优化维度**,完全可以并行跑,然后用 AGGREGATION 节点按策略合并。
3. **AGGREGATION 节点已定义但未用** — `NodeType.AGGREGATION` 已在 data_structures.py:35 保留(注释 "合并多 review"),但**整个生产路径从未使用**。是写好 GoT 拓扑的关键能力。

**本阶段目标**: 1 个 phase 解决 3 个问题,1 个实施 commit + 1 个测试 commit + push。

**当前 baseline**: 2046 passed, 0 failed, 17 skipped (Phase 6.6 末)
**目标 baseline**: 2053 passed (7 新测试) + 0 failed, ruff 0

## 1. Feature 设计 — novel_writing.yaml 拓扑变更 (5 → 7 节点)

### 现状 (Phase 7.3)

```yaml
nodes:
  - read_snapshot      # INPUT
  - write_chapter      # GENERATION
  - review_chapter     # EVALUATION
  - polish_chapter     # SYNTHESIS    ← 单节点 scenario=ai_trace_removal
  - emit_chapter       # OUTPUT
```

**问题**: `polish_chapter` 把 "优化对话 + 调整节奏 + 去 AI 痕迹" 3 件事压在一个 SYNTHESIS 节点,串行执行,且**没有利用并行机会**。

### 目标 (Phase 7.4)

```yaml
nodes:
  - read_snapshot              # INPUT
  - write_chapter              # GENERATION
  - review_chapter             # EVALUATION
  # ↓ review_chapter 后分叉:2 个并行 polish 节点 (depends_on 都指向 review_chapter)
  - polish_emotional_pacing    # SYNTHESIS    scenario=emotional_pacing
  - polish_ai_trace_removal    # SYNTHESIS    scenario=ai_trace_removal
  # ↓ AGGREGATION 节点:依赖 2 个 polish 节点
  - polish_merge               # AGGREGATION  scenario=polish_merge
  # ↓ emit_chapter 依赖 polish_merge
  - emit_chapter               # OUTPUT
```

**拓扑 (mermaid 视角)**:
```
read_snapshot → write_chapter → review_chapter
                                  ├→ polish_emotional_pacing ─┐
                                  └→ polish_ai_trace_removal ─┤
                                                                ├→ polish_merge → emit_chapter
```

**ready_nodes 行为** (infra/got/graph.py:146):
- review_chapter 完成后,`polish_emotional_pacing` 和 `polish_ai_trace_removal` **同时**满足依赖条件
- 都被 `ready_nodes()` 返回(顺序按 graph 节点顺序)
- scheduler 串行执行 (`for nid in ready` infra/got/scheduler.py:195),不真并行(避免 Python GIL + 测试不稳定)
- 两者都 COMPLETED 后,polish_merge 才 ready (depends_on=[两个节点])

### 节点 detail

| 节点 | type | scenario | depends_on | token_budget | timeout_s |
|------|------|----------|------------|--------------|-----------|
| polish_emotional_pacing | synthesis | emotional_pacing | [review_chapter] | 6000 | 90 |
| polish_ai_trace_removal | synthesis | ai_trace_removal | [review_chapter] | 6000 | 90 |
| polish_merge | **aggregation** | polish_merge | [polish_emotional_pacing, polish_ai_trace_removal] | 0 | 30 |
| emit_chapter | output | (null) | [polish_merge] | 0 | 30 |

**polish_merge 节点设计要点**:
- `type=aggregation` — 不用 SYNTHESIS(语义区分:AGGR 是合并多源,SYNTH 是综合单源多步)
- `prompt_scenario=polish_merge` — 区分于 12 个 SCENARIOS,handler 单独实现
- `token_budget=0` — 纯 Python 合并,不调 LLM(避免 token 浪费)
- `timeout_s=30` — 纯 Python 操作很快

## 2. Feature 设计 — Master 新增 2 个 polish 入口 + _handler_polish 路由

### 现状

`master_controller.py:582-585`:
```python
def polish_chapter(self, content: str) -> str:
    """润色章节(委托给polisher, LLM 化路径 Phase 7.2)"""
    result = self.polisher.polish_chapter(chapter_num=0, content=content)
    return result["content"]
```

`got_bridge.py:101-109` `_handler_polish`:
```python
def _handler_polish(master, inputs):
    content = _resolve_field(inputs, "content", "")
    if not content:
        return {"_error": "content is required for polish scenarios"}
    return {"content": master.polish_chapter(content)}
```

**问题**: 3 个 scenario (`ai_trace_removal` / `emotional_pacing` / `hook_extraction`) 都调 `master.polish_chapter`,跑**完整 3 路径**(对话 LLM + 节奏 LLM + 规则)。YAML 用哪个 scenario 都一样。

### 目标

**MasterController 新增 2 个 entry method**,把 PolisherTools 3 路径拆给 2 个 variant:

```python
# master_controller.py

def polish_emotional_pacing(self, content: str) -> str:
    """情绪节奏 variant — 对话自然化 + 节奏调整 (跳过去 AI 痕迹规则,留给 ai_trace_removal 节点)"""
    polished = self.polisher.optimize_dialogue_llm(content)
    polished = self.polisher.adjust_pacing_llm(polished)
    return polished

def polish_ai_trace_removal(self, content: str) -> str:
    """AI 痕迹 variant — 规则去 AI 痕迹 + 对话自然化 (跳过节奏 LLM,留给 emotional_pacing 节点)"""
    polished = self.polisher.remove_ai_gloss(content)
    polished = self.polisher.optimize_dialogue_llm(polished)
    return polished

def polish_chapter(self, content: str) -> str:
    """完整 3 路径润色 (向后兼容, hook_extraction 节点仍走此)"""
    return self.polisher.polish_chapter(chapter_num=0, content=content)["content"]
```

**关键设计决策**:
- 两个 variant 共享 `optimize_dialogue_llm`(对话自然化对两个目标都有用)
- `polish_emotional_pacing` 偏重节奏 (`adjust_pacing_llm`)、`polish_ai_trace_removal` 偏重痕迹 (`remove_ai_gloss` 规则)
- `polish_chapter` 保留作为完整兜底(老节点和老测试不走新 YAML 不破坏)

**`_handler_polish` 改造** (got_bridge.py):
```python
def _handler_polish(master, inputs):
    content = _resolve_field(inputs, "content", "")
    if not content:
        return {"_error": "content is required for polish scenarios"}
    # Note: scenario 在 _caller 节点是固定值,不在 inputs 中
    # 路由在 AgentComputeFn.__call__ 层做,见下方修改
    return {"content": master.polish_chapter(content)}
```

**问题**: `_handler_polish` 拿不到 `node.prompt_scenario`(handler 协议只收 `master` + `inputs`)。**改造方案**:

**方案 A** (推荐): 在 `SCENARIO_HANDLERS` 注册表里把 scenario → (handler, method_name) 关联:
```python
SCENARIO_HANDLERS: dict[str, HandlerFn] = {
    # polisher
    "ai_trace_removal": _make_polish_handler("polish_ai_trace_removal"),
    "emotional_pacing": _make_polish_handler("polish_emotional_pacing"),
    "hook_extraction": _make_polish_handler("polish_chapter"),  # 兜底
    ...
}

def _make_polish_handler(method_name: str) -> HandlerFn:
    def handler(master, inputs):
        content = _resolve_field(inputs, "content", "")
        if not content:
            return {"_error": "content is required"}
        method = getattr(master, method_name)
        return {"content": method(content)}
    return handler
```

**方案 B** (备选, 不推荐): 改 handler 协议签名为 `(master, node, inputs)`,但破坏 11 个现有 handler。

**采用方案 A**。每个 polish scenario 走一个闭包,简单清晰。

### 现有 handler 兼容性检查

- `_handler_chapter_writing` / `_handler_chapter_review` / `_handler_outline_review` / `_handler_polish` 4 个
- 改后:_handler_polish 还在,但只走 hook_extraction 兜底(如果有节点用此 scenario)
- **不破坏**:`_StubMaster` 在 test_got_bridge.py:65 有 `polish_chapter` 方法(返回 content + " (polished)"),不变

### `_handler_polish_merge` (新)

```python
def _handler_polish_merge(master, inputs):
    """scenario=polish_merge — 合并 polish_emotional_pacing + polish_ai_trace_removal 两条路径结果

    合并策略:选择两个 output.content 中较长的版本
    (避免任意截断, 优先保留更多润色结果)

    后续可扩展: 调 LLM 评分择优 / 按段落拼接 / 按 S1-S8 维度合并
    """
    # inputs 含 [upstream_id → upstream_output] 字典
    contents: list[str] = []
    for upstream_id, upstream in inputs.items():
        if isinstance(upstream, dict) and "content" in upstream:
            contents.append(upstream["content"])
    if not contents:
        return {"_error": "polish_merge requires >= 1 upstream with content"}
    return {"content": max(contents, key=len)}
```

**为什么用 `max(len)` 而不是拼接**:
- 拼接两个 variant 可能产生矛盾(dialogue 重写 + pacing 重写在同一段产生冲突)
- 选较长版本 = 选更"展开"的润色结果,更安全
- Phase 7.4 是 PoC,**留 YAGNI** 给后续 LLM 评分优化

### SCENARIO_HANDLERS 注册表更新

```python
SCENARIO_HANDLERS: dict[str, HandlerFn] = {
    # ... 11 个不变 ...
    # polisher
    "ai_trace_removal": _make_polish_handler("polish_ai_trace_removal"),
    "emotional_pacing": _make_polish_handler("polish_emotional_pacing"),
    "hook_extraction": _make_polish_handler("polish_chapter"),
    # AGGREGATION
    "polish_merge": _handler_polish_merge,
}
```

- **多 variant 数量动态配置** — 当前硬编码 2 个 (emotional_pacing + ai_trace_removal),后续可配置化
- **worktree 路径** — Phase 7.4 在 master 直接开发 (与 Phase 7.1-7.3 一致)
