# Phase 6.6 — Dashboard UI 全面 polish (A+B+D)

> Doc 4 §10 Phase 6.6 (Phase 6.x polish 段)
> 父阶段: Phase 6.1 (8 端点) → 6.2 (Vue 页面) → 6.3 (Mermaid) → 6.4 (WS 实时) → 6.5 (flock) → **6.6 (UI polish)**

## Context

Phase 6.1-6.5 已经把 dashboard 端到端打通:决策 API 8 端点、Vue 决策/工作流页面、Mermaid 渲染、WebSocket 实时推送、跨进程文件锁。但**生产用起来的可用性还有 3 个明显缺口**:

1. **决策历史完全不可见** (`DecisionsPage` "all" tab 把已解决/已推迟/已取消 跟待处理混在一起显示,`DecisionCard` 根本不显示 `status` 字段 — 用户看不出哪个是已解决的、谁解决、什么选项、何时)
2. **Mermaid 节点全是黄色 PENDING** (后端 `/api/workflows/{name}/mermaid` 调 `render_mermaid(graph, executions={})` 传空 dict,所有节点走 PENDING 分支,**Phase 6.3 的染色功能压根没启用**)
3. **决策中心没有状态分组** ("all" tab 顺序按 ID 排,Resolved 决策也是 pending 形态卡片,无法一眼看出"还有几个待办")

**本阶段目标**: 1 个 phase 解决 3 个 UI 缺口,3 commits 推送 (A / B / D 各一)。

**当前 baseline**: 2043 passed, 0 failed, 17 skipped (Phase 7.3 末)
**目标 baseline**: 2043 passed (D 加 1 个后端 test) + 0 failed,ruff 0

## 1. Feature A — DecisionsPage 状态分组 + 已解决折叠

### 现状

`DecisionsPage.vue`:
- 2 tabs: "待处理 (count)" / "全部 (count)"
- "all" tab 直接 `return all.value` — 全部已解决/已推迟/已取消 跟待处理混排
- 无 "history" 折叠

### 目标

**3 tabs 替代 2 tabs**:
- **待处理** (`pending.length`) — status === "pending"
- **已完成** (`resolved.length`) — status === "resolved" (默认折叠,可展开)
- **已取消/推迟** (`closed.length`) — status in (cancelled, deferred) (默认折叠,可展开)

**展开/折叠控件**: 像素风 `▸` / `▾` 箭头 + 计数 badge (沿用 8-bit 风格)。

**每组内部排序**: 待处理按 `priority` 升序(P0 在前),已完成按 `resolved_at` 倒序(最新在前)。

### 数据契约

`fetchAllDecisions()` 已返回 `list[dict]`,每个 dict 含 `status` 字段 (decision_queue.HumanDecision.to_dict),无 API 改动。

### 改动文件

- `dashboard/frontend/src/pages/DecisionsPage.vue` (改, ~30 行)
  - 加 `resolved` / `closed` computed
  - 加 `expanded` ref (`{ resolved: false, closed: false }`)
  - 改 `tabs` computed 返 3 项
  - 改 `displayList` computed 返 `expanded[tab] ? list : (tab === 'pending' ? pending : [])`
  - 加 CSS for 折叠箭头

### 不破坏

- API 不动 (决策状态字段早已透出)
- 现有 DecisionCard resolve/defer/cancel 操作不变
- 现有 WS 推送 (decision.snapshot) 不变 (推进 pending list,resolved 不在 WS 推送范围但 tab 切换时调 fetchAllDecisions 刷新)

## 2. Feature B — DecisionCard status badge + 元信息

### 现状

`DecisionCard.vue` 显示:
- `kind` badge (7 种 DecisionKind) ✅
- `priority` badge (P0/P1) ✅
- `node_id` (灰字) ✅
- `prompt` 文本 ✅
- `options` 按钮组 ✅
- resolve/defer/cancel 按钮组 ✅

**完全没显示**: `status` / `resolved_at` / `resolved_by` / `resolution` / `reason`

### 目标

**新增 status badge** (放在 `card-header`,紧跟 priority badge 之后):
- `status === 'pending'` → 不显示 (待处理是默认状态)
- `status === 'resolved'` → `✅ 已解决` (绿色)
- `status === 'deferred'` → `⏸ 已推迟` (黄色)
- `status === 'cancelled'` → `❌ 已取消` (灰色)

**新增元信息区** (放在 `.actions` 之后,仅当 status !== 'pending' 时显示):
```
解决人: human · 选项: approve · 时间: 2026-06-04 14:23
推迟原因: ...
取消原因: ...
```

**resolve 按钮变体**: status === 'pending' 时显示,其他状态隐藏;但提供"重新打开"按钮 (status === 'resolved' 时,允许重新 resolve,跟原操作走同一端点)。

### 数据契约

`decision_queue.HumanDecision.to_dict()` 已含:
```python
{
    "decision_id": str,
    "kind": str,
    "node_id": str,
    "prompt": str,
    "options": list[str],
    "priority": int,
    "status": str,          # ← 已有,前端没用
    "context": dict,
    "created_at": str,
    "resolution": Optional[str],   # ← 已有
    "resolved_at": Optional[str],  # ← 已有
    "resolved_by": Optional[str],  # ← 已有
    "reason": Optional[str],       # ← 已有
}
```

零 API 改动,纯前端。

### 改动文件

- `dashboard/frontend/src/components/DecisionCard.vue` (改, ~30 行)
  - 加 `statusBadge` computed (`{ label, class }` 或 `null`)
  - 加 `metaInfo` computed (拼接 resolved_by / resolution / resolved_at / reason)
  - 加 `<div class="status-badge" v-if="statusBadge">{{ statusBadge.label }}</div>`
  - 加 `<div class="meta-info" v-if="metaInfo">{{ metaInfo }}</div>`
  - 加 CSS (4 种 status 颜色:绿/黄/灰)
  - 改 `v-if="decision.status === 'pending'"` 守卫 resolve/defer/cancel 按钮

### 不破坏

- `busy` / `error` / emit 行为不变
- 现有 8-bit pixel 风格延续 (复用 `--color-success` / `--color-warning` 变量)

## 3. Feature D — Mermaid 节点状态染色 (修正 + 启用)

### 现状 (重要发现)

`/api/workflows/{name}/mermaid` 端点 (dashboard/app.py:739):
```python
mermaid_str = render_mermaid(graph, executions={}, include_classdef=True)
```
**传空 dict,所有节点按 PENDING 染色,全部黄色**。Phase 6.3 的"染色"功能**压根没启用**。

后端 `render_mermaid(graph, executions, ...)` 本身支持按 NodeStatus 染色(visualizer.py:64-107),只缺 executions 数据。

### 目标

**后端**: `/mermaid` 端点加 `?include_status=true` query 参数 (默认 `false` 保后向兼容):
- `false` (默认) → 行为不变,所有节点 PENDING
- `true` → 调 `master_controller.get_active_workflow_status()` 拿 `executions` 字段 (✅ 已有,`protocols.py:243-253` 返回,含 `paused_nodes` / `completed` / `failed` count),如果没活跃工作流 → 仍传空 dict (PENDING)

**返回结构**:
```python
class WorkflowMermaidResponse(BaseModel):
    workflow_name: str
    mermaid: str
    node_count: int
    has_decision_nodes: bool
    # NEW (only present when include_status=true and active workflow matches):
    status_applied: bool = False  # true = 染色基于 active workflow
    node_statuses: dict[str, str] = Field(default_factory=dict)  # {node_id: "completed"|...}
```

**前端**: `WorkflowsPage.vue` 的 `loadGraph` 加 query param:
```js
const result = await fetchWorkflowGraph(selected.value.name, { includeStatus: true });
```
- `WorkflowGraph.vue` **不动** — 它接收 mermaid 字符串,染色已烘进 mermaid 的 `class <nid> node-completed` 声明
- 前端只用 `node_statuses` 做**额外的小 status 图例** (可选,见下)

**染色状态 → 颜色映射** (沿用 visualizer.py 既有 `_CLASSDEF`):
- `completed` → 绿 (#90EE90)
- `running` → 蓝 (#87CEEB)
- `waiting` (DECISION 节点暂停) → 黄 (#FFD700)
- `pending` → 灰 (#D3D3D3)
- `failed` → 红 (#FF6347)
- `skipped` → 浅灰

### 数据契约

`master_controller.get_active_workflow_status()` 当前**没返 `executions` 字典** (只返 count + paused_nodes) → **需要扩展 adapter**。

**改** `dashboard/protocols.py:209-253`:
```python
def get_active_workflow_status(self) -> dict[str, Any]:
    # ... 现有代码 ...
    executions_dict = {}
    for nid, ex in executions.items():
        executions_dict[nid] = ex.status.value  # "completed" | "running" | "waiting" | "pending" | ...
    return {
        # ... 现有字段 ...
        "executions": executions_dict,  # ← NEW
    }
```

**改** `dashboard/app.py` 的 `WorkflowStatusResponse` (Pydantic model,line 126):
```python
class WorkflowStatusResponse(BaseModel):
    workflow_name: Optional[str] = None
    is_active: bool = False
    completed: int = 0
    failed: int = 0
    paused: bool = False
    paused_nodes: list[str] = Field(default_factory=list)
    node_count: int = 0
    steps: int = 0
    pending_decisions: list[dict] = Field(default_factory=list)
    executions: dict[str, str] = Field(default_factory=dict)  # ← NEW
```

### 改动文件

- `dashboard/protocols.py` (改 `get_active_workflow_status` 加 `executions` 字段)  — 1 method 改, ~5 行
- `dashboard/app.py`:
  - `WorkflowStatusResponse` 加 `executions` field  — 1 行
  - `WorkflowMermaidResponse` 加 `status_applied` + `node_statuses`  — 2 行
  - `/mermaid` 端点接受 `include_status: bool = False` query + 调 controller + 传 executions 给 render_mermaid  — 8 行
- `dashboard/frontend/src/api/index.js`: `fetchWorkflowGraph(name, opts)` 加 `includeStatus` 参数  — 3 行
- `dashboard/frontend/src/pages/WorkflowsPage.vue`: `loadGraph` 传 `{ includeStatus: true }`  — 1 行
- `dashboard/frontend/src/components/WorkflowGraph.vue`: **不动** (颜色已烘进 mermaid 字符串)

### 不破坏

- `?include_status` 默认 `false` → 旧调用方行为完全不变
- `WorkflowStatusResponse.executions` 默认 `{}` → 旧调用方行为不变
- `protocols.MasterControllerAdapter.get_active_workflow_status` 加字段,既有 dashboard tests 走 `_ControllableStub` 不依赖此字段 → 不破坏
- 视觉: mermaid 默认主题是白底 + 灰节点,现在变彩色 → 是 **改进** 不是破坏

## 8. 风险 + 缓解

| 风险 | 缓解 |
|------|------|
| A/B 改 DecisionsPage/DecisionCard 视觉破坏现有布局 | 沿用 8-bit pixel 风格 + 复用 CSS 变量 (`--color-success` / `--color-warning`),只加元素不改网格 |
| D 后端加 `executions` 字段破坏旧 adapter 测试 | adapter 已有 `_ControllableStub` 不返 `executions` 字段 → Pydantic 默认 `{}` → 不破坏 |
| D `?include_status=true` 但 controller 没活跃工作流 → 端点 500 | 显式 try/except 调 controller,失败返 `status_applied=False, node_statuses={}` |
| Mermaid 颜色可能跟 8-bit pixel 风格冲突 | 视觉上节点是 mermaid 默认白底,颜色是节点内部填色,不破坏页面整体风格 |
| 测试中 stub controller 的 `get_active_workflow_status` 没返 `executions` 字段 | stub 用 `is_active=False` 走"无活跃工作流"路径,不依赖 `executions` |
| DecisionCard 改 status 守卫后,resolved 卡片 resolve 按钮隐藏 → 用户无法重新操作 | 加 "重新打开" 按钮 (调同一 resolveDecision 端点) — 实际场景少见,优先 UI 干净 |
