## 一、背景与目标

### 1.1 现状问题

当前小说工厂在"作家卡文"时的处理方式：
- 主控介入：卡住了告诉主控，主控想办法协调
- 没有系统性支持，依赖主控的个人能力

### 1.2 优化目标

构建一个卡文突破系统，提供双重触发机制（自动检测 + 手动触发）：
- **自动检测**：作家超过 X 分钟无输入，系统自动触发突破建议
- **手动触发**：作家主动点击"卡文了"按钮，系统给出建议
- **主控协作**：系统无法处理时，转交主控介入

## 二、整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                    卡文突破引擎                                 │
│                                                              │
│  触发条件：                                                    │
│    1. 自动检测（作家超过X分钟无输入）                           │
│    2. 手动触发（作家点击"卡文了"按钮）                          │
│                                                              │
│  分析内容：                                                    │
│    1. 当前章节进度（写了多少，还剩多少）                         │
│    2. 当前章节的章纲（目标/冲突/高潮点）                         │
│    3. 前文情节摘要（卡在哪里了）                                │
│    4. 角色状态（从RAG获取）                                    │
│    5. 待回收伏笔（可能需要在这里回收）                          │
│                                                              │
│  输出建议：                                                    │
│    1. 转折思路（换个方向）                                     │
│    2. 线索提示（用伏笔/道具/人物）                             │
│    3. 冲突升级（加剧矛盾）                                     │
│    4. 情感升华（深化情绪）                                     │
│    5. 跳过当前场景（先写后面的）                               │
└─────────────────────────────────────────────────────────────────┘
```

## 三、卡文类型与对策

### 3.1 卡文类型分类

```yaml
writer_block_types:
  - type: "情节卡"
    description: "不知道接下来该发生什么"
    symptoms:
      - "写了300字后停住"
      - "反复删改开头"
    symptoms_detection:
      - "已写字数 < 500 且 5 分钟无输入"
      - "或：已写字数 > 1000 且 10 分钟无输入，且章纲未完成"
    对策:
      - "转折思路：引入新角色/事件/冲突"
      - "线索提示：从待回收伏笔中找"
      - "跳过：先写下一场景，后续补充"

  - type: "描写卡"
    description: "知道要写什么，但不知道怎么写"
    symptoms:
      - "场景画面模糊"
      - "对话写得很干"
    symptoms_detection:
      - "已写内容中描写词密度过低"
      - "或：对话连续超过10句无动作/表情描写"
    对策:
      - "感官提示：视觉/听觉/嗅觉/触觉引导"
      - "参考范例：提供同类型精彩描写片段"
      - "简化写法：先写动作，后续再润色"

  - type: "角色卡"
    description: "不知道某个角色会怎么反应"
    symptoms:
      - "角色行为不一致"
      - "对话写不下去"
    symptoms_detection:
      - "同一角色在3个连续段落中行为模式差异过大"
      - "或：角色对话超出其性格设定范围"
    对策:
      - "角色状态查询：从RAG获取当前状态"
      - "性格约束：提醒角色设定"
      - "关系提示：与他人的关系影响反应"

  - type: "情感卡"
    description: "情感描写不到位，无法让读者共鸣"
    symptoms:
      - "情绪平淡"
      - "读者反馈'没感觉'"
    symptoms_detection:
      - "情感词密度过低"
      - "或：情绪曲线过于平坦"
    对策:
      - "情感层次：从浅入深递进"
      - "关键细节：抓住最能引发共鸣的瞬间"
      - "情感意象：使用隐喻/象征强化"

  - type: "节奏卡"
    description: "节奏太快或太慢，不流畅"
    symptoms:
      - "情节跳跃"
      - "场景拖沓"
    symptoms_detection:
      - "场景时长异常（过短：<200字；过长：>2000字）"
      - "或：章节内场景数与节纲不符"
    对策:
      - "节奏调整：加入缓冲场景或加速冲突"
      - "节点检查：在关键情节点是否充分展开"
```

### 3.2 卡文分析流程

```
1. 检测到卡文（自动/手动）
2. 分析当前章节状态：
   - 写了多少字（当前进度）
   - 章纲是什么（目标/冲突/高潮）
   - 角色状态（从RAG获取）
   - 已埋设的伏笔（可能在这里回收）
3. 判断卡文类型（情节/描写/角色/情感/节奏）
4. 生成对应类型的建议（每个类型2-3条）
5. 展示给作家
6. 作家选择建议或继续自己想
```

## 四、建议生成策略

### 4.1 各类卡文建议策略

```yaml
suggestion_generation:
  情节卡:
    strategies:
      - "从伏笔库中找一条合适的伏笔，在这里回收"
      - "引入一个次要角色，制造新冲突"
      - "改变场景：换个地点或时间"
      - "逆转：让读者预期落空"

  描写卡:
    strategies:
      - "从同类型精彩片段中提取描写要点"
      - "提供感官描写提示（先写看到什么，再写听到什么）"
      - "用隐喻简化：'她的眼神像破碎的星光'"

  角色卡:
    strategies:
      - "查询角色当前状态（从RAG）"
      - "查看角色性格设定（性格/动机/恐惧）"
      - "提供角色关系网络（与周围人的关系）"
      - "提示角色当前的心理状态"

  情感卡:
    strategies:
      - "用具体细节替代抽象描述"
      - "从角色最在乎的事物入手"
      - "用对比强化情感（期待 vs 现实）"
      - "在情绪最高点突然转折"

  节奏卡:
    strategies:
      - "太慢：删减缓冲描述，加速冲突"
      - "太快：加入内心独白，给读者喘息"
      - "节奏参考：对比同类型章节的节奏分布"
```

### 4.2 建议输出格式

```yaml
卡文突破建议 - ch051

卡文类型：情节卡
分析原因：写了800字，但没有明确的下一步发展方向

【建议1】转折思路 - 从伏笔库找线索
内容：在ch020埋设的"铁蛋的纳米机甲"伏笔，可以在这里让铁蛋展示修复后的机甲能力，制造一个小高潮
操作：点击应用

【建议2】冲突升级 - 引入新角色
内容：引入星辰会的追兵，打破当前平稳的节奏
操作：点击应用

【建议3】跳过当前场景
内容：先写ch052的林夜vs莫言对峙，回头再补ch051的场景
操作：点击应用

【建议4】继续自己想
内容：不采用建议，继续自己思考
操作：点击关闭
```

## 五、与现有系统集成

### 5.1 与RAG系统的关系

```
卡文突破时 → 调用RAG获取：
  - 角色当前状态（位置/形态/情绪）
  - 已埋设伏笔列表（可能在这里回收）
  - 前文情节摘要（卡在哪里了）
  - 时间线信息（当前章节的时间点）
```

### 5.2 与作家工作室的关系

```
作家写 ch051 → 卡住超过5分钟 → 自动触发卡文突破
OR
作家点击"卡文了"按钮 → 触发卡文突破

卡文突破 → 作家选择建议 → 继续写作
```

### 5.3 与主控的关系

```
现有：卡住了告诉主控，主控想办法协调
新增：系统自动提供卡文突破建议，主控只需在复杂情况下介入

主控介入条件：
  - 系统无法判断卡文类型
  - 作家不接受任何建议
  - 需要跨章节协调（如需要修改前文大纲）
```

## 六、触发机制设计

### 6.1 自动检测

```yaml
auto_trigger:
  conditions:
    - "已写字数 > 0 且 < 2000"
    - "连续 5 分钟无输入（无键盘/鼠标活动）"
    - "章纲未完成"

  check_interval: "每30秒检测一次"
  initial_delay: "卡住后前2分钟不触发（给作家思考时间）"
```

### 6.2 手动触发

```yaml
manual_trigger:
  button_location: "作家写作界面右上角"
  button_style: "轻微提示，不显眼（避免依赖）"
  button_text: "卡文了？"
```

## 九、关键设计决策

| 决策 | 说明 |
|------|------|
| 双重触发机制 | 自动检测（5分钟无输入）+ 手动触发（按钮） |
| 5种卡文类型 | 情节/描写/角色/情感/节奏，覆盖主要场景 |
| RAG深度集成 | 卡文突破时调用RAG获取角色状态/伏笔/前文 |
| 作家自主选择 | 系统提供建议，作家选择应用或继续自己想 |
| 主控兜底 | 系统无法处理时转交主控，复杂情况主控介入 |
