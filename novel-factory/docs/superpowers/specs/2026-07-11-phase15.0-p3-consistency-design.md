# Phase 15.0 Spec — P3 一致性 (大重构)

> **路线**: Phase 14.0 P2 性能收口 → Phase 15.0 P3 一致性。Phase 9-12 审计留 5 块结构性 MEDIUM+ 全部本 phase 收口。
> **Architecture**: 5 T 全 **refactor (无 behavior 改)**, 0 改 baseline 0 测试代码, 0 改 production behavior, 0 真 LLM in tests, 0 改 .env/.state。`local Studio 默认无 auth` 同 §0.1。
> **关键指标**: dashboard/app.py 6265 行 → 多文件 < 800 行; SQLite 15+ files / 3 套 → 1 套; master_controller 1420 → 拆分; CLI 17 cmd + dual layout → 收敛; 227 `raise HTTPException` → 中央 handler。

## Phase 15.0 启动背景

Phase 9-12 性能/质量审查留 5 块结构性 MEDIUM+, Phase 13.0 (止血) + Phase 14.0 (性能) 都没动:
- **MEDIUM**: dashboard/app.py 6265 行 (164 routes + 227 HTTPException + 242 funcs/classes)
- **MEDIUM**: SQLite 实例散落 15+ files (state/cvg/cost/budget/workflow/reading_power 5 套 backend)
- **MEDIUM**: master_controller 1420 行 (5 agent × `_impl`+`with_usage` 重复 30+ 方法)
- **MEDIUM**: CLI `commands.py` + `commands/` 双 layout + 7 个 helper module 散落
- **MEDIUM**: 227 个 inline `raise HTTPException` 散落 app.py, 无中央 handler (重复 status_code 4xx/5xx 模板)

这 5 块都是结构性 refactor, 跟性能/止血无关但每个都是 1+ commit 大改造。本 phase 一次收口。

### T5 (REFACTOR) — HTTPException 中央 handler

**现状**:
- 227 个 `raise HTTPException(...)` 散落 app.py
- 唯一已注册 handler 是 `RateLimitExceeded` (line 2562)
- 没有 `APIError` 基类, 没有统一 error envelope (success envelope 在, error 没有)

**方案**:
- 新 `dashboard/errors.py`:
  - `APIError(Exception)` 基类: `status_code: int` + `code: str` + `message: str`
  - 子类: `NotFoundError(404)` / `ValidationError(400)` / `ConflictError(409)` / `UnauthorizedError(401)` / `RateLimitError(429)`
- 新 `@app.exception_handler(APIError)` handler: 返回 `{code, message, request_id}` JSON envelope
- 替换 227 个 `raise HTTPException(...)` 为 `raise APIErrorSubclass(...)` (保留 status_code 等价)
- RateLimitExceeded handler 改用统一 envelope

**不动**:
- HTTP status_code 0 改 (404/400/409/422 全部保留)
- response detail 字段名 `detail` → 改为 `message` (envelope 一致化, 但需要测试更新... 见下)
- 既有错误响应 `{"detail": "..."}` → 改为 `{"code": "...", "message": "..."}`
  - 测试中可能有断言 `response.json()["detail"] == "..."` → 需逐一更新
  - 这是 additive only 做不到的 (语义改) → 走 explicit semantic change

**Risk**: 错误响应字段改 `detail` → `message` 破坏 7+ 测试. Mitigation: 保留 detail alias (response 加 `detail` 字段, 同时加 `code` + `message`), 这样老客户端 0 改.
