# Phase 9.19 — Cascade BFS User-Controllable Depth

## Context

Phase 9.10 built the cross-volume ripple data model + persistence. Phase 9.16 added weighted BFS cascade (`v2_weighted`) with `max_depth=3` and `max_nodes_cap=100` as hardcoded defaults in `reference_graph.py:148`. The cascade depth is fire-and-forget inside `append_ripple` — callers (endpoint, CLI) cannot override it.

Current API gaps:
- `GET /api/cvg/ripples/{id}/cascade` and `/cascade/preview` read the **persisted** cascade (written at `append_ripple` time with depth=3). No way to request a different depth.
- No `lingwen.py cascade` subcommand exists.
- `RippleStorage` has no method to re-run BFS without persisting.

## Goals

1. `GET /api/cvg/ripples/{id}/cascade?max_depth=N` — re-run BFS up to N hops (live, not persisted) for any N in 1..10; `max_depth=0` (or absent) returns persisted (backward compat).
2. Same for `/cascade/preview`.
3. `lingwen.py cascade <ripple_id> [--max-depth N]` — new CLI subcommand, default N=3.
4. `RippleStorage.preview_cascade(ripple_id, max_depth)` — internal method, re-runs BFS without persisting.

## Non-Goals

- No config file / env var for default depth (caller passes the value).
- No persistent re-cascade (preview only, does not overwrite `ripple_cascade` row).
- No `weighted=True/False` toggle (always `v2_weighted`).
- No WS push on user-triggered cascade (read paths only).
- No changes to CascadeGraph.vue (already renders all nodes up to 100).
- No changes to existing Phase 9.10-9.18 source API behavior.

## Design

### Data Flow

```
client → GET /cascade/{id}?max_depth=5
  → app.py endpoint
  → storage.preview_cascade(ripple_id, max_depth=5)
  → reference_graph.trigger_cascade(ripple, max_depth=5, weighted=True)
  → returns CascadedRipple (NOT persisted)
  → endpoint returns CascadeResponse with depth_reached=actual, max_depth=5
```

### Endpoint Changes

**`GET /api/cvg/ripples/{ripple_id}/cascade`** (`dashboard/app.py:923-956`):
- Add `max_depth: int | None = Query(default=None)` parameter.
- If `max_depth is None or max_depth == 0`: return persisted cascade (existing path, backward compat).
- If `1 <= max_depth <= 10`: call `storage.preview_cascade(ripple_id, max_depth)`, return response with `max_depth` populated from query.
- If `max_depth < 0 or max_depth > 10`: raise `HTTPException(400, "max_depth must be 0 (persisted) or 1..10")`.

**`GET /api/cvg/ripples/{ripple_id}/cascade/preview`** (`dashboard/app.py:958-995`):
- Same logic as above.

### Storage Method

**`RippleStorage.preview_cascade(self, ripple_id: str, max_depth: int) -> CascadedRipple`** (`storage.py`):
- Fetch the `CrossVolumeRipple` by `ripple_id`.
- Call `self._graph.trigger_cascade(ripple, max_depth=max_depth, weighted=True)`.
- Return the resulting `CascadedRipple` — do NOT call `record_cascade` (no persistence).
- Raise `KeyError` if ripple not found (propagate to endpoint 404).

### CLI Command

**`lingwen.py cascade <RIPPLE_ID> [--max-depth N]`** (`infra/cli/commands/cascade.py`):
- Parser: `add_cascade_parser(subparsers)` with `RIPPLE_ID` positional + `--max-depth` optional (default 3, choices 1..10).
- Options dataclass: `CascadeOptions` (`infra/cli/options.py`).
- Command: look up ripple in storage, call `storage.preview_cascade(ripple_id, max_depth)`, print summary table (ripple_id, depth_reached, node_count, edge_count).
- Exit codes: 0 success, 1 error (not found, invalid max_depth).

File structure mirrors `ripple_audit` / `ripple_rollback`:
```
infra/cli/
├── commands/
│   ├── __init__.py   # add "cascade" → CascadeCommand
│   └── cascade.py    # CascadeCommand (~60L)
├── parsers/
│   ├── __init__.py   # add add_cascade_parser
│   └── cascade.py    # argparse parser (~25L)
└── options.py        # add CascadeOptions dataclass (~8L)
lingwen.py            # add elif command == "cascade"
```

### Schema (No Changes)

Existing fields are sufficient — just populate them:
- `CascadeResponse.max_depth: int = 0` (`protocols.py:738`) — return user-supplied cap or 0.
- `CascadeResponse.depth_reached: int` — already present, populated from `cascaded.depth_reached`.
- `CascadePreviewResponse.max_depth: int = 0` (`protocols.py:755`) — same.

### Bounds

| max_depth | Behavior |
|-----------|----------|
| absent / None / 0 | Return persisted cascade (backward compat) |
| 1..10 | Re-run BFS with cap, return live result |
| < 0 or > 10 | HTTP 400 / CLI exit 1 |

Re-run always honors `max_nodes_cap=100` (unchanged, line `reference_graph.py:165`).

## Risks + Mitigations

| Risk | Mitigation |
|------|------------|
| Expensive re-BFS on every request | Both `max_depth ≤ 10` AND `max_nodes_cap=100` bound worst case to ~100 nodes |
| Cycle detection breaks with depth override | `trigger_cascade` uses `visited: set` + `if depth < max_depth` (not `depth <=`), no cycle risk |
| Endpoint contract change breaks dashboard | `max_depth` is optional; absent → persisted (100% backward compat) |
| CLI parser inconsistent with existing | 1:1 mirror of `ripple_audit` parser pattern |
