## 3. Architecture

### 3.1 Layer 1 — Backend (T1 weighted BFS)

**File**: `infra/cross_volume/reference_graph.py:160-200` (现有 `trigger_cascade` 方法)

**改动** (3 处):
1. `trigger_cascade(ripple, max_depth: int = 3, weighted: bool = True)` 加 `weighted` 入参, default True 启用 v2_weighted
2. BFS 容器 `from collections import deque` → `import heapq`, 局部声明 `pq: list[tuple[float, str, ReferenceNode, int]] = []`, 入队 `heapq.heappush(pq, (-weight, node_id, node, depth))`, 出队 `heapq.heappop(pq)` 解 4-tuple
3. ties 稳定: weight 相等时用 node_id 字典序, heapq tuple 排序天然保证 (Python tuple 比较 lexicographic)

**`CascadedRipple` 改动** (1 处):
- `bfs_algorithm_version: str = "v1"` (hardcoded default) → 0 改 dataclass field, 仅在 `trigger_cascade` 末尾根据 `weighted` 选 `"v2_weighted"` / `"v1"` 后构造 `CascadedRipple`

**`record_ripple` 改动** (1 处):
- 内部 `trigger_cascade(ripple, max_depth=3, weighted=True)` 透传 (跟 Phase 9.14 audit hook 1:1)

**`RippleStorage` 改动**: 0 改 (`record_cascade` / `get_cascade_by_ripple_id` 已支持任意 `bfs_algorithm_version` string)

### 3.2 Layer 2 — Frontend (T2 hover 渐入)

**File**: `dashboard/frontend/src/components/CascadeGraph.vue:60-90` (现有 ECharts option 构造)

**改动** (5 行):
- `series` 配置加 `emphasis: { focus: 'self', itemStyle: { opacity: 1.0 } }` + `select: { itemStyle: { opacity: 0.3 } }` (默认非 hover 状态 opacity 已在 `itemStyle.opacity = 0.85`, 渐入通过 emphasis focus 强化)
- 0 新 emit / 0 新 handler / 0 新 props

### 3.3 Layer 3 — WS Push (T3)

**3 触点**:

**触点 1: Backend trigger** — `infra/cross_volume/storage.py` (cascade hook in `record_ripple`):
- `record_cascade(cascaded)` 之后, 调 `from dashboard.app import get_ws_manager; ws_manager = get_ws_manager(); ws_manager.broadcast({type: 'cascade.update', payload: {ripple_id, cascade_node_count, cascade_edge_count, depth_reached, bfs_algorithm_version}})` (跟 9.14 audit 1:1 best-effort: try/except + logger.warning)
- 注: 当前 `dashboard/app.py` 已有 `ConnectionManager` (Phase 6.4) + `get_ws_manager` factory, 9.16 复用, 0 改

**触点 2: WebSocket handler** — `dashboard/app.py` `/api/ws/workflows` 端点 (Phase 6.4 既有):
- 0 改 handler base (Phase 6.4 3 event type + 8.11 snapshot)
- 0 改 (新 event type 在 frontend handler 解析, backend 仅 broadcast)

**触点 3: Frontend subscribe** — `dashboard/frontend/src/composables/useWorkflowSocket.js:80-100` (现有 `onmessage`):
- 加 1 行 `else if (data.type === 'cascade.update' && data.payload) { latestCascadeUpdates.value = [...latestCascadeUpdates.value, data.payload].slice(-10); registeredHandlers.cascade.forEach(h => h(data.payload)); }`
- 加 1 module-level ref: `const latestCascadeUpdates = ref([])`
- 加 1 handler registry: `const registeredHandlers = { cascade: new Set() }`
- 加 1 API: `function onCascadeUpdate(handler) { registeredHandlers.cascade.add(handler); onBeforeUnmount(() => registeredHandlers.cascade.delete(handler)); }`

**触点 4: Drawer auto-refresh** — `dashboard/frontend/src/components/RippleDrawer.vue:80-100` (现有 `onMounted`):
- 加 `onCascadeUpdate((payload) => { if (payload.ripple_id === ripple.value?.id) { store.loadCascade(ripple.value.id); store.loadCascadePreview(ripple.value.id); } })` 注册
- 不匹配 ignore (next open 走 9.15 GET 返 fresh data)

### 3.4 数据流 (T3 end-to-end)

```
[backend] record_ripple() called
  ↓
  trigger_cascade(ripple, weighted=True)  # v2_weighted BFS
  ↓
  record_cascade(cascaded)  # sqlite3 INSERT
  ↓
  ws_manager.broadcast({type: 'cascade.update', payload: {...}})  # 9.16 NEW
  ↓
  [WebSocket] /api/ws/workflows → all connected clients
  ↓
  [frontend] useWorkflowSocket.onmessage → cascade.update branch
  ↓
  latestCascadeUpdates ref + registeredHandlers.cascade.forEach
  ↓
  [RippleDrawer] onCascadeUpdate handler
  ↓
  if match → store.loadCascade + loadCascadePreview → re-render CascadeGraph
```

## 5. Risks + mitigations

| 风险 | 缓解 |
|------|------|
| heapq 不稳定: ties weight 相等时 cascade_nodes 顺序漂移 | ties 用 `node_id` 字典序 (heapq tuple 比较天然 lexicographic), 1 pytest 覆盖 "same weight 不同 order 返 stable 顺序" |
| WS broadcast 失败导致 cascade 持久化失败 | `ws_manager.broadcast` 内置 try/except + dead-connection prune (Phase 6.4 既有), cascade INSERT 已 commit before broadcast, 广播失败仅 warning 不 propagate (best-effort 跟 9.14 record_audit 1:1) |
| 9.15 既有 14 e2e/cascade/storage test 0 改 (v1 FIFO 仍可读) | `weighted` 入参 default True 仅影响新 cascade, 旧 cascade 通过 `weighted=False` 显式重算或保留 v1 |
| ECharts emphasis 性能: 100 节点 hover 渲染开销 | ≤100 节点 (9.15 cap), emphasis API 用 GPU compositor (transform / opacity), 0 ms 计算开销, 1 vitest 覆盖 100 节点 hover |
| `useWorkflowSocket` 加 cascade 分支破 9.13 既有 connected/workflow.status/decision.snapshot 3 event | 1 行 `else if` 分支, 0 改既有 3 event 处理, 0 改 singleton state, 1 vitest 覆盖 3 旧 event 仍正常 |
| `record_ripple` cascade hook 0 改 v1 路径 | 9.15 既 hook 通过 `trigger_cascade(weighted=True)` 透传 (default True), 旧 v1 caller 走 `weighted=False` 显式 opt-out (YAGNI, 0 改) |
| Multi-tab multi-drawer race: 同 ripple 多 tab 都收到 cascade.update | 仅 ripple 匹配当前打开 drawer re-fetch, 0 global state, 0 冲突 (single user 假设) |
| WebSocket reconnect 增量补偿: 重连后漏收 cascade.update | Phase 6.4 `connected` event 带 snapshot, cascade delta 丢失不致命 — 用户 reopen drawer 时 9.15 GET 返 fresh data |
| `latestCascadeUpdates` ref 累积过大 | `.slice(-10)` 保留最近 10 条, 防内存累积 (实测 10 足够了) |
| `record_ripple` 透传 `weighted=True` 影响所有 9.15 旧测试 | 9.15 BFS tests 是 0 改 (16 tests 0 改), 新 BFS tests 显式 `weighted=True` 验证, 旧 FIFO behavior 0 regression |
| `dashboard/app.py` 1411L 加 1 行 (无变化, 复用 Phase 6.4 ws_manager) | 0 改 app.py, ws_manager 调用从 `storage.py` cascade hook 处走 |
| API key 泄漏 / 0 .env 改 | 0 provider 改, 0 .env 改, 0 ANTHROPIC_API_KEY 引用, 跟 Phase 9.10-9.15 一致 |
