# Phase 7.3 — novel_writing.yaml 加入 polish 节点 + 清理 4 LOW

> Doc 4 (GoT 适配设计 v1.0) §11 Phase 7.3
> 父阶段: Phase 7 (LLM 集成 E2E) → Phase 7.1 (production gap fixes) → Phase 7.2 (PolisherTools LLM-化) → **Phase 7.3 (workflow 接入 polish)**

## Context

Phase 7.2 已经把 `PolisherTools` 从"纯正则 + master 二次包装"升级为 **LLM 增强** —— `optimize_dialogue_llm` + `adjust_pacing_llm` 两条 LLM 路径,`polish_chapter` 主入口串联 2 个 LLM + 1 个规则 (`remove_ai_gloss` 兜底)。

但 **production workflow `novel_writing.yaml` 还没有 polish 节点**:

```yaml
# 当前 4 阶段
read_snapshot → write_chapter → review_chapter → emit_chapter
                       ↑                ↑
                  GENERATION         EVALUATION
                 (chapter_writing)  (chapter_review)
                                        ↓
                                    (content 已透传,Phase 7.1 fix)
                                        ↓
                                   emit_chapter (output,旁路)
```

结果: `master.polish_chapter` 这条 LLM 增强路径**实际生产中从未被调用**。生产链路是:
1. write_chapter 调 LLM 生成内容 (1 次)
2. review_chapter 调 LLM 审计 (1 次,Phase 7.1 修复后)
3. emit_chapter 直接落盘

**缺少 polish** 这意味着:
- LLM 写出的"AI 味"句子没经过 LLM 二次润色
- Phase 7.2 的代码改动在测试里验证了,但**生产工作流没接上** —— 用户感知不到

本阶段目标:**把 polish 节点插进 novel_writing.yaml**,让 LLM 增强的 polisher 真正跑在 production 数据上。同时把 Phase 7.2 code review 留下的 4 个 LOW 顺手清掉。

---

## 1. novel_writing.yaml 拓扑变更

### 当前 (Phase 7.2 后)

```yaml
nodes:
  - id: read_snapshot        # input, 旁路
  - id: write_chapter        # GENERATION, scenario=chapter_writing, 1 LLM
  - id: review_chapter       # EVALUATION, scenario=chapter_review, 1 LLM (Phase 7.1)
  - id: emit_chapter         # output, 旁路
```

`emit_chapter.depends_on: [review_chapter]`

### 目标 (Phase 7.3)

```yaml
nodes:
  - id: read_snapshot        # input, 旁路
  - id: write_chapter        # GENERATION, scenario=chapter_writing, 1 LLM
  - id: review_chapter       # EVALUATION, scenario=chapter_review, 1 LLM
  - id: polish_chapter       # SYNTHESIS,  scenario=ai_trace_removal, 2 LLM (dialogue + pacing)  ← NEW
  - id: emit_chapter         # output, 旁路 (depends_on 改为 [polish_chapter])
```

**NodeType 选择 SYNTHESIS 的理由**:
- `SYNTHESIS = "综合多源"` — polish 把审核报告 + 原内容 + style_guide 综合成润色版
- 不选 `EVALUATION` (评分),polish 是"内容改写"不是"打分"
- 不选 `GENERATION`,polish 的输入约束比 GENERATION 紧 (依赖 review 输出),语义更近 SYNTHESIS
- 8 种 NodeType 中**没有 TRANSFORMATION/REFINEMENT** (Doc 4 §3 锁定的枚举),SYNTHESIS 是最接近的语义

**为什么选 `ai_trace_removal` scenario**:
- SCENARIO_HANDLERS (got_bridge.py:120-137) 已注册 3 个 polish 场景,全部走 `_handler_polish` → `master.polish_chapter`
- 选 `ai_trace_removal` 是因为 production 痛点就是 LLM 输出"AI 味",与这个 scenario 名字最契合
- 另一个选项 `emotional_pacing` / `hook_extraction` 也都 OK,只是名字不够直白

### depends_on 调整

```yaml
- id: polish_chapter
  type: synthesis
  name: Polish Chapter
  description: LLM 增强润色(对话自然化 + 节奏调整 + AI 痕迹去除)
  depends_on: [review_chapter]    # ← 新增节点
  prompt_scenario: ai_trace_removal
  output_schema: PolishedDraft
  token_budget: 6000              # 2 LLM 路径 (3000+3000)
  timeout_s: 90

- id: emit_chapter
  type: output
  name: Emit Chapter
  description: 将润色后的章节写入 content_repo + 更新索引
  depends_on: [polish_chapter]    # ← 改:从 [review_chapter] 改为 [polish_chapter]
  prompt_scenario: null
  output_schema: null
  token_budget: 0
  timeout_s: 30
```

### 数据流验证 (Phase 7.1 已铺垫)

```
review_chapter (output):
  {content, report, scores, chapter_num, ...}  ← setdefault("content", content) (Phase 7.1 fix)
       ↓
_collect_inputs 收集上游 output → inputs["review_chapter"] = {content, ...}
       ↓
_handler_polish → _resolve_field(inputs, "content", "") → 拿到 content ✅
       ↓
master.polish_chapter(content) → 2 LLM + 1 规则 → 返回 str
       ↓
{"content": polished_str}  ← 透传给 emit_chapter
       ↓
emit_chapter (output 旁路) → 落盘 polished_str (而非原 content)
```

**关键**: Phase 7.1 的 `report.setdefault("content", content)` 已经把 content 透传给下游,Phase 7.3 不需要改 got_bridge.py。

## 5. 不在本阶段 (YAGNI)

- **minimal_e2e.yaml 不加 polish** — 它是测试 fixture,保持 2 节点 linear
- **polish 节点失败的重试/backtrack 策略** — 默认 GoT 调度处理,Phase 7.3 不定制
- **polish 节点跳过 (LLM 失败) 的开关** — Phase 7.2 韧性契约已在 master.polish_chapter 里处理 (try/except + remove_ai_gloss 兜底)
- **多变体 polish 节点** (e.g. 并行 emotional_pacing + ai_trace_removal) — 留待 Phase 8
- **polisher variant 集成到工作流** — Phase 7.2 的 reader_id 仍在 master 入口参数,生产工作流未来再加
