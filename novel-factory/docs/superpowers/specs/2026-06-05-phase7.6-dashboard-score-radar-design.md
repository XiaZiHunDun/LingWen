# Phase 7.6 — Dashboard S1-S8 评分雷达图 (消费 Phase 7.5 LLM 评分)

> Doc 4 §10/§11 Phase 7.6 (Polish 段, 继 7.4/7.5)
> 父阶段: Phase 7.5 (polish_merge LLM S1-S8 评分) → **7.6 (评分可视化)**

## Context

Phase 7.5 把 `polish_merge` 从 `max(len)` 升级到 LLM S1-S8 8 维评分加权选高者,`MasterController.polish_merge_synthesis` 返回 dict 含 `scores_a` / `scores_b` / `scores_total_*` / `winner` / `fallback`,通过 `_handler_polish_merge` 透传到 `NodeExecution.output`。

**问题**: 这些评分数据写进 `output` 字段后**没人消费** — dashboard 不知道有 S1-S8 评分,用户调完 merge 节点也看不到"为什么 A 赢了"。

**本阶段目标**: 把 S1-S8 评分数据从 backend 暴露到 API, 前端用 ECharts 雷达图可视化 — 8 维轴, 2 variant 重叠, winner 高亮, fallback 兜底信息明示。

**当前 baseline**: 2069 passed, 0 failed, 17 skipped (Phase 7.5 末)
**目标 baseline**: ~2076 passed (3-4 backend tests + 1-2 frontend/E2E tests), 0 failed, ruff 0

## 1. Feature A — Backend `score_data` 字段提取

### 现状

`MasterControllerAdapter.get_active_workflow_status()` (`dashboard/protocols.py:209-253`) 收集 `executions: dict[node_id, NodeExecution]`,但 `WorkflowStatusResponse.executions` (`dashboard/app.py:137`) 只暴露 `dict[str, str]` (node_id → status_str)。`NodeExecution.output` 含 7.5 评分但前端收不到。

### 目标

**新字段** `WorkflowStatusResponse.score_data: dict[str, dict] = {}` — 抽出每个含 S1-S8 评分 output 的 node, 转成前端友好的 dict。

**Adapter 提取逻辑** (在 `get_active_workflow_status` 末尾, 在 return dict 前):
```python
score_data: dict[str, dict[str, Any]] = {}
for nid, ex in executions.items():
    output = getattr(ex, "output", None) or {}
    if not isinstance(output, dict):
        continue
    if "scores_a" in output and "scores_b" in output:
        # 透传 7.5 评分 + label / fallback
        score_data[nid] = {
            "scores_a": output.get("scores_a", {}),
            "scores_b": output.get("scores_b", {}),
            "scores_total_a": output.get("scores_total_a", 0.0),
            "scores_total_b": output.get("scores_total_b", 0.0),
            "scores_delta": output.get("scores_delta", 0.0),
            "winner": output.get("winner"),
            "label_a": output.get("winner") if output.get("winner") == "<unknown>" else None,
            "label_b": None,
            "fallback": output.get("fallback"),  # None / "empty_content" / "identical" / "llm_fail" / "single_upstream"
        }
        # 抽 label_a / label_b (从 _handler_polish_merge 排序逻辑: input key sorted)
        # 因为 7.5 不在 output 里返 label — 我们从 _labels 字段提取
        labels = output.get("_labels", ())
        if len(labels) == 2:
            score_data[nid]["label_a"] = labels[0]
            score_data[nid]["label_b"] = labels[1]
```

**关键设计决策**:
- `score_data` 是**可空 dict** (默认 `{}`) — 旧客户端忽略, 后向兼容
- **只识别含 `scores_a` + `scores_b` 的 output** — 避免误判 (e.g. write_chapter output 是 `{content: ...}` 不抽)
- **label_a / label_b 优先用 `_labels` 字段** — 7.5 plan 没把这个字段加进 output, 7.6 需要在 `_handler_polish_merge` 透传 `labels` 给 ComputeResult.output (实际 master 调 polish_merge_synthesis(labels=...) 已经传了, 但 `output` 字段不存)
- **fallback 透传** — 让前端知道"是不是真的 LLM 评分"

### 改 `_handler_polish_merge` 透传 labels (跟 7.5 spec 同步改动)

**现状** (`got_bridge.py:_handler_polish_merge`):
```python
return master.polish_merge_synthesis(
    content_a=variants[label_a],
    content_b=variants[label_b],
    labels=(label_a, label_b),
)
```
master 内部只用 `labels` 算 `winner`, 不会回写到 result dict。

**改动**: 改 `_merge_synthesis_llm` 和 `_merge_synthesis_len_fallback` 都把 `labels` 写进 result:
```python
return {
    "content": content,
    "winner": winner,
    "scores_a": scores_a, ...
    "_labels": list(labels),  # 7.6: 透传供 dashboard 抽 label_a/b
}
```

**注意**: `_labels` 是**内部字段** (下划线前缀), 前端不直接读; adapter 抽 `label_a/b` 后丢弃。

### 韧性契约

- 旧 master 没 `_labels` 字段 → adapter `output.get("_labels", ())` 返空 tuple → `label_a/b` 为 None, 雷达图仍能显示 (用 "A" / "B" 默认)
- `output` 不是 dict (None / str) → adapter `isinstance` 检查跳过
- `scores_a/b` 缺失 → 跳过该 node (不写入 `score_data`)

## 2. Feature B — Frontend `ScoreRadarChart.vue` (ECharts 雷达)

### 新文件

`dashboard/frontend/src/components/ScoreRadarChart.vue` (~120L, 镜像 `CoolpointChart.vue` 模式)

**Props**:
```typescript
interface Props {
  scoresA: { S1: number, S2: number, ...S8: number }  // 0-10
  scoresB: { S1: number, ...S8: number }
  labelA: string   // e.g. "polish_emotional_pacing"
  labelB: string
  winner: string   // labelA or labelB or ""
  fallback?: string | null
}
```

**ECharts radar config**:
```js
const option = {
  title: { text: 'S1-S8 评分对比', left: 'center', textStyle: { fontSize: 14, color: '#333' } },
  tooltip: { trigger: 'item' },
  legend: { data: [labelA, labelB], top: 30 },
  radar: {
    indicator: [
      { name: 'S1 剧情', max: 10 },
      { name: 'S2 逻辑', max: 10 },
      { name: 'S3 文笔', max: 10 },
      { name: 'S4 情感', max: 10 },
      { name: 'S5 节奏', max: 10 },
      { name: 'S6 可读', max: 10 },
      { name: 'S7 主角', max: 10 },
      { name: 'S8 弧光', max: 10 },
    ],
    shape: 'polygon',  // 8 边形
    splitNumber: 5,
  },
  series: [{
    type: 'radar',
    data: [
      {
        name: labelA,
        value: [S1, S2, S3, S4, S5, S6, S7, S8],  // scoresA
        itemStyle: { color: winner === labelA ? '#67c23a' : '#909399' },
        lineStyle: { color: winner === labelA ? '#67c23a' : '#909399', width: 2 },
        areaStyle: { color: winner === labelA ? 'rgba(103,194,58,0.3)' : 'rgba(144,143,153,0.1)' },
      },
      { name: labelB, value: [...], 类似 colors },
    ],
  }],
}
```

**状态**:
- `fallback != null` → 顶部加一行黄底警告 "⚠️ LLM 评分未生效, 兜底: {fallback}"
- `winner == ""` (empty content) → 2 series 都不高亮
- 加载/解析错误 → "评分数据不可用" placeholder

**风格**: 沿用 Phase 6.2 像素风 (8-bit pixel), 字体 Press Start 2P, 配色:
- winner 绿 `#67c23a`
- loser 灰 `#909399`
- 警告黄 `#e6a23c`
- 雷达底色 `rgba(255,255,255,0.05)` (半透明)

### `WorkflowStatus.vue` 集成

**改**: 在 4-stat 网格下方加 `<ScoreRadarChart v-if="hasScores" ...>`, 数据从 `active.score_data.polish_merge` 取 (或第一个有 score_data 的 key)。

```vue
<template>
  <div class="workflow-status">
    <!-- 4-stat grid (existing) -->
    <div class="stat-grid">...</div>

    <!-- Phase 7.6 NEW: 雷达图 -->
    <ScoreRadarChart
      v-if="hasScores"
      :scores-a="firstScoreData.scores_a"
      :scores-b="firstScoreData.scores_b"
      :label-a="firstScoreData.label_a || 'Variant A'"
      :label-b="firstScoreData.label_b || 'Variant B'"
      :winner="firstScoreData.winner || ''"
      :fallback="firstScoreData.fallback"
    />

    <!-- pending decisions (existing) -->
    <div v-if="status.paused">...</div>
  </div>
</template>

<script setup>
import { computed } from 'vue'
import ScoreRadarChart from './ScoreRadarChart.vue'

const props = defineProps<{ status: Object }>()

const scoreEntries = computed(() => {
  const data = props.status.score_data || {}
  return Object.entries(data)
})
const firstScoreData = computed(() => scoreEntries.value[0]?.[1] || null)
const hasScores = computed(() => scoreEntries.value.length > 0)
</script>
```

### `api/index.js` 扩展

**无需新方法** — `fetchActiveWorkflow()` 已存在, 后端 `WorkflowStatusResponse.score_data` 是新字段, 前端读 `data.score_data` 即可 (Pydantic 自动序列化新字段)。

## 7. 风险 + 缓解

| 风险 | 缓解 |
|------|------|
| `output` 字段可能巨大 (chapter content 5K+) | adapter 只抽 `scores_a/b/labels/winner/fallback` 不含 content, payload 增量 ~200 bytes |
| NodeExecution.output 不一定含 `_labels` (旧 master) | `output.get("_labels", ())` 兜底返空, label_a/b 为 None, 前端用默认 "A"/"B" |
| ECharts 雷达 performance (8 维, 实时) | <16ms 渲染 (ECharts benchmark), dashboard 2s 轮询足够 |
| ECharts bundle 已在 (Phase 6.2), 不增加 bundle 体积 | 沿用 import `* as echarts from 'echarts'` |
| `_labels` 字段是内部 API 不优雅 | YAGNI, Phase 7.6+ 可重构成 `metadata` 子 dict |
| 旧 dashboard 客户端 (没 score_data 字段) | Pydantic 默认 `{}` 后向兼容, 前端 `.score_data` 返 undefined 不崩 |
| 雷达图跟现有 8-bit pixel 风格不一致 | 用 Press Start 2P 字体 + 像素边框, 镜像 ScoreRadarChart 配色 |
| E2E 测试需要 dev server 启动 | 沿用现有 Playwright setup, 复用 `tests/e2e/dashboard.spec.js` 的 server fixture |
