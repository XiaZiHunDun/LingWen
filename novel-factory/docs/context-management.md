# 上下文管理规范

> 基于 Claude Code Game Studios context-management.md 改写
> 版本：1.0 | 日期：2026-05-19

---

## 核心原则

**文件是记忆，对话是短暂的。**

对话将被压缩或丢失。文件在磁盘上持久存在，跨会话保留。

---

## 一、会话状态文件（Session State）

维护 `production/session-state/active.md` 作为动态 checkpoint。

### 内容结构

```markdown
# 会话状态 · [小说名]

## 当前任务
- **当前阶段**：卷1大纲 / 卷1_阶段3正文 / ...
- **当前步骤**：审核中 / 修订中 / ...
- **进度**：[x]/[y] 章节完成

## 已完成
- [x] 卷1总体大纲定稿
- [x] 卷1_阶段1-2大纲定稿
- [ ] 卷1_阶段3大纲审核中

## 关键决策
- **ADR-001**：卷2反派动机从复仇改为救赎
- **ADR-002**：卷3时间线调整为卷1前20年

## 当前文件
- 正在编辑：`03_内容仓库/02_卷大纲/卷1/卷大纲.md`
- 审核中：`06_意见仓库/02_卷大纲_审核/卷1_审核意见_v1.md`

## 待决策
- [ ] 待用户确认：卷1_阶段3的章节数量（当前设计10章 vs 编辑建议12章）

## 待处理意见
- [审核意见] 卷1_阶段3大纲：第5章支线比例过高，建议调整
- [读者反馈] 第7章：节奏过慢，第三章可合并

---

*最后更新：2026-05-19*
```

### 更新时机

- 每一阶段大纲完成 → 更新
- 重大决策做出 → 更新
- 审核循环完成 → 更新
- 遇到阻塞 → 记录

### 崩溃恢复

会话崩溃后，**首先读取状态文件**，然后读取正在编辑的文件，从上次位置继续。

---

## 二、增量写作协议

长篇小说是超大文档，必须分块处理。

### 章节级写作

```
1. 创建骨架文件
   └── 04_正文/卷1_阶段3/
       ├── 第001章_草稿.md   # 空壳，只有标题
       ├── 第002章_草稿.md
       └── ...

2. 一次只写一章
   └── 在对话中只讨论当前章节
   └── 历史中只保留当前章节的讨论

3. 写完一章 → 立即写入文件 → 更新状态

4. 进入下一章前 /clear 压缩上下文
```

### 大纲级写作

```
1. 创建骨架
   └── 03_阶段大纲/卷1_阶段3/阶段大纲.md
       ## 章节规划
       - 第001章：[标题]（空）
       - 第002章：[标题]（空）
       ...

2. 一次只讨论一个章节的设计

3. 讨论完成 → 写入文件 → 更新状态
```

## 三、主动压缩规则

### 压缩时机

| 上下文使用 | 行动 |
|-----------|------|
| 60-70% | 考虑 /clear，保存关键点 |
| 80%+ | 强制压缩 |
| 90%+ | 紧急压缩，可能丢失信息 |

### 自然压缩点

在以下节点后主动压缩：
- 完成一章节写入文件后
- 完成一轮审核意见输出后
- 完成一个决策后
- 开始新话题前

### 压缩内容

保留：
- 当前任务状态
- 待决策问题
- 正在编辑的文件路径
- 最近的决策记录

可压缩：
- 已完成章节的讨论历史
- 已确认的审核意见
- 过期的状态更新

## 四、上下文预算

| 任务类型 | 启动预算 | 任务描述 |
|---------|---------|---------|
| **轻量** | ~3k tokens | 状态检查、文件读取、格式化 |
| **中量** | ~8k tokens | 单章节创作、单卷大纲审核 |
| **重量** | ~15k tokens | 全文大纲创作、多卷联动设计 |

### 重量任务的策略

对于需要多章节/多卷上下文的任务：
1. 使用 subagent 研究子问题
2. 主会话只保留汇总结果
3. 文件承载详细信息

---

## 五、Subagent 使用准则

### 使用 Subagent 当

- 研究跨多个文件的问题（>5k tokens 读取）
- 探索不熟悉的代码/内容区域
- 并行执行独立任务

### 直接读取当

- 知道需要读取哪 1-2 个文件
- 需要精确匹配的内容

### Subagent 不继承对话历史

在 prompt 中提供完整背景和上下文。

## 六、长篇汇总的上下文策略

### 问题

全文汇总（100+ 章节）超出任何 LLM 的上下文限制。

### 解决：层级聚合

```
正文层：单章 3000 字
  ↓ 汇总为章概要
阶段层：5-10 章节 → 阶段汇总 ~2000 字
  ↓ 汇总
卷层：4-6 阶段 → 卷汇总 ~1500 字
  ↓ 汇总
全文层：3-5 卷 → 全文汇总 ~1000 字
```

### 汇总时只读取子节点文件

```markdown
# 错误做法
[读取全部正文 100+ 章节] → 超出上下文

# 正确做法
[读取卷1所有阶段汇总] → 读取卷2所有阶段汇总
→ 合并为全文汇总
```

---

## 七、压缩后恢复流程

1. 读取 `production/session-state/active.md`
2. 读取 `正在编辑的文件` 获得当前状态
3. 读取最近的 `ADR` 获得决策上下文
4. 继续工作

---

*上下文管理是长篇创作的生死线。主动压缩胜于被动溢出。*
