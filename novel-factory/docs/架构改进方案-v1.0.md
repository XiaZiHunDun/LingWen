# 架构改进方案 v1.0

> **版本**: v1.0
> **日期**: 2026-06-01
> **状态**: 待实施
> **目标**: 统一质量检测体系，消除架构毛刺，提升可维护性

---

## Context

当前项目存在以下架构问题：

1. **双检测器体系未统一** - `consistency/` (28 checkers) 和 `quality/` (2 checkers) 是两套独立体系
2. **QualityCoordinator 未启用** - Phase 1-2 建立的统一框架没有被实际使用
3. **quality_tools 与 consistency 职责重叠** - 软评分器和硬检测器功能重复
4. **工具散落多处** - `tools/`, `infra/tools/`, `infra/quality_tools/` 三处
5. **Inspector/Checker 双基类** - 接口不一致，无法统一调用
6. **hooks/actions 与主系统紧耦合** - 无法独立测试

**目标**：合并检测器体系，启用 QualityCoordinator，统一工具入口，降低复杂度。

---

## 问题详述

### P0: 双检测器体系未统一

**现状**：
```
infra/
├── consistency/           # 28个BaseChecker子类
│   └── checkers/          # timeline_checker, character_checker, etc.
└── quality/                # 2个Inspector子类
    └── checkers/          # WorldviewChecker, AITraceChecker
```

**问题**：
- `QualityCoordinator` 定义了 `UnifiedIssue` 接口，但无法协调 `consistency/` 的28个checker
- 工具使用混乱：有的导入 `from infra.quality import WorldviewChecker`，有的导入 `from infra.consistency.checkers import WorldviewChecker`
- 去重机制无法跨体系工作

**接口对比**：
```python
# quality/inspector.py - Inspector基类
class Inspector:
    def check(self, chapter_num: int) -> List[Issue]: ...

# consistency/base_checker.py - BaseChecker基类
class BaseChecker(ABC):
    def check(self, chapter_content: str, chapter_num: int, context: Optional[Dict] = None) -> List[Issue]: ...
```

### P0: Inspector/Checker 双基类体系

**问题**：
- `Inspector.check(chapter_num)` - 检测器内部读取内容
- `BaseChecker.check(chapter_content, chapter_num, context)` - 外部传入内容

**后果**：无法统一调用，adapter 需要处理两种接口。

### P1: QualityCoordinator 未启用

**现状**：Phase 1-2 建立了 `infra/quality/coordinator.py`，但：
- 37处 `from infra.quality import` 都只使用旧模块（Issue、WorldviewChecker、AITraceRepairer）
- `QualityCoordinator` 只在 docstring 示例中出现，无实际调用
- tools/ 下的工具都直接调用具体的 checker/repairer，不通过 coordinator

**影响**：统一质量报告功能无法实现，rule-based 和 LLM 检测仍然各自为战。

### P1: quality_tools 与 consistency 职责重叠

**示例**：
| 功能 | quality_tools/ | consistency/ |
|------|----------------|---------------|
| 紧张度 | `soft_scorers/tension.py` | `checkers/pacing_checker.py` |
| 冗余 | `soft_scorers/redundancy.py` | `checkers/chapter_redundancy.py` |
| 视角 | `hard_validators/perspective.py` | `checkers/narrative_perspective_checker.py` |

### P1: 工具散落多处

**现状**：
```
novel-factory/tools/           # batch_repair.py, verify_quality.py, llm_*.py
infra/tools/                   # workflow/lib.py, heartbeat.py, check_stale_tasks.py
infra/quality_tools/           # QualityGate, MultiStyleDrafter, soft_scorers/, hard_validators/
```

**问题**：
- 新开发者不知道该去哪里找工具
- 功能重复（batch_repair.py 和 llm_quality_deep_check.py 都做修复）
- 没有统一的工具入口

### P2: hooks/actions 与主系统紧耦合

**现状**：`UpdateStateAction` 直接 import `WorkflowStateManager`

**问题**：无法独立测试，无法切换不同的状态后端

## 改进方案

### Phase 1: 统一检测器基类（P0）

#### 1.1 建立统一的Checker接口

**方案**：将 `consistency/checkers/base_checker.py` 的 `BaseChecker` 改为继承 `quality/inspector.py` 的 `Inspector`，统一接口。

```python
# infra/consistency/checkers/base_checker.py
from infra.quality.inspector import Inspector, Issue as UnifiedIssue

class BaseChecker(Inspector):
    """继承统一Inspector基类"""

    def __init__(self, checker_type: CheckerType):
        super().__init__()
        self.checker_type = checker_type

    def check(self, chapter_num: int) -> List[UnifiedIssue]:
        """统一接口 - 检测器内部读取内容"""
        content = self.read_chapter(chapter_num)
        return self._check_content(content, chapter_num, None)

    @abstractmethod
    def _check_content(
        self,
        chapter_content: str,
        chapter_num: int,
        context: Optional[Dict[str, Any]]
    ) -> List[UnifiedIssue]:
        """内部实现 - 子类实现具体检测逻辑"""
        pass
```

#### 1.2 迁移 consistency/checkers/ 到统一体系

**步骤**：
1. 修改 `BaseChecker` 继承 `Inspector`
2. 将 `check(self, chapter_content, chapter_num, context)` 改为 `_check_content(self, chapter_content, chapter_num, context)`
3. 添加 `check(self, chapter_num)` 方法，内部调用 `read_chapter()` 和 `_check_content()`
4. 更新所有28个checker的接口

#### 1.3 统一Issue类型

**方案**：
- `quality/inspector.py` 的 `Issue` 作为统一Issue类型
- `consistency/engine/data_structures.py` 的 `Issue` 标记为 `@deprecated`
- 迁移所有checker使用统一Issue

### Phase 2: 启用 QualityCoordinator（P0）

#### 2.1 修改工具使用Coordinator

**步骤**：
1. `tools/verify_quality.py` → 使用 `QualityCoordinator`
2. `tools/batch_repair.py` → 通过 `QualityCoordinator` 获取修复器
3. `infra/cli/commands.py` → `QualityCommand` 通过 `QualityCoordinator` 运行

**示例**：
```python
# tools/verify_quality.py - 重构后
from infra.quality.coordinator import QualityCoordinator

coordinator = QualityCoordinator(
    consistency_engine=None,  # lazy-load
    llm_analyzers=["all"],
    deduplicate=True,
    parallel=True
)

def verify(chapter_num: int) -> UnifiedQualityReport:
    paths = ProjectPaths.get()
    content = paths.read_chapter(chapter_num)
    return coordinator.check_chapter(chapter_num, content)
```

#### 2.2 实现ConsistencyEngine到Coordinator的适配

**步骤**：
1. 在 `QualityCoordinator` 中添加 `consistency_engine` 属性（lazy-load）
2. 实现 `_check_rule_based()` 调用 `ConsistencyEngine`
3. 将 `ConsistencyReport` 通过 `RuleBasedAdapter` 转换为 `UnifiedQualityReport`

### Phase 3: 合并 quality_tools 到 quality/（P1）

#### 3.1 迁移 soft_scorers/ 到 quality/llm/

**方案**：
- `quality_tools/soft_scorers/` → `quality/llm/scorers/`
- 软评分器实现 `LLMScorer` 接口，纳入 `QualityCoordinator` 管理

#### 3.2 迁移 hard_validators/ 到 quality/rule_based/

**方案**：
- `quality_tools/hard_validators/` → `quality/rule_based/validators/`
- 硬验证器实现 `RuleBasedValidator` 接口

### Phase 4: 统一工具入口（P1）

#### 4.1 建立 tools/ 统一目录

**方案**：
- `novel-factory/tools/` → `infra/quality/tools/`
- `infra/tools/` 中的 `workflow/` 保留（状态管理相关）
- `infra/quality_tools/` → `infra/quality/tools/quality_tools/`（重命名避免混淆）

**最终结构**：
```python
infra/quality/tools/
├── __init__.py
├── batch_repair.py      # 批量修复
├── verify_quality.py     # 质量验证
├── llm_analyzers/       # LLM分析器（现有tools/llm_*.py迁移）
│   ├── llm_foreshadow_analyzer.py
│   ├── llm_emotional_resonance_checker.py
│   └── ...
└── regression_tracker.py
```

#### 4.2 消除重复工具

| 重复 | 保留 | 废弃 |
|------|------|------|
| `batch_repair.py` | `infra/quality/tools/batch_repair.py` | `tools/track_a_repair.py`, `track_b_repair.py`, `track_c_repair.py` |
| `llm_quality_deep_check.py` | `infra/quality/tools/llm_quality_deep_check.py` | - |

### Phase 5: 解耦 hooks/actions（P2）

#### 5.1 建立Action接口抽象

**方案**：
```python
# infra/hooks/actions/base.py
class BaseAction(ABC):
    @property
    @abstractmethod
    def action_type(self) -> str:
        pass

    @abstractmethod
    def execute(self, params: Dict[str, Any], context: Dict[str, Any]) -> ActionResult:
        pass

# StateBackend接口
class StateBackend(ABC):
    @abstractmethod
    def get(self, key: str) -> Any: ...

    @abstractmethod
    def set(self, key: str, value: Any) -> None: ...
```

#### 5.2 实现可切换的后端

```python
# infra/state/backends/
class SQLiteBackend(StateBackend):
    def __init__(self, db_path: str): ...

class JSONBackend(StateBackend):
    def __init__(self, file_path: str): ...

# UpdateStateAction使用后端注入
class UpdateStateAction(BaseAction):
    def __init__(self, state_backend: StateBackend):
        self._backend = state_backend
```

## 风险与缓解

### 风险1: 大规模重构影响现有功能

**缓解**：
- Phase 1-5 分阶段实施，每阶段有验证点
- 保留老接口的 adapter/backward compatible直到新接口稳定
- 使用 feature flag 切换

### 风险2: 检测器接口变更影响28个checker

**缓解**：
- 提供基类模板，子类只需实现 `_check_content()`
- 批量脚本辅助迁移

### 风险3: 工具迁移导致路径变化影响CLI

**缓解**：
- 更新 `lingwen.py` 的工具路径配置
- 保留旧的工具入口作为 alias
