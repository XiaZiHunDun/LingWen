## 一、问题回顾

### 1.1 核心问题

| 问题类型 | 问题描述 | 影响 |
|----------|----------|------|
| **工具路径错误** | `llm_polish_chapters.py` 的 `PROJECT_ROOT` 多了一级，导致章节读取失败 | 修复流水线失效 |
| **正则不适用中文** | 使用 `\b词边界\b` 正则在中文文本中无法匹配 | 世界观统一失败 |
| **LLM润色无效** | 使用MiniMax LLM对AI痕迹润色，反而产生更多模板句式 | AI痕迹从69处增至151处 |
| **API Provider混用** | 部分工具用Anthropic，部分用MiniMax | API认证失败 |
| **工作流耦合** | 检测、修复、验证混在一个脚本中 | 难以独立运行和调试 |

### 1.2 经验总结

1. **规则替换** 适合消除模板句式等简单模式
2. **LLM分析** 适合角色一致性等需要理解的任务
3. **中文处理** 应使用简单字符串替换，不用正则词边界
4. **工具路径** 必须统一基准，建立中央配置
5. **增量修复** 优于全量重处理

### 3.2 核心模块

#### 3.2.1 paths.py - 统一路径管理

```python
from pathlib import Path

class ProjectPaths:
    """项目路径配置"""

    def __init__(self):
        # 统一以 novel-factory 为基准
        self.root = Path(__file__).parent.parent
        self.chapters = self.root / "03_内容仓库" / "04_正文"
        self.characters = self.root / "03_内容仓库" / "角色设定"
        self.outline = self.root / "03_内容仓库" / "02_大纲"
        self.logs = self.root / "logs"
        self.rules = self.root / "tools" / "rules"

    @classmethod
    def get(cls) -> "ProjectPaths":
        """单例获取"""
        if not hasattr(cls, "_instance"):
            cls._instance = cls()
        return cls._instance
```

#### 3.2.3 inspector.py - 检测器基类

```python
class Inspector:
    """检测器基类"""

    def __init__(self, paths: ProjectPaths):
        self.paths = paths

    def check(self, chapter_num: int) -> List[Issue]:
        """检测单章"""
        raise NotImplementedError

    def check_batch(self, chapter_nums: List[int]) -> List[Issue]:
        """批量检测"""
        return [issue for ch in chapter_nums for issue in self.check(ch)]
```

## 五、工作流设计

### 5.1 实时写作质量门控

```
写作时 ──[Hook]──> 实时检测
                ↓
         发现AI痕迹/世界观问题 ──[自动修复]──> 提交
```

### 5.2 批量修复工作流

```
定期检测 ──[Inspector]──> 生成Issue列表
                ↓
         人工确认修复优先级
                ↓
         [Repairer] ──[dry_run]──> 预览
                ↓
         确认执行 ──[Repairer]──> 写入修复
                ↓
         [Inspector]──> 验证效果
```

### 5.3 增量修复流程

```
检测到变更 ──> 识别变更章节
                ↓
         只对变更章节执行检测+修复
                ↓
         更新质量报告
```

## 八、验收标准

| 验收项 | 标准 |
|--------|------|
| 路径管理 | 所有工具使用 `ProjectPaths.get()` 获取路径 |
| LLM服务 | 任何工具调用LLM使用 `LLMService.get()` |
| 检测器 | 继承 `Inspector` 基类，实现 `check()` 方法 |
| 修复器 | 继承 `Repairer` 基类，实现 `repair()` 方法 |
| 规则配置 | 规则存储在YAML中，支持版本号 |
| 增量支持 | 修复变更章节时不影响其他章节 |
| 测试覆盖 | 核心模块有单元测试 |

## 九、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 规则冲突 | 多个规则作用于同一文本可能冲突 | 规则优先级+互斥检测 |
| LLM分析不稳定 | 不同调用返回格式不一致 | 统一解析层+重试机制 |
| 回滚困难 | 修复后难以还原 | 每次修复前保存快照 |
| 性能瓶颈 | 全量检测耗时 | 并行+增量方案 |
