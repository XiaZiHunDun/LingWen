# 大纲锁定与变更管理规范

> **版本**：v1.0
> **日期**：2026-05-19
> **目的**：保护已审核通过的大纲不被轻易推翻，确保创作阶段稳定推进

---

## 核心理念

**锁定是保护，不是限制。**

大纲锁定是为了保护已验证的创作方向，让写作团队安心推进不被反复推翻。变更审批是为了在必要时有序调整，避免"随时改"导致的无限返工。

> **核心原则**：已锁定的大纲如需变更，必须走审批流程；新版本另存，不覆盖原版。

---

## 一、锁定等级定义

### 1.1 锁定等级总览

| 等级 | 触发时机 | 文件命名格式 | 状态标签 |
|------|----------|--------------|----------|
| **L0-草稿** | 未提交审核 | `大纲_草稿_v0.9.md` | 无标签 |
| **L1-审核中** | 提交审核，待结果 | `大纲_审核中_v1.0.md` | `【审核中】` |
| **L2-已锁定** | 审核通过，进入写作 | `大纲_已锁定_v1.0.md` | `【已锁定】` |
| **L3-变更申请中** | 变更申请待审批 | `大纲_变更申请_v1.1.md` | `【变更申请中】` |
| **L4-新版本已锁定** | 变更批准，新版本 | `大纲_已锁定_v1.1.md` | `【已锁定_v1.1】` |

### 1.2 锁定等级说明

```
L0(草稿) → L1(审核中) → L2(已锁定) → L3(变更申请中) → L4(新版本锁定)
                ↑                                      │
                └──────── 审核不通过，返回L0 ───────────┘
```

**状态流转规则**：
- 审核通过后标记为 `【已锁定_vX.X】`
- 变更申请批准后，生成新版本文件，原版本保留
- 已锁定文件禁止直接覆盖

## 二、锁定机制

### 2.1 锁定触发条件

进入以下阶段前，对应层级大纲必须已锁定：

| 创作阶段 | 必须锁定的内容 |
|----------|----------------|
| **规划期结束** | 全文大纲（三幕式骨架 + 核心驱动链） |
| **验证期结束** | 核心样章 + 情节骨架验证结论 |
| **写作期-卷N开始** | 卷N章纲（至少完成前三章章纲） |
| **写作期结束** | 全部章纲 + 初稿全文 |

### 2.2 锁定文件命名规则

**格式**：
```
[大纲类型]_[锁定状态]_[版本号].md
```

**示例**：
```
大纲_已锁定_v1.0.md          # 全文大纲v1.0，已锁定
卷一_已锁定_v1.0.md          # 卷一章纲v1.0，已锁定
世界观_已锁定_v1.0.md        # 世界观设定v1.0，已锁定
核心驱动链_已锁定_v1.0.md    # 核心驱动链v1.0，已锁定
```

### 2.3 锁定标签

在文件开头添加锁定标签：

```markdown
---
锁定状态：【已锁定_v1.0】
锁定日期：2026-05-19
锁定审批人：[主编/审核员名称]
锁定原因：通过大纲审核，进入写作期
---
```

### 2.4 已锁定大纲的处理

| 操作 | 允许？ | 说明 |
|------|--------|------|
| 阅读 | ✅ | 随时可查阅 |
| 新版本另存 | ✅ | 另存为 `大纲_已锁定_v1.1.md` |
| 直接编辑原文件 | ❌ | 禁止，先申请变更 |
| 覆盖原文件 | ❌ | 禁止 |

## 三、变更审批流程

### 3.1 变更类型分类

| 类型 | 定义 | 示例 | 处理方式 |
|------|------|------|----------|
| **Minor（微调）** | 不影响下游的局部调整 | 错别字修正、语气调整、细节补充、伏笔微调 | 执行人自检后直接更新版本号 |
| **Major（重要变更）** | 影响下游但可修复 | 删除/合并章节、调整章节顺序、核心角色设定调整 | 必须走变更审批流程 |
| **Blocker（阻断变更）** | 改变核心驱动链或主线结局 | 改变故事主线、改变结局走向、改变核心冲突基础 | 禁止（除非主公特批） |

### 3.2 变更审批流程

```
变更申请人
    ↓ 提交变更申请表
[审核员/主编]
    ↓ 评估影响
├── 影响小（Minor）→ 批准 → 更新版本号 → 通知相关方
├── 影响大（Major）→ 评估影响范围 → 审批 → 新版本另存 → 通知相关方
└── 阻断（Blocker）→ 拒绝 → 返回修改方案
```

### 3.3 变更申请提交内容

提交变更申请时，必须包含：

1. **变更原因**：为什么要改
2. **变更内容摘要**：具体改什么
3. **影响评估**：对下游（章纲、初稿、已发布内容）的影响
4. **替代方案**（如有）：有没有不改的方案

### 3.4 审批决策

| 决策 | 含义 |
|------|------|
| **批准** | 可执行，按流程更新版本 |
| **拒绝** | 不可执行，维持原大纲 |
| **批准+条件** | 可执行，但需满足特定条件 |
| **转主公裁决** | 超出审核员权限，需主公决定 |

## 四、变更后处理

### 4.1 新版本管理

变更批准后：
1. 原版本文件**保留不动**，加注 `【已归档】` 标签
2. 新版本另存，文件名体现新版本号
3. 更新版本记录表

```markdown
---
锁定状态：【已归档_v1.0】
归档日期：2026-05-20
归档原因：v1.1版本已锁定，v1.0归档
---

---
锁定状态：【已锁定_v1.1】
锁定日期：2026-05-20
锁定审批人：[主编名称]
锁定原因：核心角色动机调整，变更申请已批准
---
```

### 4.2 下游通知

变更批准后，需要通知的相关方：

| 变更范围 | 需通知方 |
|----------|----------|
| 全文大纲变更 | 全体创作人员、审核组 |
| 单卷大纲变更 | 该卷写作组、审核员 |
| 核心角色设定变更 | 主编、相关写作组 |
| 世界观设定变更 | 全体创作人员 |

### 4.3 回溯检查

大纲变更后，相关下游工作需要检查：

| 下游 | 检查项 |
|------|--------|
| 章纲 | 是否需要同步调整 |
| 已写初稿 | 是否需要回溯修订 |
| 已发布内容 | 是否需要记录勘误 |

## 五、与现有7部门架构的兼容性

### 5.1 变更申请流程中的部门职责

| 部门/角色 | 职责 |
|-----------|------|
| **主编** | 变更审批（Major级）、转主公裁决 |
| **审核员** | Minor变更审批、Major变更初审 |
| **作家** | 提交变更申请、执行批准后的变更 |
| **设定监理** | 设定类变更的专业审核 |
| **主控调度** | 协调通知、状态更新 |

### 5.2 变更申请路由规则

```
变更申请 → 主控调度接收
    ↓
根据变更类型路由：
├── Minor → 审核员审批
├── Major → 主编审批
└── Blocker → 主公裁决（主编上报）
```

### 5.3 与锁定门检查清单的衔接

变更管理规范与 `锁定门检查清单.md` 配合使用：

| 锁定门 | 对应变更管理 |
|--------|--------------|
| 第一锁：大纲锁定 | L2锁定，变更需走Major审批 |
| 第二锁：初稿锁定 | L2锁定，变更需走Major审批 |
| 第三锁：修改锁定 | L2锁定，变更需走Major审批 |
