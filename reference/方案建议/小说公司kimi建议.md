## 一、架构评审

### ✅ 设计亮点
| 亮点 | 说明 |
|------|------|
| **分层解耦** | 灵感→大纲→正文→汇总，符合出版业实际流程 |
| **画像驱动** | 作家/审核/读者都用Markdown画像+Subagent，便于角色一致性控制 |
| **分层审核** | 先粗后细，避免早期错误传递到后期造成返工 |
| **读者模拟** | 多角色读者反馈比单一审核更接近真实市场反应 |

### ⚠️ 潜在风险与改进建议

**1. 上下文窗口瓶颈**
长篇小说正文汇总时容易超出Claude的上下文限制。
- **对策**：汇总部门不要一次性读取全部正文，而是**逐层聚合**（章节→阶段→卷→全文），每一步只读取当前层级的子节点汇总。

**2. 多作家风格漂移**
不同Subagent作家写出来的章节风格可能不一致。
- **对策**：在灵感部门输出中增加**`style_guide`**字段；每位作家的画像中增加"必须遵守的风格指南"约束；在正文阶段前增加一个**"风格校准章节"**（先写第1章，全员审核通过风格后再并行写后续）。

**3. 循环死锁**
审核/读者循环可能无限进行。
- **对策**：每个循环设置**最大迭代次数**（如审核3轮、读者反馈2轮），超限时由"总编Subagent"强制仲裁。

**4. 并行化机会未充分利用**
各卷、各阶段、各章节之间理论上可以并行。
- **对策**：同层级无依赖的内容（如不同卷的大纲）可以**并行生成**，用Claude Code的多个终端会话或脚本并行调用。

## 二、工作流补全（步骤18起）

你缺失的步骤18及后续，补全如下：

```text
18. 【汇总部门】根据内容仓库中已通过读者审核的章节正文，按层次生成分阶段汇总
    → 输出：汇总仓库/阶段汇总/phase_XX_summary.md

19. 【汇总部门】根据阶段汇总，生成各卷汇总（整合阶段衔接、卷首卷尾过渡）
    → 输出：汇总仓库/卷汇总/volume_XX_summary.md

20. 【汇总部门】根据卷汇总，生成全文汇总（添加前言、目录、后记，统一格式）
    → 输出：汇总仓库/全文汇总/full_novel.md（即最终发布稿）

21. 【审核部门】对全文汇总进行最终出版级审核（侧重整体连贯性、前后照应、市场定位）
    → 输出：意见仓库/final_review.md

22. 【作家部门】根据最终审核意见，对全文汇总进行最终修订
    → 输出：汇总仓库/全文汇总/full_novel_final.md

23. 【灵感部门/项目经理】确认最终版本，标记为 RELEASED，项目完结
```

### 可选增强步骤（推荐加入）
- **步骤0. 风格校准**：在全文总体大纲后，先写"样本章节"（第1章），经审核+读者反馈确定风格基线，再进入批量正文生产。
- **步骤X. 冲突检测**：汇总部门在生成卷汇总时，运行一致性检查（如人物年龄、地点名称、时间线），输出`consistency_report.md`。

### 2. Subagent 启动范式（Claude Code中）

在Claude Code里，你无法直接像编程框架那样`spawn agent`，但可以通过以下方式模拟：

**方式A：终端多会话（推荐用于并行）**
开多个终端，每个终端运行`claude`，通过`@`引用文件作为上下文：
```bash
# 终端1：启动作家1写第1章
claude --prompt "你是作家'墨痕'，请根据以下画像和阶段大纲写ch_001.md ..." --file 02_writers/writer_01/profile.md --file 03_content/.../phase_outline.md

# 终端2：同时启动作家2写第2章
claude --prompt "你是作家'青灯'..."
```

**方式B：单会话内角色切换（推荐用于串行审核）**
在同一个Claude Code会话中，通过明确的系统提示切换角色：
```markdown
现在进入【审核员模式】。
请读取 04_reviewers/reviewer_01/profile.md 作为你的身份设定。
请读取 03_content/volumes/vol_01/volume_outline.md 作为审核对象。
请输出审核意见到 06_feedback/review/vol_01_rev_01.md。
```

**方式C：Shell/Python编排脚本（最自动化）**
写一个`run_workflow.sh`，用`expect`或简单`echo`向Claude Code的标准输入发送指令序列。

### 3. 关键文档模板示例

```markdown

## 基础信息
- 笔名：墨痕

- 语言风格：古典白话，善用环境烘托心理
- 弱点：不擅长写感情戏，战斗场面偏短

## 角色约束
- 必须遵守项目根目录的 `03_content/style_guide.md`
- 每次写作前必须阅读对应层级的 `phase_outline.md`
- 正文字数控制在3000±200字/章
- 章节结尾必须留下小悬念或情绪钩子

## 记忆（动态追加）
- 2026-05-12：已确认主角'李默'性格为外冷内热
- ...
```

**审核员画像模板** (`04_reviewers/reviewer_XX/profile.md`)：
```markdown
# 审核员画像：严师

## 审核维度
1. 逻辑一致性（时间线、人物动机、因果链）
2. 叙事节奏（铺垫与高潮比例）
3. 世界观自洽（设定不矛盾）
4. 市场适配度（是否符合目标读者期待）

## 输出格式
对每个维度给出：【通过/警告/不通过】+ 具体意见
最后给出总体结论：【通过/需修改】
如为【需修改】，必须列出优先级排序的修改清单（Top 3）
```

## 五、给你的最小可行版本（MVP）

如果你现在就想跑起来，建议先精简：

1. **先不并行**：1个作家、1个审核、2个读者，跑通单卷流程
2. **先不循环**：每个层级审核只给1轮反馈，不通过就人工介入
3. **用Claude Code的`/agent`或新终端窗口**作为Subagent边界
4. **最关键**：把`style_guide.md`在灵感阶段就定死，这是多作家协作的**契约**

你的整体思路是**业界级别的Agent协作设计**，主要缺的是**工程化的终止条件、并行策略和Claude Code的具体调用范式**。补上这些，这个项目可以直接作为开源框架发布了。
